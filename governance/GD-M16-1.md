## Decision 4 — Schema Scope

### Authorized Table

M16 is authorized to create exactly one new table:

```sql
CREATE TABLE recruiting.candidates (
    id         UUID            NOT NULL,
    tenant_id  UUID            NOT NULL,
    first_name VARCHAR(100)    NOT NULL,
    last_name  VARCHAR(100)    NOT NULL,
    email      VARCHAR(255)    NOT NULL,
    phone      VARCHAR(50),
    status     VARCHAR(50)     NOT NULL DEFAULT 'ACTIVE',
    source     VARCHAR(100),
    notes      TEXT,
    created_at TIMESTAMPTZ(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ(6)  NOT NULL,
    deleted_at TIMESTAMPTZ(6),
    CONSTRAINT "candidates_pkey" PRIMARY KEY ("id")
);
```

This definition is authoritative per GD-PRE-PHASE3-002 Decision 3. Where this
definition differs from spec/05, this GD and GD-PRE-PHASE3-002 govern. spec/05
must eventually be amended; that amendment is deferred to the future spec/05
amendment PR.

### Candidate Status Values

```text
ACTIVE    — candidate is active and eligible for applications (default at creation)
ARCHIVED  — candidate is soft-deleted; not returned in normal list/detail flows
```

Status is not a free-form field. Only ACTIVE and ARCHIVED are valid values.
The transition ACTIVE → ARCHIVED is governed by the archive endpoint only.
No API endpoint may set status directly to ARCHIVED via update body.

### Required Indexes

All three indexes are required in the migration:

```sql
-- Standard tenant lookup index
CREATE INDEX "idx_candidates_tenant"
    ON "recruiting"."candidates"("tenant_id");

-- Partial index: active candidates by status, excluding archived/deleted records
CREATE INDEX "idx_candidates_tenant_status"
    ON "recruiting"."candidates"("tenant_id", "status")
    WHERE "deleted_at" IS NULL;

-- Partial unique index: email uniqueness per tenant among non-deleted records
-- See Decision 5 for the partial unique index implementation rule.
CREATE UNIQUE INDEX "idx_candidates_tenant_email"
    ON "recruiting"."candidates"("tenant_id", "email")
    WHERE "deleted_at" IS NULL;
```

## Decision 5 — Partial Unique Index Rule and Prisma Drift Prevention

### The Rule

Candidate email must be unique per tenant among non-deleted, non-archived records.

The uniqueness constraint is:
```sql
UNIQUE (tenant_id, email) WHERE deleted_at IS NULL
```

A re-applicant whose prior candidate record has been archived (deleted_at IS NOT
NULL) may create a new candidate record with the same email address. The prior
archived record does not block the new record.

### Why a Partial Index Is Required

A full unique index `(tenant_id, email)` without a WHERE predicate would prevent
re-applicant scenarios. A candidate archived by the agency after a prior hiring
process could not be re-entered with the same email address. This is operationally
incorrect for a government recruiting platform where candidates may apply to
multiple vacancies over time.

### Implementation Rule: Hand-Edit Required

Prisma does not support partial unique index predicates (`WHERE` clauses) in the
`schema.prisma` model definition as of the current version in use.

**Do NOT add `@@unique([tenantId, email], ...)` to the Candidate model in schema.prisma.**

A `@@unique()` directive in schema.prisma generates a full non-partial unique
constraint in the Prisma-managed migration SQL. If the migration SQL is then
hand-edited to make the index partial, and schema.prisma still carries `@@unique`,
Prisma will detect schema drift on the next `prisma migrate dev` invocation and
may generate a migration that drops the partial index and recreates it as a full
unique constraint.

### Approved Implementation Pattern

This approach combines two raw SQL migration precedents already established in this
repository:

```text
M13 raw-SQL unique index precedent:
  File: apps/api/prisma/migrations/20260620032716_add_m13_skills_certifications/migration.sql
  SQL:  CREATE UNIQUE INDEX "idx_skills_tenant_name"
            ON "workforce"."skills"("tenant_id", "name");
  Notes: Full unique index; no WHERE clause. Establishes the pattern of raw SQL
         unique indexes in this repo's migration history.

M15 partial WHERE clause precedent:
  File: apps/api/prisma/migrations/20260622120000_add_m15_position_linkage/migration.sql
  SQL:  CREATE INDEX "idx_employees_position"
            ON "workforce"."employees"("position_id")
            WHERE "position_id" IS NOT NULL;
  Notes: Partial non-unique index; WHERE clause is hand-edited into migration SQL
         because schema.prisma @@index() cannot carry a partial predicate.
         Prisma CLI does not reconcile the WHERE clause difference once the
         migration is applied.
```

M16 is the first migration in this repository that requires a PARTIAL UNIQUE index —
a combination of both precedents above. The M13 precedent establishes that unique
indexes are expressed in raw migration SQL. The M15 precedent establishes that WHERE
clause predicates are hand-edited into migration SQL and intentionally kept out of
schema.prisma. M16 applies both techniques to a single index.

M16 must follow the combined pattern for the candidate email uniqueness index:

```text
schema.prisma Candidate model:
  @@index([tenantId, email], map: "idx_candidates_tenant_email")
  — A standard (non-unique) @@index(), NOT @@unique()
  — Tells Prisma an index named idx_candidates_tenant_email exists
  — Does NOT generate a unique or partial constraint from schema.prisma

Migration SQL (hand-edited after prisma migrate dev --create-only):
  Replace the Prisma-generated index line with:
  CREATE UNIQUE INDEX "idx_candidates_tenant_email"
      ON "recruiting"."candidates"("tenant_id", "email")
      WHERE "deleted_at" IS NULL;
```

The migration comment must state why the index was hand-edited:
```sql
-- Partial unique index: email uniqueness per tenant among non-deleted records only.
-- GD-M16-1 Decision 5; GD-PRE-PHASE3-002 D3.
-- Prisma cannot represent partial unique indexes (WHERE predicate) in schema.prisma.
-- Do not replace this with a @@unique() in schema.prisma — doing so would cause
-- Prisma to generate a full non-partial unique constraint on the next migration,
-- breaking the re-applicant scenario.
-- Precedent: M13 raw-SQL unique index (idx_skills_tenant_name, no WHERE);
--            M15 partial WHERE clause (idx_employees_position WHERE position_id IS NOT NULL).
--            M16 combines both: partial + unique.
```

### Service-Layer Defense in Depth

In addition to the database-level partial unique index, CandidateService must
enforce email uniqueness at the service layer:

```text
On createCandidate:
  Check: SELECT 1 FROM recruiting.candidates
         WHERE tenant_id = :tenantId AND email = :email AND deleted_at IS NULL
  If exists → return { tag: 'EMAIL_ALREADY_EXISTS' }

On updateCandidate (if email is being changed):
  Check: SELECT 1 FROM recruiting.candidates
         WHERE tenant_id = :tenantId AND email = :newEmail AND deleted_at IS NULL
               AND id != :candidateId
  If exists → return { tag: 'EMAIL_ALREADY_EXISTS' }
```

Service-layer enforcement provides a deterministic, legible error code before
the database constraint fires. The database constraint is the final safety net.

## Decision 6 — Tenant Isolation Rules

Per GD-PRE-PHASE3-002 Decision 1 (SEC-003):

```text
1. tenantId derived from JWT/session only.
   CandidateController extracts tenantId from the @CurrentUser() decorator,
   which returns the RequestUser populated by JwtStrategy.
   No candidate endpoint may read tenantId from the request body.

2. tenantId not accepted in request body.
   Any request body containing a tenantId field must be stripped by the
   ValidationPipe (whitelist: true) or rejected (forbidNonWhitelisted: true).
   CandidateDto classes must not define a tenantId field.

3. tenantId not exposed in response bodies.
   CandidateResponseDto must not include a tenantId field.
   Serialization at the controller layer must exclude tenantId from all
   candidate response objects.

4. All database queries must include a tenant_id predicate.
   Every Prisma query in CandidateService must include:
     where: { tenantId: tenantId, ... }
   This applies to findMany, findFirst, findUnique, update, and updateMany.

5. NOT_FOUND is returned for absent, soft-deleted, and cross-tenant records.
   CandidateService must not distinguish between "does not exist" and "belongs
   to a different tenant" in its response. Returning NOT_FOUND for cross-tenant
   records prevents tenant enumeration attacks (SEC-003).
```

## Decision 7 — Soft Delete and Archive Behavior

Per GD-PRE-PHASE3-002 Decision 2:

### Archive Operation Contract

The archive operation (`POST /api/v1/candidates/:id/archive`) must:

```text
1. Verify the candidate exists and belongs to the caller's tenant.
   WHERE tenant_id = :tenantId AND id = :id AND deleted_at IS NULL
   If not found: return { tag: 'NOT_FOUND' } → HTTP 404

2. Set both fields in a single atomic update:
   deleted_at = NOW()    — activates soft-delete filter
   status = 'ARCHIVED'   — makes status column accurate independently

3. Emit RECRUITING_CANDIDATE_ARCHIVED audit event.

4. Return { tag: 'ok' } → HTTP 204 No Content.
```

### Visibility of Archived Records

Archived candidates are invisible to normal operations:

```text
GET /api/v1/candidates      — does NOT return archived records (deleted_at IS NULL filter)
GET /api/v1/candidates/:id  — returns NOT_FOUND for archived candidates
PUT /api/v1/candidates/:id  — returns NOT_FOUND for archived candidates
POST /api/v1/candidates/:id/archive (repeat) — returns NOT_FOUND (already archived)
```

### Active Application Check (Stub for M16)

GD-PRE-PHASE3-003 Decision 2 requires: "a candidate with one or more active
non-terminal applications cannot be archived until all active applications reach
a terminal state."

In M16, the `recruiting.applications` table does not yet exist. The check must
be implemented as a documented no-op stub:

```typescript
// GD-PRE-PHASE3-003 D2: active application check deferred to M17.
// In M16 the applications table does not exist; this check trivially passes.
// M17 implementation must replace this stub with a real query:
//   WHERE tenant_id = :tenantId AND candidate_id = :candidateId
//         AND deleted_at IS NULL AND status NOT IN ('REJECTED', 'WITHDRAWN', 'HIRED')
```

The return type `{ tag: 'HAS_ACTIVE_APPLICATIONS' }` must be defined in the
discriminated union now so the controller handles it. In M16 this branch is
unreachable. In M17 it becomes reachable.

### CandidateRecord Type

```typescript
type CandidateRecord = {
  id: string;
  firstName: string;
  lastName: string;
  email: string;
  phone: string | null;
  status: string;
  source: string | null;
  notes: string | null;
  createdAt: Date;
  updatedAt: Date;
  // tenantId excluded — not returned to callers (SEC-003)
  // deletedAt excluded — implementation detail
};
```

### Service Rules

```text
Rule 1: All queries are tenant-scoped.
  Every Prisma query includes WHERE tenant_id = :tenantId.

Rule 2: All normal queries exclude deleted records.
  Every query for active records includes AND deleted_at IS NULL.

Rule 3: NOT_FOUND for all non-visible records.
  absent + soft-deleted + cross-tenant all return { tag: 'NOT_FOUND' }.
  This prevents tenant enumeration.

Rule 4: Email uniqueness checked before insert or email-changing update.
  Service checks explicitly before delegating to Prisma. This provides a
  deterministic error code (EMAIL_ALREADY_EXISTS) before the database
  constraint fires. See Decision 5.

Rule 5: Audit events emitted after successful writes, not inside transactions.
  Pattern from EmployeeService (EMP-700): audit emission is decoupled from
  the database transaction.

Rule 6: Audit metadata must not include PII field values.
  No audit event metadata may contain: firstName, lastName, email, phone, notes.
  Field names (not values) may appear in metadata (e.g., { fields: ['phone', 'source'] }
  for an update event). This extends the EMP-401 pattern to the recruiting domain.

Rule 7: Archive sets both deleted_at and status.
  archive() must update both fields atomically:
    { deleted_at: new Date(), status: 'ARCHIVED' }

Rule 8: HAS_ACTIVE_APPLICATIONS is a stub in M16.
  See Decision 7 for stub implementation requirement.
```

## Decision 12 — Audit Events

### M16 Audit Event Values

M16 must use the following AuditEventType values:

```text
RECRUITING_CANDIDATE_CREATED   — emitted by createCandidate on success
RECRUITING_CANDIDATE_UPDATED   — emitted by updateCandidate on success
RECRUITING_CANDIDATE_ARCHIVED  — emitted by archiveCandidate on success
```

`RECRUITING_CANDIDATE_CREATED` already exists in `audit-event-type.enum.ts`.

`RECRUITING_CANDIDATE_UPDATED` and `RECRUITING_CANDIDATE_ARCHIVED` do not yet
exist in the enum. M16 implementation Step 2 must add both values to
`audit-event-type.enum.ts` under the existing `// AUD-500: Recruiting Events`
section before CandidateService is written.

### Audit Event Payload Rules

All three M16 audit events must conform to the following rules:

```text
Required in every audit payload:
  actorId      — user performing the action (from @CurrentUser())
  tenantId     — from JWT (from @CurrentUser())
  entityType   — 'CANDIDATE'
  entityId     — the candidate UUID
  eventType    — the AuditEventType value

Prohibited in audit payload metadata:
  firstName    — PII
  lastName     — PII
  email        — PII
  phone        — PII
  notes        — PII

Permitted in audit payload metadata:
  Field names changed (for UPDATED): e.g., { fieldsChanged: ['phone', 'source'] }
  Source value (for CREATED): e.g., { source: 'USAJOBS' } — source is not PII
  Status transition (for ARCHIVED): e.g., { from: 'ACTIVE', to: 'ARCHIVED' }
```

The `source` field is not considered PII (it describes how the candidate was
referred, not who they are) and may appear in audit metadata.

Pattern authority: EMP-401 (employee audit PII prohibition), extended here to
the recruiting domain.

## Decision 13 — Error Codes

M16 introduces the following domain error codes:

```text
CANDIDATE_NOT_FOUND              — 404 Not Found
  Candidate does not exist, is archived, or belongs to a different tenant.

CANDIDATE_EMAIL_ALREADY_EXISTS   — 409 Conflict
  An active (non-archived) candidate with this email already exists in the tenant.

CANDIDATE_HAS_ACTIVE_APPLICATIONS — 409 Conflict
  Archive blocked because the candidate has active non-terminal applications.
  Unreachable in M16. Becomes reachable in M17 when applications table exists.

INVALID_CANDIDATE_STATUS         — 400 Bad Request
  If the status query filter contains a value other than ACTIVE or ARCHIVED.
  Controller maps this from ListCandidatesQueryDto validation failure.
```

Standard platform codes from spec/06 also apply:
```text
VALIDATION_ERROR  — 400 Bad Request (DTO validation failure)
UNAUTHORIZED      — 401 Unauthorized (no JWT or invalid JWT)
FORBIDDEN         — 403 Forbidden (role not authorized for endpoint)
INTERNAL_ERROR    — 500 Internal Server Error (unhandled exception)
```
