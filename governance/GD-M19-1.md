## Decision 1 — M19 Classification

M19 Hire-to-Employee Conversion is:

```text
Phase 3 Recruiting & Staffing — fourth implementation milestone
Backend only — no frontend, no UI, no Recruiter Workspace pages
Hire-to-Employee Conversion — authorizes a single action endpoint that
  connects the Recruiting domain to the Workforce domain
VAC-401 Path B — Competitive Appointment hire event (GD-PRE-M13-001)
Built on top of M18 Interview and Offer Management
Not a frontend milestone (M20)
Not a notification milestone (Phase 5)
Not an AI milestone (Phase 4)
Not a batch-processing or bulk-hire milestone
Not an employee onboarding workflow milestone beyond creating the
  PENDING_ONBOARDING employee record
```

M19 completes the accepted-offer-to-employee backend path. After M19, the
Phase 3 Recruiting & Staffing backend workflow is end-to-end: candidate
created → application submitted → interview conducted → offer accepted →
employee record created.

M19 does not advance Phase 3 to a user-visible state. The recruiting
workflow remains backend-only until M20 (Phase 3 UI milestone).

The following single endpoint is authorized for M19 implementation:

```text

```

- An action endpoint, not a CRUD endpoint
- Idempotent in success outcome: attempting to hire an already-HIRED

```text
POST /api/v1/applications/{id}/offer — M18 scope (not authorized)
Any new application offer endpoints
Any Recruiter Workspace UI — M20 scope
```

## Decision 4 — Preconditions

The hire action must validate all of the following preconditions before
beginning any write operation. Validation must fail fast — the first
failing precondition returns an error without evaluating subsequent ones.

Preconditions are evaluated in this order:

```text
1. Application exists:
   recruiting.applications WHERE id = :id AND tenant_id = :actorTenantId
     AND deleted_at IS NULL
   Failure: APPLICATION_NOT_FOUND (404)

2. Application has not already been hired — idempotency guard:
   application.status != 'HIRED'
   This check fires before the OFFER status check so that a repeat hire
   attempt returns the specific 409 idempotency error rather than the
   generic 422 workflow error. See Decision 13 for HTTP mapping.
   Failure: APPLICATION_ALREADY_HIRED (409)

3. Application is at OFFER status:
   application.status = 'OFFER'
   Rule 2 has already excluded the HIRED case. Any remaining non-OFFER
   status — APPLIED, SCREENING, INTERVIEW, EVALUATION, REJECTED,
   WITHDRAWN — maps to this failure.
   Failure: APPLICATION_NOT_AT_OFFER_STATUS (422)

4. An ACCEPTED offer exists:
   recruiting.offers WHERE application_id = :applicationId
     AND status = 'ACCEPTED' AND deleted_at IS NULL
   Failure: ACCEPTED_OFFER_NOT_FOUND (422)

5. Candidate exists and is not archived:
   recruiting.candidates WHERE id = :candidateId
     AND tenant_id = :actorTenantId AND deleted_at IS NULL
   Failure (absent or deleted): CANDIDATE_NOT_FOUND (404)
   Failure (status = 'ARCHIVED'): CANDIDATE_ARCHIVED (422)

6. Vacancy exists and is in a hireable state:
   workforce.vacancies WHERE id = :vacancyId
     AND tenant_id = :actorTenantId AND deleted_at IS NULL
   Failure (absent or deleted): VACANCY_NOT_FOUND (404)
   Acceptable vacancy statuses: OPEN, IN_RECRUITMENT, CLOSED
   (CLOSED is included because a vacancy may be administratively closed
   while a hire is in progress; the hire must still complete)
   Unacceptable statuses: FILLED, CANCELLED
   Failure: VACANCY_NOT_AVAILABLE (422)

7. Vacancy has a valid positionId:
   vacancy.positionId must not be null
   workforce.positions WHERE id = :positionId
     AND tenant_id = :actorTenantId AND deleted_at IS NULL
   Failure (positionId null): POSITION_NOT_FOUND (404)
   Failure (position absent or deleted): POSITION_NOT_FOUND (404)

8. Position is available for assignment:
   Position must exist in same tenant (confirmed by rule 7).
   No non-separated employee may currently hold this positionId:
     workforce.employees WHERE position_id = :positionId
       AND employment_status IN ('PENDING_ONBOARDING', 'ACTIVE', 'ON_LEAVE', 'SUSPENDED')
       AND deleted_at IS NULL
   This matches the incumbency check in employee.service.ts createEmployee()
   (lines 326–334). tenant_id is not required in this query: positionId is a
   UUID that uniquely identifies a single position within one tenant; no
   position from a different tenant can share the same UUID.
   Failure: POSITION_NOT_AVAILABLE (422)
```

All preconditions use tenant_id = :actorTenantId derived from the JWT.
Cross-tenant records return not-found errors (SEC-003 — prevents enumeration).

## Decision 5 — Employee Creation Rules

On successful hire, a single workforce employee record is created.

### System-assigned fields (never accepted from request body):

```text
tenantId:             actorTenantId derived from JWT (SEC-003)
appointmentAuthority: COMPETITIVE_APPOINTMENT — system-assigned;
                      never from caller input (GD-PRE-M13-001 D3)
employmentStatus:     PENDING_ONBOARDING — initial status for all
                      created employees (GD-M12-1/EMP-001)
hireDate:             server-side current date/time at moment of hire
employeeNumber:       generated by hire service per Decision 11
```

### Fields derived from the vacancy and candidate chain:

```text
positionId:           vacancy.positionId
departmentId:         position.departmentId (resolved from vacancy chain)
firstName:            candidate.firstName
lastName:             candidate.lastName
email:                candidate.email (null if candidate has no email)
```

### Request body:

The POST /applications/{id}/hire endpoint accepts NO request body.
No DTO with create fields is required. If a body is sent, it must be
ignored. A HireApplicationDto is not needed. The {id} path parameter
identifies the application — no additional inputs are accepted.

### Cross-module boundary:

HireService must NOT inject EmployeeService. HireService uses
PrismaService directly for all writes within the hire transaction.
This follows the module boundary reasoning in GD-M17-1 D8:
RecruitingModule does not import WorkforceModule; adding that
dependency for a single transactional operation is unnecessary coupling.

## Decision 6 — Application Transition Rules

### Selected application:

```text
On successful hire, the hired application transitions:
  application.status → 'HIRED'

HIRED is now a reachable terminal status for the first time in Phase 3.
M17 and M18 blocked this transition. M19 authorizes it exclusively via
POST /applications/{id}/hire. No other endpoint may set status = HIRED.
```

### Displaced active applications:

```text
All other active applications for the same vacancy_id as the hired
application are set to REJECTED.

"Active" for displacement purposes:
  recruiting.applications WHERE vacancy_id = :vacancyId
    AND id != :hiredApplicationId
    AND tenant_id = :actorTenantId
    AND deleted_at IS NULL
    AND status NOT IN ('HIRED', 'REJECTED', 'WITHDRAWN')

Displacement uses Prisma updateMany with the above WHERE clause.
The exact count of displaced applications is not required in the audit
event but may be included as displacedCount metadata.
```

### Terminal status choice for displaced applications:

GD-PRE-PHASE3-001 D6 and GD-M18-1 D3 referred to remaining applications
becoming "CLOSED." Inspection finds no CLOSED status in the canonical
application status enum (GD-PRE-PHASE3-002 D8). Adding CLOSED_BY_HIRE
would require a new schema enum value, a migration amendment, and partial
unique index updates.

**This decision uses REJECTED as the terminal status for displaced
applications.** Rationale:

```text
1. REJECTED is already a terminal status in the enum and schema.
2. The partial unique index in M17 excludes REJECTED — displaced
   applications do not hold the uniqueness slot, allowing re-application
   to a future vacancy by the same candidate.
3. Audit metadata for displaced applications includes trigger: 'M19_HIRE'
   to distinguish system-generated rejections from human rejections.
4. No schema migration is required.
```

### Idempotency:

```text
An already-HIRED application → APPLICATION_ALREADY_HIRED (409).
A REJECTED or WITHDRAWN application → APPLICATION_NOT_AT_OFFER_STATUS (422)
  (HIRED status check fires first per Decision 4 ordering).
A HIRED application may not be hired twice under any circumstances.
The hire operation is not idempotent-on-repeat — each call that succeeds
creates exactly one employee record.
```

## Decision 7 — Vacancy and Position Rules

### On successful hire:

```text
vacancy.status → 'FILLED'

This is the VAC-401 Path B outcome (GD-PRE-M13-001; GD-PRE-PHASE3-001 D5).
The FILLED transition is part of the hire transaction.
```

### Audit event for vacancy fill:

```text
Use the new WORKFORCE_VACANCY_FILLED_FROM_HIRE event type (Decision 12),
not the existing WORKFORCE_VACANCY_FILLED.

Rationale: WORKFORCE_VACANCY_FILLED is an existing event used by the
vacancy workflow (workforce domain). WORKFORCE_VACANCY_FILLED_FROM_HIRE
distinguishes the hire-path fill from a manual administrative fill,
enabling audit trail filtering by source.
```

### Position assignment:

```text
Position assignment occurs through the created employee's positionId
field (set to vacancy.positionId during employee creation). There is no
separate "assign-position" call in M19 — the position is set at employee
creation time, inside the hire transaction.

The occupancy check in Decision 4 (rule 8) ensures the position is
unoccupied before the transaction begins. No unique index enforces this
at the database layer (GD-PRE-M13-002 D1 confirmed this is service-layer
only). The pre-check plus the transaction minimizes (but does not
eliminate) the race condition window. See Decision 8 for race condition
handling.
```

### Position not available:

```text
If the position becomes occupied between the precondition check (Decision 4)
and the transaction commit (race condition window), the following occurs:

No DB-level unique constraint on employee.positionId exists. Two employees
could theoretically receive the same positionId. This is the accepted
known risk documented in GD-PRE-M13-002 D1 for the assign-position
endpoint.

The hire transaction does not re-check occupancy inside the transaction.
The pre-check in Decision 4 provides a best-effort guard. This is
consistent with the existing position assignment behavior throughout the
codebase.

This risk is recorded here and in PROGRESS.md as a known limitation.
A future governance decision may introduce a DB-level unique partial index
on employee.positionId if concurrent hire collisions are observed.
```

## Decision 8 — Offer Rules

```text
Offer acceptance alone does not create an employee.
Only POST /applications/{id}/hire creates an employee.
This no-hire invariant (GD-M18-1 D9) remains unconditionally in effect.

Hire requires an ACCEPTED offer (Decision 4, rule 4):
  recruiting.offers WHERE application_id = :applicationId
    AND status = 'ACCEPTED' AND deleted_at IS NULL

The offer record is NOT modified by the hire action. After hire:
  offer.status remains 'ACCEPTED'
  application.status becomes 'HIRED'
  vacancy.status becomes 'FILLED'

If multiple historical offers exist for the application (e.g., original
offer withdrawn, re-offer issued and accepted), the hire action uses
whichever current offer has status = 'ACCEPTED' and deleted_at IS NULL.
Only one ACCEPTED offer can exist per application at a time due to the
partial unique index on offers(application_id) — see GD-M18-1 D5.

Active offers on displaced applications:
  Displaced applications are set to REJECTED (Decision 6). Their
  associated offers may be in any state. Actively withdrawing displaced
  offers (SENT or APPROVED offers on REJECTED applications) is deferred.
  No offer status update is performed for displaced applications in M19.
  Displaced application offers remain in their current state.

  Rationale: The offer record's application relationship is unchanged.
  The application is REJECTED — this provides sufficient workflow signal.
  Withdrawn offer automation for displaced offers is deferred to a future
  governance decision when business need is confirmed.

  This deferral is recorded in the PROGRESS.md M19 closeout.
```

## Decision 9 — Transactionality

The hire operation MUST be fully transactional. All of the following must
succeed atomically, or the entire hire fails with no partial writes:

```text
1. Employee record created (workforce.employees INSERT)
2. Selected application set to HIRED (recruiting.applications UPDATE)
3. Vacancy set to FILLED (workforce.vacancies UPDATE)
4. Displaced active applications set to REJECTED (recruiting.applications
   UPDATE WHERE vacancy_id = :vacancyId AND status NOT IN terminal statuses
   AND id != :hiredApplicationId)
```

### Implementation:

```text
Use prisma.$transaction() wrapping steps 1–4. All four writes execute
inside a single database transaction.

Audit events are emitted AFTER the transaction commits successfully —
never inside the transaction. AuditService.logEvent() (or equivalent)
is called using the shared PrismaService instance, NOT the transaction
client passed to the $transaction() callback. The existing AuditService
infrastructure writes to audit.audit_events via PrismaService directly;
it does not need to be included in the hire transaction and must not be
included. Emitting audit events outside the transaction ensures the
audit writes can proceed even after the transaction context is released.

If the transaction succeeds but an audit emission call fails, the hire
is still considered successful. Audit failure is logged at ERROR level
but does not roll back the hire and does not affect the 201 response.

This is a governed deviation from the EMP-700 pattern. The EMP-700
pattern uses sequential Prisma calls without a transaction wrapper.
M19 requires a transaction wrapper because:
  — Employee creation is irreversible
  — Partial write (employee created, application not HIRED, vacancy not
    FILLED) produces an invalid system state
  — The EMP-700 pattern's acceptable "brief inconsistency window" is not
    acceptable for a multi-domain atomic operation of this consequence
```

### Transaction failure handling:

```text
If prisma.$transaction() throws:
  — Log the error at ERROR level
  — Return INTERNAL_ERROR outcome
  — No partial writes committed (transaction rolled back by Prisma)
  — No audit events emitted

No retry logic is required for M19. A failed hire must be retried
explicitly by the authorized user.
```

## Decision 10 — Tenant Isolation (SEC-003)

The following rules apply to the hire endpoint and HireService:

```text
1. tenantId on all created or updated records is derived from the
   authenticated user's JWT token only. It is never accepted from the
   request body, URL parameters, or query parameters.

2. All precondition lookups include: WHERE tenant_id = :actorTenantId

3. Cross-tenant application, candidate, vacancy, or position records
   return not-found errors (APPLICATION_NOT_FOUND, CANDIDATE_NOT_FOUND,
   VACANCY_NOT_FOUND, POSITION_NOT_FOUND). Callers must not be able to
   distinguish "this record doesn't exist" from "this record exists in
   another tenant."

4. The created employee record receives tenantId = actorTenantId.
   No cross-tenant employee may be created.

5. tenantId must not appear in any HTTP response body.

6. deletedAt must not appear in any HTTP response body.
```

## Decision 11 — Employee Number Generation

No auto-generation helper for employeeNumber exists anywhere in the
codebase as of M18. The existing POST /employees endpoint requires
the caller to supply employeeNumber. The hire endpoint has no human
caller supplying an employee number.

### Generation format:

```text
EMP-{N}

Where N is a zero-padded decimal integer, minimum 3 digits.
Examples: EMP-001, EMP-042, EMP-100, EMP-1024

The prefix EMP- is consistent with the existing seed data and test
fixtures throughout the codebase. It is formally authorized here for
the hire path.
```

### Generation strategy:

```text
Tenant-scoped sequential counter. employeeNumber generation occurs
BEFORE entering the hire transaction.

Phase A — Pre-transaction number generation (outside prisma.$transaction()):

  Step 1: Query the count of existing non-deleted employees in the tenant:
    WHERE tenant_id = :tenantId AND deleted_at IS NULL

  Step 2: Candidate number = count + 1, formatted as EMP-{zero-padded
    to minimum 3 digits}.

  Steps 1 and 2 produce a candidate employeeNumber. No INSERT is
  attempted in Phase A. Phase A is fast and non-transactional.

Phase B — Inside prisma.$transaction():

  Step 3: INSERT the employee record using the Phase A candidate number.

  Step 4: If the INSERT throws a Prisma P2002 error (unique constraint
    violation on idx_employee_number_tenant), the transaction is in an
    error state. Catch P2002 and return EMPLOYEE_NUMBER_GENERATION_FAILED.
    Do NOT retry the INSERT inside the transaction — a P2002 inside a
    Prisma interactive transaction leaves the transaction in a failed
    state; further operations within the same callback would also fail.
    The transaction rolls back cleanly on throw.

Retry semantics:

  There is no retry loop within a single hire() call. If P2002 occurs
  inside the transaction (true concurrent collision), the hire returns
  EMPLOYEE_NUMBER_GENERATION_FAILED (mapped to 500). The authorized user
  may retry the hire action via a second HTTP request. On retry, Step 1
  returns an updated count reflecting the concurrent insert that caused
  the collision, and Step 2 produces a collision-free candidate number.

  Rationale: the count-based approach makes P2002 extremely rare —
  it requires two simultaneous hire actions in the same tenant completing
  at the exact same instant. Cross-request retry is simpler and safer
  than an in-transaction retry loop, which would require restarting the
  entire $transaction() block.

The Prisma unique constraint on employeeNumber within a tenant
(GD-M12-6/EMP-201) ensures correctness under concurrent hire operations.
```

### Module placement:

```text
Implement the generation logic as a private function inside HireService.
Do not create a shared utility file — the generation strategy is simple
and does not warrant a reusable abstraction in M19. If future hire paths
(e.g., Phase 6 bulk import) require the same logic, a shared helper can
be extracted at that time.
```

### Immutability:

```text
Once assigned, employeeNumber is immutable (GD-M12-6/EMP-304). The
generated number must not be altered after the hire transaction commits.
```

## Decision 13 — Error Codes and HTTP Mapping

M19 defines the following service outcome vocabulary and HTTP mapping.
String values must appear in error response bodies using the existing
platform error response style.

```text
APPLICATION_NOT_FOUND             — 404 Not Found
  application absent, soft-deleted, or cross-tenant

APPLICATION_ALREADY_HIRED         — 409 Conflict
  application.status is already HIRED; hire already completed
  Justification for 409 (not 422):
    409 Conflict means "the request conflicts with the current state of
    the resource." An already-HIRED application is an idempotency
    conflict — a second hire attempt conflicts with the existing HIRED
    state. This is semantically equivalent to APPLICATION_ALREADY_EXISTS
    (409) in M17. 422 is reserved for workflow precondition failures
    on in-progress applications, not for terminal-state conflicts.

APPLICATION_NOT_AT_OFFER_STATUS   — 422 Unprocessable Entity
  application.status is not OFFER (and not HIRED — HIRED check fires first)

ACCEPTED_OFFER_NOT_FOUND          — 422 Unprocessable Entity
  no offer exists with status ACCEPTED for this application

CANDIDATE_NOT_FOUND               — 404 Not Found
  candidate absent, soft-deleted, or cross-tenant

CANDIDATE_ARCHIVED                — 422 Unprocessable Entity
  candidate exists but status = ARCHIVED; cannot be hired

VACANCY_NOT_FOUND                 — 404 Not Found
  vacancy absent, soft-deleted, or cross-tenant

VACANCY_NOT_AVAILABLE             — 422 Unprocessable Entity
  vacancy status is FILLED or CANCELLED; not hireable

POSITION_NOT_FOUND                — 404 Not Found
  vacancy.positionId is null, or position absent/deleted/cross-tenant

POSITION_NOT_AVAILABLE            — 422 Unprocessable Entity
  position exists but is currently occupied by an active employee

EMPLOYEE_NUMBER_GENERATION_FAILED — 500 Internal Server Error
  all 3 employeeNumber generation attempts collided; hire cannot proceed

INTERNAL_ERROR                    — 500 Internal Server Error
  unexpected server error; transaction rolled back
```

### HTTP success code:

```text
POST /applications/{id}/hire → 201 Created on success

The response body must include the created employee record in the same
shape as GET /api/v1/employees/{id} — using the existing toEmployeeShape()
projection from employee.controller.ts. This allows callers to confirm
the new employee ID, employeeNumber, and employmentStatus without a
second GET request.

tenantId must not appear in the response body (SEC-003).
deletedAt must not appear in the response body.
```
