# Context

spec/05_database_schema.md defines workforce.skills and workforce.certifications without
tenant_id columns. spec/04_domain_model.md's Entity Ownership Matrix does not list Skill
or Certification as owned entities, though it explicitly lists Employee Skill and Employee
Certification. spec/05_database_schema.md's Tenant Strategy states: "All tenant-owned
tables shall include tenant_id UUID NOT NULL."

The omission of tenant_id from the catalog tables creates an ambiguity between a
platform-global shared catalog (spec-literal) and a per-tenant catalog (spec-consistent
with the overall architecture). This decision resolves that ambiguity before any M13
schema migration is written.

The spec/06_api_contracts.md role model defines all administrator roles (System
Administrator, HR Director) as per-tenant constructs. No platform-level administrator
role exists. Under a global catalog, any tenant's System Administrator could create,
modify, or accumulate catalog entries that appear in all other tenants' reads. This is
a multi-tenant isolation risk the specification provides no mechanism to prevent.

---

# Decisions

## Decision 1

workforce.skills and workforce.certifications are per-tenant resources.

Each government agency tenant maintains its own independent skills and certifications
library. Catalog entries are not visible across tenant boundaries.

## Decision 2

Both workforce.skills and workforce.certifications shall include:

```text
tenant_id UUID NOT NULL
```

sourced exclusively from the authenticated user's JWT on all create and read operations.
This column is added in the M13 schema migration.

## Decision 3

SEC-003 applies in full to all Skills and Certifications catalog endpoints.

tenant_id must never be accepted from a request body, query parameter, or route
parameter. A request for a catalog entry belonging to a different tenant returns
HTTP 404. The response is identical for absent and cross-tenant records. No tenant
enumeration is permitted.

## Decision 4

Catalog management authorization (create, update) is scoped to System Administrator
and HR Director roles within the authenticated tenant — the same authorization model
applied to departments, positions, and employees.

## Decision 5

All authenticated roles may read catalog entries scoped to their own tenant, subject
to the RBAC matrix established in GD-M13-2.

## Decision 6

Skills and certifications assigned to an employee always belong to the same tenant as
the employee. No cross-tenant skill or certification references are permitted. A skill
or certification ID from a different tenant used in an employee assignment returns
HTTP 422 with error code SKILL_NOT_FOUND or CERTIFICATION_NOT_FOUND respectively.

## Decision 7 — Complete Column Set (Implementation Confirmation — Shared Column Resolution 2026-06-19)

The complete column set for workforce.skills is:

```sql
id              UUID PRIMARY KEY
tenant_id       UUID NOT NULL                 -- GD-M13-1 Decision 2
name            VARCHAR(255) NOT NULL
category        VARCHAR(100)
description     TEXT
created_at      TIMESTAMPTZ NOT NULL          -- spec/05 Shared Columns principle
updated_at      TIMESTAMPTZ NOT NULL          -- spec/05 Shared Columns principle
deleted_at      TIMESTAMPTZ NULL              -- spec/05 soft-delete strategy
```

The complete column set for workforce.certifications is:

```sql
id                   UUID PRIMARY KEY
tenant_id            UUID NOT NULL            -- GD-M13-1 Decision 2
name                 VARCHAR(255) NOT NULL
issuer               VARCHAR(255)
expiration_required  BOOLEAN NOT NULL DEFAULT FALSE
created_at           TIMESTAMPTZ NOT NULL     -- spec/05 Shared Columns principle
updated_at           TIMESTAMPTZ NOT NULL     -- spec/05 Shared Columns principle
deleted_at           TIMESTAMPTZ NULL         -- spec/05 soft-delete strategy
```

These column sets are authoritative for the M13 schema migration.
No additional columns are added beyond those declared here.

The junction tables (workforce.employee_skills, workforce.employee_certifications) do
NOT receive tenant_id columns. Tenant isolation for junction records is derived
through the employee FK chain (employee → employees.tenant_id) and enforced at the
service layer per Decision 6. This follows spec/05 exactly for junction table structure.

# Rationale

(1) The per-tenant role model provides no mechanism to prevent cross-tenant catalog
pollution under a global catalog design.

(2) Government agencies operate under distinct and often sensitive competency
frameworks. Federal, state, DoD, and intelligence community HR systems use different
occupational series and qualification standards. A shared catalog cannot accommodate
agency-specific or classification-sensitive competency definitions.

(3) Per-tenant design is consistent with SEC-003 and the pattern applied uniformly
to all other tenant entities throughout the system. A carve-out for skills and
certifications would introduce two authorization patterns — one for catalog endpoints
and one for all other endpoints — increasing implementation error risk.

(4) The spec/04 entity ownership matrix omission of Skill and Certification is a
spec oversight. Every other workforce entity in the system is per-tenant. There is
no declared architectural principle supporting global catalog behavior. The spec/05
Tenant Strategy principle ("All tenant-owned tables shall include tenant_id") applies.

(5) Per-tenant catalog does not prevent a future platform-level seeded catalog. A
future milestone may introduce a platform-administered base catalog with per-tenant
override capability. That is additive. Implementing a global catalog now would require
a disruptive migration to add tenant_id later.

---

# Spec Deviation

workforce.skills and workforce.certifications will include tenant_id UUID NOT NULL.
This deviates from spec/05 table definitions as written.

Deviation authority: this decision (GD-M13-1).

Deviation justification: correction of spec oversight; consistent with spec/05's own
Tenant Strategy principle and the overall system architecture.

The addition of created_at, updated_at, and deleted_at (Decision 7) also deviates from
spec/05 table DDL for these specific tables. Authority: spec/05 Shared Columns principle
and soft-delete strategy, applied consistently with all other tenant-owned entities.
