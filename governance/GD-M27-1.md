# Background

## Backend Gap — No Update Endpoint

UsersController exposes three HTTP methods as of M26:

```text

GET    /api/v1/users        — list users (M13/M25)
```

No PATCH, PUT, or DELETE handler exists. No `updateUser()` method exists in
UsersService. No `UpdateUserDto` exists.

All post-creation user management — changing a user's name or email, suspending
or deactivating a user, reactivating a previously suspended user, and
reassigning a user's platform roles — requires a write endpoint that M27 must
deliver.

As of M26, /admin/users/[id] is explicitly declared read-only:

```text
// Read-only — no edit, deactivate, suspend, delete, or role change
// affordances (GD-M25-1 D3).
```

The page displays a StatusPill with ACTIVE/INVITED/SUSPENDED/DEACTIVATED color
mapping already implemented, but no action to change that status. It displays
role names but no way to change them. No Edit button exists. No BFF route for
user mutation exists except POST /api/users (create only).

## Frontend Gap — No Edit Page

No /admin/users/[id]/edit route exists. Contrast with departments, which has
a complete edit flow:


# Decisions

## Decision 1 — Scope Boundary: User Management Completion

M27 is the User Management Completion milestone. It delivers all post-creation
user lifecycle operations available within Phase 1 scope: field edits, status
management, and role reassignment.

```text
M27 deliverables:
  Backend:  PATCH /api/v1/users/:id (new endpoint)
  Backend:  UpdateUserDto (new DTO)
  Backend:  UsersService.updateUser() (new service method)
  Backend:  5 new AuditEventType values
  BFF:      PATCH /api/users/[id] (new route)
  Frontend: /admin/users/[id]/edit (new page)
  Frontend: EditUserForm component (new component)
  Frontend: "Edit" button on /admin/users/[id] detail page
  Frontend: Status action buttons on /admin/users/[id] detail page

M27 does not deliver:
  Password change or reset (NotificationModule blocker)
  User invitation or activation (NotificationModule blocker)
  DELETE /api/v1/users/:id (no delete actions policy)
  Audit log viewer (Phase 5 governance required)
  Dashboard aggregate endpoint (Phase 5; current approach is functional)
  Any Prisma schema changes or database migrations
```

Rationale: The three concerns — field edits, status management, and role
reassignment — share a single PATCH endpoint and a single service method
with a common $transaction. Splitting them would produce multiple milestones
of near-identical overhead for work that is architecturally unified. The role
reassignment infrastructure (GET /api/v1/roles endpoint, FORBIDDEN_ROLE_ASSIGNMENT
guard, role checkbox component) was deliberately built in M26 to make M27
role reassignment incremental rather than additive. Combined scope is correct
here and closes the user lifecycle chapter within Phase 1.

## Decision 2 — PATCH /api/v1/users/:id Endpoint Definition

A new PATCH /api/v1/users/:id endpoint is added to UsersController. The
endpoint uses PATCH semantics (partial update) rather than the spec's PUT
because partial field updates are the primary use case — requiring a complete
user representation on every write would force the caller to know the full
current state including passwords the caller cannot read.

```text
Endpoint:
  PATCH /api/v1/users/:id

RBAC:
  JwtAuthGuard + RolesGuard (class-level guards on UsersController)
  System Administrator: may update any user (subject to Decision 5 guard)
  HR Director: may update non-SA users only (see Decision 6)
  Workforce Planner, Recruiter, Hiring Manager, Compliance Officer,
  Executive User: receive HTTP 403 from class-level RolesGuard

Route parameter:
  :id — UUID v4 of the target user; validated with ParseUUIDPipe

Request body:
  {
    "firstName": "string (optional)",
    "lastName": "string (optional)",
    "email": "string (optional)",
    "status": "SUSPENDED | ACTIVE | DEACTIVATED (optional)",
    "roleIds": ["uuid", ...] (optional)
  }
  All fields optional. At least one meaningful field must be present.
  An empty body or a body with only unrecognized fields returns 400.

UpdateUserResult discriminated union:
  | { outcome: 'SUCCESS'; user: UserRecord }
  | { outcome: 'NOT_FOUND' }
  | { outcome: 'NO_MEANINGFUL_CHANGE' }
  | { outcome: 'EMAIL_CONFLICT' }
  | { outcome: 'ROLE_NOT_FOUND'; missingIds: string[] }
  | { outcome: 'FORBIDDEN_USER_MANAGEMENT' }
  | { outcome: 'FORBIDDEN_ROLE_ASSIGNMENT'; forbiddenRoleId: string }
  | { outcome: 'LAST_SYSTEM_ADMINISTRATOR' }
  | { outcome: 'INVALID_STATUS_TRANSITION'; from: string; to: string }
  | { outcome: 'INTERNAL_ERROR' }

HTTP status mapping:
  SUCCESS                    → 200 { success: true, data: toResponseShape(user) }
  NOT_FOUND                  → 404 { code: 'NOT_FOUND' }
  NO_MEANINGFUL_CHANGE       → 400 { code: 'VALIDATION_ERROR', message:
                                     'Request contains no updatable fields' }
  EMAIL_CONFLICT             → 409 { code: 'CONFLICT' }
  ROLE_NOT_FOUND             → 400 { code: 'VALIDATION_ERROR' }
  FORBIDDEN_USER_MANAGEMENT  → 403 { code: 'FORBIDDEN_USER_MANAGEMENT' }
  FORBIDDEN_ROLE_ASSIGNMENT  → 403 { code: 'FORBIDDEN_ROLE_ASSIGNMENT' }
  LAST_SYSTEM_ADMINISTRATOR  → 422 { code: 'LAST_SYSTEM_ADMINISTRATOR' }
  INVALID_STATUS_TRANSITION  → 422 { code: 'INVALID_STATUS_TRANSITION' }
  INTERNAL_ERROR             → 500 { code: 'INTERNAL_ERROR' }
```

The response on SUCCESS is the same toResponseShape() helper already used by
createUser, listUsers, and getUserById — returning id, firstName, lastName,
email, status, roles (name strings), createdAt, lastLoginAt.

## Decision 3 — Updatable Fields Policy

The following fields are updatable via PATCH /api/v1/users/:id:

```text
firstName
  Type: string
  Optional: yes
  Validation: non-empty when provided (same as CreateUserDto)
  Effect: replaces the stored firstName

lastName
  Type: string
  Optional: yes
  Validation: non-empty when provided
  Effect: replaces the stored lastName

email
  Type: string
  Optional: yes
  Validation: valid email format (IsEmail decorator); normalized to
  lowercase().trim() before storage and uniqueness check, consistent
  with createUser() behavior
  Effect: replaces the stored email; must be unique within tenant
  Conflict: same P2002 guard as createUser → EMAIL_CONFLICT outcome
  Note: if provided email is identical to current stored email (after
  normalization), no uniqueness conflict occurs; the write proceeds
  normally. This is DB-correct behavior — the unique constraint is
  tenant+email and the target row already holds that value.

status
  Type: string
  Optional: yes
  Validation: must be one of: ACTIVE, SUSPENDED, DEACTIVATED
  Values INVITED and any other string are rejected (400 VALIDATION_ERROR)
  Effect: subject to transition rules (Decision 4) and last-SA guard
  (Decision 5)

roleIds
  Type: string[] (UUID array)
  Optional: yes
  Validation when provided:
    IsArray: must be an array
    ArrayMinSize(1): must contain at least one element
    IsUUID('4') each element: each must be a valid UUID v4
    ArrayUnique: deduplication (same as CreateUserDto)
  Effect: replaces the complete role assignment set (see Decision 7)
  roleIds omitted (undefined): role assignments are NOT changed
  roleIds [] (empty array): DTO validation error — not a no-op

Fields NOT accepted:
  password       — no password change in M27
  tenantId       — SEC-003; rejected at BFF before reaching NestJS
  id             — immutable
  createdAt      — immutable
  deletedAt      — no delete policy
  failedLoginAttempts, lockedUntil — internal auth state; not exposed
  updatedAt      — managed by Prisma @updatedAt

NO_MEANINGFUL_CHANGE detection:
  If the parsed body contains none of: firstName, lastName, email,
  status, roleIds — the service returns { outcome: 'NO_MEANINGFUL_CHANGE' }.
  This is a service-layer guard, not a DTO-layer guard, so that the
  service remains transport-agnostic.
```

## Decision 4 — Status Transition Matrix

Only the following status transitions are permitted in M27:

```text
ALLOWED TRANSITIONS:

  ACTIVE → SUSPENDED
    Effect: User loses platform access; login blocked.
    Subject to: Decision 5 (LAST_SYSTEM_ADMINISTRATOR guard) if target
    user holds System Administrator role.

  ACTIVE → DEACTIVATED
    Effect: User is soft-removed. Login blocked. User appears in list
    with DEACTIVATED status; deletedAt remains null (not a hard delete).
    Subject to: Decision 5 guard.

  SUSPENDED → ACTIVE
    Effect: Reinstates user access.
    No guard required (reinstating cannot reduce SA count).

  SUSPENDED → DEACTIVATED
    Effect: Directly deactivates a suspended user without reactivation.
    Subject to: Decision 5 guard (SA guard applies because target
    could be an SA user in SUSPENDED state).

  DEACTIVATED → ACTIVE
    Effect: Reactivates a previously deactivated user.
    No guard required.
```

```text
DISALLOWED TRANSITIONS (return INVALID_STATUS_TRANSITION):

  INVITED → any status
    Reason: INVITED lifecycle is managed by NotificationModule (not
    built). M27 does not manage the INVITED state.

  any status → INVITED
    Reason: Setting status to INVITED is not a valid operator action.
    INVITED is set only at account creation by the invitation flow
    (deferred to NotificationModule).

  DEACTIVATED → SUSPENDED
    Reason: DEACTIVATED is operationally the more terminal state.
    The operator must reactivate (DEACTIVATED → ACTIVE) before
    suspending. This preserves a clear lifecycle sequence.
```

```text
NO-OP DETECTION:
  If status provided equals the user's current stored status, the
  service does not emit audit events and does not issue a database
  write for the status field. It proceeds with any other changes
  in the same request (field updates, role updates). This is not
  an INVALID_STATUS_TRANSITION — it is treated as "no status
  change requested."
```

Rationale: The INVITED state is intentionally excluded from M27 management
because the INVITED lifecycle (activation, resend, expiration) depends on
NotificationModule infrastructure that does not exist in Phase 1. Allowing
arbitrary transitions into or out of INVITED via PATCH would create an
inconsistent state where users are INVITED but no activation email was ever
sent. DEACTIVATED → SUSPENDED is excluded to enforce a predictable reactivation
path: an administrator who wants to re-restrict a previously deactivated user
must reactivate them first, making the lifecycle history auditable.

## Decision 5 — Last Active System Administrator Safety Guard

The backend must prevent any operation that would leave the authenticated
tenant with zero ACTIVE System Administrator users.

```text
Guard trigger conditions:

  Condition A — Status change to SUSPENDED or DEACTIVATED:
    Target user currently holds the System Administrator role
    AND the requested status is SUSPENDED or DEACTIVATED.

  Condition B — Role replacement removes System Administrator:
    roleIds is provided in the request
    AND the target user currently holds the System Administrator role
    AND the new roleIds set does not include the System Administrator
    role UUID
    AND the target user's current status is ACTIVE.

Guard evaluation query (for either condition):
  Count of users in the same tenant where:
    status = 'ACTIVE'
    AND deletedAt IS NULL
    AND user holds role 'System Administrator'
    AND id ≠ targetUserId

  If count = 0: the target user is the last active SA.
    Return { outcome: 'LAST_SYSTEM_ADMINISTRATOR' }
  If count > 0: the operation is permitted; proceed.

Evaluation point:
  The count query and the subsequent update must execute within the
  same $transaction to minimize the window for a concurrent race.
  The exact implementation uses Prisma's $transaction with the count
  query issued first:

    await this.prisma.$transaction(async tx => {
      const remaining = await tx.userRole.count({
        where: {
          role: { name: 'System Administrator' },
          user: { status: 'ACTIVE', deletedAt: null, tenantId,
                  id: { not: targetId } },
        },
      });
      if (remaining === 0) throw new LastSaError();
      // ... apply updates
    });

Concurrency note — known Phase 1 limitation:
  Under Prisma's default READ COMMITTED isolation, two concurrent
  PATCH requests targeting the last active SA could both pass the
  guard (both see count = 0 referring to a user other than themselves)
  and both succeed. This is an operationally low-risk edge case:
  it requires two concurrent deactivation/suspension requests for
  the same user, which is unlikely in normal operations. Full
  protection requires either serializable transaction isolation or
  application-level advisory locking. Both are Phase 6 hardening
  items. The guard provides strong protection for the common case.

Controller response for LAST_SYSTEM_ADMINISTRATOR (HTTP 422):
  {
    "success": false,
    "error": {
      "code": "LAST_SYSTEM_ADMINISTRATOR",
      "message": "Cannot modify the last active System Administrator"
    }
  }

No audit event is emitted for a blocked guard. The operation is
rejected before any state change occurs. AUTHZ_ACCESS_DENIED is
not appropriate here — this is a safety invariant, not an
authorization violation.
```

Rationale: A platform with zero active System Administrators is locked out —
no actor can add more SAs because the SA-only operations are inaccessible.
The guard is a safety invariant, not a policy decision. It must be enforced
at the service layer (not the frontend) because direct API calls bypass the
frontend.

## Decision 6 — HR Director Authority Boundary for User Management

HR Director authority in M27 applies only to users who do NOT hold the
System Administrator role.

```text
Policy:

  HR Director MAY:
    Edit firstName, lastName, email for non-SA users
    Change status (suspend, deactivate, reactivate) for non-SA users
      (subject to Decision 5 guard — guard also applies to HRD operations)
    Reassign roles for non-SA users
      (subject to FORBIDDEN_ROLE_ASSIGNMENT guard — no SA role assignment)

  HR Director MAY NOT:
    Edit any field for a user who holds System Administrator
    Change status for a user who holds System Administrator
    Reassign roles for a user who holds System Administrator
    Assign System Administrator to any user (carried from M26 Decision 3)

Backend enforcement in UsersService.updateUser():

  If actor does NOT include 'System Administrator' in actorRoles:

    Step 1 — fetch target user's current roles:
      const targetRoles = (await this.prisma.userRole.findMany({
        where: { userId: targetId },
        select: { role: { select: { name: true } } },
      })).map(ur => ur.role.name);

    Step 2 — target user SA check:
      if (targetRoles.includes('System Administrator')):
        emit AUTHZ_ACCESS_DENIED audit event
        return { outcome: 'FORBIDDEN_USER_MANAGEMENT' }

    Step 3 — proposed role assignment SA check (if roleIds provided):
      (same FORBIDDEN_ROLE_ASSIGNMENT guard as createUser, GD-M26-1 D3)

Controller response for FORBIDDEN_USER_MANAGEMENT (HTTP 403):
  {
    "success": false,
    "error": {
      "code": "FORBIDDEN_USER_MANAGEMENT",
      "message": "HR Director may not manage System Administrator users"
    }
  }

Controller response for FORBIDDEN_ROLE_ASSIGNMENT (HTTP 403):
  (same as M26 — code: 'FORBIDDEN_ROLE_ASSIGNMENT')

AUTHZ_ACCESS_DENIED audit event is emitted for both forbidden outcomes,
consistent with GD-M26-1 Decision 3 audit policy.

Frontend enforcement (defense-in-depth, not authoritative):
  Edit button on /admin/users/[id] is hidden for SA users when actor
  is HRD. Status action buttons are hidden for SA users when actor is HRD.
  Backend enforcement is the authoritative control.
```

Rationale: The HRD authority boundary for user management is consistent with
the assignable-role restriction established in M26 (GD-M26-1 Decision 3). HR
Directors manage HR operations but cannot act on the accounts of platform
administrators whose privilege level exceeds their own. This is the principle
of least privilege applied to user management actions, not just role assignment.

## Decision 7 — Existing-User Role Reassignment Policy

When roleIds is provided in the PATCH body, the backend replaces the user's
complete role assignment set atomically.

```text
Semantics:
  roleIds undefined (omitted): role assignments are unchanged
  roleIds [] (empty array): DTO validation error (400 VALIDATION_ERROR)
  roleIds [id1, id2, ...]: replace all current role assignments with
  exactly the provided set (minimum 1 role enforced by ArrayMinSize(1))

Role existence validation:
  All provided roleIds must exist in identity.roles.
  If any roleId is not found:
    return { outcome: 'ROLE_NOT_FOUND'; missingIds: string[] }
  (Same pattern as createUser — fail before touching any data.)

FORBIDDEN_ROLE_ASSIGNMENT guard:
  If actor does not include 'System Administrator' in actorRoles
  AND any element of new roleIds resolves to role name 'System Administrator':
    emit AUTHZ_ACCESS_DENIED audit event
    return { outcome: 'FORBIDDEN_ROLE_ASSIGNMENT'; forbiddenRoleId: saRoleId }
  (Same guard as GD-M26-1 Decision 3, extended to updateUser.)

Role replacement transaction:
  The deleteMany and createMany operations execute within the same
  $transaction as field updates and status changes:

    await tx.userRole.deleteMany({ where: { userId: targetId } });
    await tx.userRole.createMany({
      data: foundRoles.map(r => ({ userId: targetId, roleId: r.id })),
    });

  If either fails, the entire transaction is rolled back — no partial
  role state is possible.

Role discovery for edit form (frontend):
  The edit form populates the role selector from GET /api/v1/roles
  (already built in M26). The server component for the edit page
  fetches both the user and the role list, then maps current role
  names to IDs for pre-population:

    const roleMap = new Map(roles.map(r => [r.name, r.id]));
    const defaultRoleIds = user.roles
      .map(name => roleMap.get(name))
      .filter((id): id is string => id !== undefined);

  No hardcoded role UUIDs are permitted anywhere in apps/web/src/.
  The role selector shows only roles returned by GET /api/v1/roles
  for the actor's role level (SA sees 7; HRD sees 6).

Audit events for role replacement (see Decision 8):
  After the $transaction commits:
    For each role in the previous set NOT in the new set:
      emit AUTHZ_ROLE_REMOVED
    For each role in the new set NOT in the previous set:
      emit AUTHZ_ROLE_ASSIGNED
  No event is emitted for roles that appear in both sets (unchanged).
```

Rationale: Complete replacement is simpler to reason about than incremental
add/remove semantics. The administrator selects the desired final role set
from a live checkbox group; the backend implements that intent atomically.
Incremental semantics (add one role, remove another) would require explicit
diff tracking in the DTO, adding complexity without benefit for the current
Admin Workspace use case.

## Decision 8 — Audit Events Policy

M27 requires 5 new AuditEventType values. These are added to
audit-event-type.enum.ts under the existing AUD-250 and AUD-300 blocks.
String values must be stable after first use — they are stored verbatim in
audit.audit_events.action and must never change after the first commit.

```text
New AuditEventType values:

  AUD-250 block — Identity User Management Events:
    IDENTITY_USER_UPDATED    = 'IDENTITY_USER_UPDATED'
    IDENTITY_USER_SUSPENDED  = 'IDENTITY_USER_SUSPENDED'
    IDENTITY_USER_DEACTIVATED = 'IDENTITY_USER_DEACTIVATED'
    IDENTITY_USER_REACTIVATED = 'IDENTITY_USER_REACTIVATED'

  AUD-300 block — Authorization Events:
    AUTHZ_ROLE_REMOVED = 'AUTHZ_ROLE_REMOVED'

Reused values (no change to enum):
  AUTHZ_ROLE_ASSIGNED       — emitted for each role newly added during update
  AUTHZ_ACCESS_DENIED       — emitted for FORBIDDEN_USER_MANAGEMENT and
                              FORBIDDEN_ROLE_ASSIGNMENT outcomes (same as M26)

Emission rules:

  IDENTITY_USER_UPDATED:
    Emitted when any of firstName, lastName, or email changes.
    One event per successful update containing field changes.
    entityType: 'USER', entityId: targetUserId
    metadata: (none — field values are PII and must not appear in
    audit metadata per GD-M17-1 PII-safe rule)

  IDENTITY_USER_SUSPENDED:
    Emitted when status transitions to SUSPENDED.
    entityType: 'USER', entityId: targetUserId

  IDENTITY_USER_DEACTIVATED:
    Emitted when status transitions to DEACTIVATED.
    entityType: 'USER', entityId: targetUserId

  IDENTITY_USER_REACTIVATED:
    Emitted when status transitions to ACTIVE from SUSPENDED or DEACTIVATED.
    entityType: 'USER', entityId: targetUserId

  AUTHZ_ROLE_REMOVED:
    Emitted once per role removed from the user during role replacement.
    entityType: 'USER', entityId: targetUserId
    metadata: { roleId: string; roleName: string }
    (roleId and roleName are identifiers, not PII — safe in metadata)

  AUTHZ_ROLE_ASSIGNED:
    (Reused from M26) Emitted once per role newly assigned to the user.
    entityType: 'USER', entityId: targetUserId
    metadata: { roleId: string; roleName: string }

  AUTHZ_ACCESS_DENIED:
    (Reused from M26) Emitted for FORBIDDEN_USER_MANAGEMENT and
    FORBIDDEN_ROLE_ASSIGNMENT outcomes.
    entityType: 'ROLE', entityId: (SA roleId for FORBIDDEN_ROLE_ASSIGNMENT;
    targetUserId for FORBIDDEN_USER_MANAGEMENT)

Emission timing:
  All audit events are emitted AFTER $transaction commits, consistent
  with the AUD-1300 rule that audit failure must not block the primary
  operation. AuditService.logEvent() swallows its own errors.

A request that changes fields AND status AND roles emits all applicable
events in sequence:
  1. IDENTITY_USER_UPDATED (if fields changed)
  2. IDENTITY_USER_SUSPENDED / DEACTIVATED / REACTIVATED (if status changed)
  3. AUTHZ_ROLE_REMOVED × n (for each removed role)
  4. AUTHZ_ROLE_ASSIGNED × n (for each newly added role)
```

## Decision 9 — BFF Route Contract: PATCH /api/users/[id]

A new BFF route is added for the Edit User and Status Actions mutations.

```text
New BFF route file:
  apps/web/src/app/api/users/[id]/route.ts

Handler: PATCH

Required behavior:
  1. Parse request body as JSON; return 400 on malformed JSON.

  2. FORBIDDEN_FIELD enforcement (SEC-003):
     If 'tenantId' is present in the request body, return HTTP 400
     with { code: 'VALIDATION_ERROR' } immediately.
     Do not forward to NestJS.
     (Same pattern as POST /api/users and all M25 BFF routes.)

  3. Read JWT from SESSION_COOKIE. If absent → 401 UNAUTHORIZED.

  4. Extract [id] from route params (Next.js dynamic segment).
     Validate that it is a non-empty string.
     (UUID format validation is performed by ParseUUIDPipe on the
     NestJS controller; do not duplicate UUID parsing in the BFF.)

  5. Forward the request body to:
       PATCH /api/v1/users/:id
     with:
       Content-Type: application/json
       Authorization: Bearer <token>

  6. Forward NestJS HTTP status and response body unchanged:
       200  — success
       400  — validation failure (NO_MEANINGFUL_CHANGE, ROLE_NOT_FOUND,
              malformed body, INVALID_STATUS value)
       403  — FORBIDDEN_USER_MANAGEMENT or FORBIDDEN_ROLE_ASSIGNMENT
       404  — NOT_FOUND
       409  — EMAIL_CONFLICT
       422  — LAST_SYSTEM_ADMINISTRATOR or INVALID_STATUS_TRANSITION
       500  — internal server error

  7. Network failure (fetch throws) → HTTP 503 INTERNAL_ERROR.
     { success: false, error: { code: 'INTERNAL_ERROR',
       message: 'Unable to reach the server' } }

No other handlers on this route file:
  No GET (user read is serverFetch in server components)
  No POST (create is handled by /api/users/route.ts)
  No DELETE (no delete policy)

Response types (apps/web/src/features/admin/types.ts must include):
  UpdateUserBffResponse:
    | { success: true; data: UserRow }
    | { success: false; error: { code: string; message: string } }
```

Rationale: The PATCH /api/users/[id] BFF route follows the same contract
policy established in GD-M25-1 Decision 7 and extended to POST /api/users
in GD-M26-1 Decision 4. Consistent BFF contract enforcement reduces
implementation risk and review surface. Truthful error propagation is
mandatory — the admin must know exactly what went wrong with a user
modification that has direct operational consequences.

## Decision 11 — Status Action Buttons Specification

Status management actions appear on /admin/users/[id] (the user detail page),
not on the edit form. They are discrete operations on user lifecycle state
and are implemented as a client component that issues PATCH /api/users/[id].

```text
Component:
  apps/web/src/features/admin/components/user-status-actions.tsx
  ('use client' — interacts with PATCH BFF route)

Actions rendered by current user status:

  ACTIVE user:
    Suspend button  → PATCH { status: 'SUSPENDED' }
    Deactivate button → PATCH { status: 'DEACTIVATED' }

  SUSPENDED user:
    Reactivate button → PATCH { status: 'ACTIVE' }
    Deactivate button → PATCH { status: 'DEACTIVATED' }

  DEACTIVATED user:
    Reactivate button → PATCH { status: 'ACTIVE' }

  INVITED user:
    No status action buttons.
    Render a read-only note: "This user has a pending invitation.
    Status changes are not available until the invitation flow is complete."
    Rationale: INVITED lifecycle is managed by NotificationModule (not
    built). Providing buttons would either do nothing or trigger an
    INVALID_STATUS_TRANSITION error, which is a poor user experience.

Actor visibility rules:
  SA actor: all applicable action buttons shown for all users.
  HRD actor: action buttons shown only when target user does NOT hold
  System Administrator role. For SA users, HRD sees no action buttons.
  (Backend remains the authoritative enforcement layer.)

Interaction model:
  Direct action (no confirmation modal) for Phase 1.
  Button click → PATCH request → success or error inline near buttons.
  On success: router.refresh() — status pill on detail page updates.
  No redirect on status change; the user stays on the detail page.
  Rationale: Status changes are reversible (except DEACTIVATED → ACTIVE
  requires deliberate reactivation). A confirmation modal adds friction
  without materially improving safety for Phase 1. Phase 6 hardening
  can add confirmation for irreversible-feeling actions if needed.

Error display (inline, near the action buttons):
  LAST_SYSTEM_ADMINISTRATOR (422):
    "Cannot modify the last active System Administrator in this tenant."

  INVALID_STATUS_TRANSITION (422):
    "This status change is not permitted."
    (Should not appear in practice if action buttons render correctly
    by current status — this is a defensive fallback.)

  FORBIDDEN_USER_MANAGEMENT (403):
    Not shown — button is hidden for HRD on SA users.
    Defensive: if it reaches the frontend anyway, show:
    "You do not have permission to manage this user."

  Network failure:
    "Unable to reach the server. Please try again."

Button styling:
  Suspend: destructive-secondary style (outline, amber or red tint
  consistent with the status pill color — to be decided by implementing
  engineer within the established M25/M26 palette)
  Deactivate: destructive style (outline, red tint)
  Reactivate: constructive style (outline, green tint or blue #2563eb)
  All buttons: rounded-md, text-sm, consistent with M25/M26 action
  buttons on department detail page.
  No new color values outside the established platform palette
  (#dc2626 red, #d97706 amber, #16a34a green, #2563eb blue, #e2e8f0 border).
```
