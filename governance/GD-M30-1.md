# Background

## Current Intelligence State

The platform has no intelligence layer. No NestJS module at any path under
`apps/api/src/` relates to intelligence, forecasting, risk, or analytics.
The spec/11_ai_architecture.md mandates an `intelligence` module as the home
for all AI and analytical services, but this module does not exist.

The current dashboard at `apps/web/src/app/(dashboard)/dashboard/page.tsx`
performs 22 parallel API fetches returning raw counts and lists. No computed
score, risk level, confidence value, or reasoning string appears anywhere in
the dashboard or any other page. A Workforce Planner reviewing 15 open
vacancies receives no guidance about which vacancies represent the greatest
organizational risk — they must infer this manually from raw fields.

## Vacancy Risk Data Availability

The `workforce.vacancies` table (schema.prisma Vacancy model) contains all
inputs required for deterministic vacancy risk scoring without any schema
changes:

```
createdAt         — vacancy age computation (days open)
priority          — CRITICAL / HIGH / MEDIUM / LOW / null
expectedFillDate  — fill date proximity and overdue detection
filledAt          — historical fill time (for closed vacancies)
status            — OPEN vs IN_RECRUITMENT risk differentiation
positionId        — join to positions for title
```

Scoring can begin the moment M30 is implemented with no migrations.

## Why Deterministic First

FR-401 (Vacancy Risk Prediction) does not mandate AI or LLM — it mandates
that "risk calculated" and "explanation available" be satisfied. A deterministic
formula satisfies both requirements fully while eliminating external API
dependencies, eliminating per-call cost, producing fully reproducible results,
and enabling comprehensive unit testing. The spec/11 AI architecture describes
OpenAI integration for future explanation narratives, not for initial risk
computation. Delivering deterministic vacancy risk in M30 closes FR-401 at
a verifiable maturity level and provides the data surface that future LLM
explanation layers will decorate.

# Decisions

## Decision 1 — Scope Boundary: Phase 4 Entry, Deterministic Vacancy Risk Only

M30 is the entry milestone for Phase 4 Intelligence. Its scope is strictly
bounded to:

```text
What M30 builds:
  Backend  — IntelligenceModule NestJS foundation (module, controller,
             service stub pattern, explainability interface, audit event type)
  Backend  — VacancyRiskService (deterministic scoring, fully unit-testable,
             no external calls)
  Backend  — GET /api/v1/intelligence/vacancy-risk endpoint
  Frontend — Vacancy risk panel on the existing dashboard
  Governance only — Role-Based Intelligence Matrix (see Decision 11)

What M30 does not build:
  Any other Phase 4 intelligence service
  Any external AI or LLM integration
  Any role-personalized dashboard redesign
  Any new database schema
```

Rationale: The intelligence layer must be established before any intelligence
service exists. M30 delivers the module skeleton, the explainability contract,
and the first concrete intelligence signal in the smallest verifiable step.
Deterministic scoring requires no external dependency, is fully testable, and
produces results from day one. Future milestones (M31+) extend the module with
additional services and eventually LLM-generated narratives.

## Decision 2 — IntelligenceModule Path and NestJS Structure

The IntelligenceModule shall be created at:

```text
apps/api/src/intelligence/
```

Rationale for path: spec/11_ai_architecture.md specifies
`apps/api/src/modules/intelligence/` but the codebase does not use a
`/modules/` subdirectory — all domain modules live directly under
`apps/api/src/{domain}/` (identity, organization, workforce, recruiting,
users, audit). This governance decision authorizes the path
`apps/api/src/intelligence/` as the correct path for the intelligence domain,
consistent with the established module co-location convention. This is a
named spec deviation, authorized here.

Internal structure for M30:

```text
apps/api/src/intelligence/
├── intelligence.module.ts
├── intelligence.controller.ts
├── services/
│   └── vacancy-risk.service.ts
├── dto/
│   └── vacancy-risk-response.dto.ts
└── interfaces/
    └── intelligence-explainability.interface.ts
```

The `services/` subdirectory is authorized because Phase 4 will eventually
contain four service files (forecasting, matching, attrition, vacancy-risk).
Placing them in a `services/` subdirectory avoids a flat directory with
mixed module scaffolding and service files. The `dto/` and `interfaces/`
subdirectories follow the same rationale.

NestJS registration:

```typescript
// intelligence.module.ts
@Module({
  imports: [PrismaModule],
  controllers: [IntelligenceController],
  providers: [VacancyRiskService],
})
export class IntelligenceModule {}
```

```typescript
// app.module.ts — add to imports array:
IntelligenceModule
```

The module is registered globally in AppModule alongside all existing domain
modules. No circular dependency risk — IntelligenceModule imports PrismaModule
only; no other domain module imports IntelligenceModule in M30.

## Decision 3 — Tenant Isolation on Intelligence Endpoints

All intelligence endpoints must enforce tenant isolation using the same
pattern as all other domain endpoints in the platform:

```typescript
// IntelligenceController — every handler reads tenantId from JWT
@Get('vacancy-risk')
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles('System Administrator', 'HR Director', 'Workforce Planner')
async getVacancyRisk(@Req() req: { user: RequestUser }, @Query() query: VacancyRiskQueryDto) {
  return this.vacancyRiskService.score(req.user.tenantId, query);
}
```

VacancyRiskService must filter all Prisma queries with `tenantId`:

```typescript
// Every Prisma query in VacancyRiskService must include:
where: { tenantId: tenantId, deletedAt: null }
```

The `tenantId` must never be accepted from the request body or query
parameters — it is always taken from `req.user.tenantId` (JWT context).

Tenant isolation is not optional and cannot be relaxed for intelligence
endpoints. A cross-tenant data leak in an intelligence response would be a
SEC-003 violation regardless of the data being a computed score.

## Decision 5 — Deterministic Vacancy Risk Scoring Formula

The `VacancyRiskService` shall compute a deterministic risk score for each
eligible vacancy using the following governed formula. This formula version
is identified as `deterministic-v1`. Any future change to weights, thresholds,
or inputs requires a new governance decision and a new formula version
identifier.

**Eligible vacancies:**

Only vacancies with `status` of `OPEN` or `IN_RECRUITMENT` and `deletedAt`
equal to null are scored. Vacancies with status `DRAFT`, `FILLED`, `CLOSED`,
or any other value, and all soft-deleted vacancies, are excluded from results.

**Scoring inputs:**

| Input | Source field | Notes |
|---|---|---|
| Vacancy age | `createdAt` → `floor((now - createdAt) / 86_400_000)` days | Integer days open |
| Priority | `priority` VARCHAR | CRITICAL / HIGH / MEDIUM / LOW / null |
| Fill date proximity | `expectedFillDate` DATE | Nullable; days until fill date |
| Vacancy status | `status` VARCHAR | OPEN vs IN_RECRUITMENT |

**Factor weights (sum of all factors = riskScore; max = 100):**

```text
Vacancy Age Factor   (max 40 pts):
  daysOpen >= 90  →  40 pts
  daysOpen >= 60  →  30 pts
  daysOpen >= 30  →  20 pts
  daysOpen >= 14  →  10 pts
  daysOpen <  14  →   0 pts

Priority Factor   (max 40 pts):
  CRITICAL        →  40 pts
  HIGH            →  25 pts
  MEDIUM          →  10 pts
  LOW or null     →   0 pts

Fill Date Factor   (max 15 pts):
  expectedFillDate is not null AND overdue (daysUntilFill < 0)   →  15 pts
  expectedFillDate is not null AND daysUntilFill <= 7            →  10 pts
  expectedFillDate is not null AND daysUntilFill <= 14           →   5 pts
  expectedFillDate is null OR daysUntilFill > 14                 →   0 pts

  daysUntilFill = floor((expectedFillDate - now) / 86_400_000)
  Negative daysUntilFill means overdue.

Vacancy Status Factor   (max 5 pts):
  OPEN            →   5 pts   (not yet in active recruitment — higher risk)
  IN_RECRUITMENT  →   0 pts   (actively being worked — lower marginal risk)
```

**riskScore:**

```text
riskScore = min(100, ageFactor + priorityFactor + fillDateFactor + statusFactor)
```

The maximum achievable score under this formula is 100 (40 + 40 + 15 + 5).

**factors array:**

The service must return a `factors` array that itemizes each non-zero
contribution to the score. Zero-contribution factors should be omitted.
Each factor entry must include:

```typescript
interface RiskFactor {
  name: string;        // 'vacancyAge' | 'priority' | 'fillDateProximity' | 'vacancyStatus'
  contribution: number; // points contributed by this factor
  detail: string;      // human-readable explanation of the input value
}
```

Example factors array for a 30-day-old HIGH priority overdue vacancy:

```json
[
  { "name": "vacancyAge", "contribution": 20, "detail": "Open 30–59 days" },
  { "name": "priority", "contribution": 25, "detail": "Priority: HIGH" },
  { "name": "fillDateProximity", "contribution": 15, "detail": "Expected fill date overdue" },
  { "name": "vacancyStatus", "contribution": 5, "detail": "Status OPEN — not yet in active recruitment" }
]
```

## Decision 6 — Score Range and Risk Level Thresholds

The `riskScore` is always an integer in the range 0–100 inclusive.

**Risk level classification thresholds:**

| riskScore range | riskLevel |
|---|---|
| 0–24 | `LOW` |
| 25–49 | `MEDIUM` |
| 50–74 | `HIGH` |
| 75–100 | `CRITICAL` |

These thresholds are governed values and must not be adjusted in
implementation without a new governance decision. They align with the four
risk levels defined in FR-401.

**Default sort order:**

Results are returned sorted by `riskScore` descending (highest risk first).
Vacancies with equal scores are sorted by `createdAt` ascending (oldest first).

**Pagination:**

The endpoint accepts `?pageSize` (integer, default 10, max 50) and optionally
`?status` (OPEN | IN_RECRUITMENT — omit to include both). No offset
pagination is required in M30; the top-N highest-risk vacancies is the
primary use case.

## Decision 7 — Confidence Behavior and Data Insufficiency Handling

The `confidence` field represents how reliable the computed `riskScore` is
given available input data. It is an integer 0–100.

**Confidence rules:**

| Available inputs | confidence |
|---|---|
| priority known AND expectedFillDate set AND daysOpen > 0 | 100 |
| priority known AND expectedFillDate null AND daysOpen > 0 | 70 |
| priority null AND expectedFillDate set AND daysOpen > 0 | 50 |
| priority null AND expectedFillDate null AND daysOpen > 0 | 40 |
| daysOpen == 0 (vacancy created today, same calendar day) | multiply computed confidence × 0.6, floor to integer |

**Data insufficiency rules:**

These are defensive rules that handle edge cases at runtime.

```text
Vacancy has deletedAt set:
  Excluded from results entirely — do not score.

Vacancy status is not OPEN or IN_RECRUITMENT:
  Excluded from results entirely — do not score.

Vacancy createdAt is in the future (clock skew or corrupt data):
  Excluded from results entirely. Log a warning.

No eligible vacancies exist for the tenant:
  Return empty items array with total: 0.
  This is not an error condition.
```

The service must never throw an unhandled exception for malformed vacancy
data. Any vacancy that cannot be scored due to unexpected data must be
logged and excluded from results — it must not crash the response.

## Decision 8 — Explainability Response Contract

Every vacancy risk result item in the response must include all of the
following fields. An unexplained score is a violation of FR-900 and
AI-002 and must not be returned.

**Explainability interface (authoritative for all future intelligence services):**

```typescript
// apps/api/src/intelligence/interfaces/intelligence-explainability.interface.ts

export interface IntelligenceExplainabilityOutput {
  riskScore: number;        // 0–100
  riskLevel: string;        // LOW | MEDIUM | HIGH | CRITICAL
  confidence: number;       // 0–100; how reliable the score is
  reasoning: string;        // one human-readable sentence summarizing why
  factors: RiskFactor[];    // itemized non-zero contributing factors
  computedAt: string;       // ISO 8601 timestamp of score computation
  formulaVersion: string;   // 'deterministic-v1' for M30
}

export interface RiskFactor {
  name: string;
  contribution: number;
  detail: string;
}
```

This interface is the foundation for all future intelligence services in
Phase 4. ForecastingService, AttritionService, MatchingService, and
WorkforceReadinessService must all return an `IntelligenceExplainabilityOutput`
(or a superset of it). No future Phase 4 service may return a score without
explanation.

**Full response shape for GET /api/v1/intelligence/vacancy-risk:**

```json
{
  "data": {
    "items": [
      {
        "vacancyId": "uuid",
        "positionTitle": "string",
        "departmentName": "string | null",
        "status": "OPEN",
        "daysOpen": 45,
        "priority": "HIGH",
        "riskScore": 60,
        "riskLevel": "HIGH",
        "confidence": 70,
        "reasoning": "HIGH priority vacancy open 30–59 days with no expected fill date set.",
        "factors": [
          { "name": "vacancyAge", "contribution": 20, "detail": "Open 30–59 days" },
          { "name": "priority", "contribution": 25, "detail": "Priority: HIGH" },
          { "name": "vacancyStatus", "contribution": 5, "detail": "Status OPEN — not yet in active recruitment" }
        ],
        "computedAt": "2026-07-14T14:23:00.000Z",
        "formulaVersion": "deterministic-v1"
      }
    ],
    "total": 7,
    "scoredAt": "2026-07-14T14:23:00.000Z",
    "formulaVersion": "deterministic-v1"
  }
}
```

The `reasoning` field must be generated from the actual score inputs — it
must not be a hardcoded template string. The service must compose a
reasoning sentence from the contributing factors dynamically.

## Decision 9 — AI Audit Event Stub Policy

Every call to `GET /api/v1/intelligence/vacancy-risk` must generate an
audit event, even though M30 does not call OpenAI. This establishes the
audit trail pattern that FR-904 requires and ensures all intelligence queries
are auditable from the first day of Phase 4.

**New AuditEventType value to add:**

```typescript
// In apps/api/src/audit/enums/audit-event-type.enum.ts
// Add to new INTELLIGENCE block:

// Intelligence Events
INTELLIGENCE_VACANCY_RISK_QUERIED = 'INTELLIGENCE_VACANCY_RISK_QUERIED',
```

**Audit event data to record:**

```typescript
await this.auditService.logEvent({
  tenantId: requestUser.tenantId,
  userId:   requestUser.userId,
  entityType: 'intelligence',
  entityId:   null,              // no single entity; intelligence is aggregate
  action:     AuditEventType.INTELLIGENCE_VACANCY_RISK_QUERIED,
  result:     'SUCCESS',
  metadata: {
    formulaVersion: 'deterministic-v1',
    itemsReturned:  results.length,
    pageSize:       query.pageSize ?? 10,
    statusFilter:   query.status ?? 'ALL',
  },
});
```

The audit event metadata must never include individual `riskScore` values,
employee PII, or candidate data. Only aggregate metadata is permitted.

## Decision 10 — Dashboard Intelligence Panel: Placement and Design

A vacancy risk panel shall be added to the existing dashboard page at
`apps/web/src/app/(dashboard)/dashboard/page.tsx`.

The panel is inserted as a new full-width grid row between the existing
KPI row (row 1) and the Pipeline + Open Vacancies row (row 2):

```text
[Row 1] KPI cards — Active Workforce / Active Positions / Unfilled Vacancies / Critical Vacancies
[Row 2] Vacancy Risk Signals panel (NEW — M30; role-gated)
[Row 3] Recruiting Pipeline (2/3) + Open Vacancies (1/3)
[Row 4] Workforce Status (1/3 + 2/3 conditional certifications)
```

This placement gives the intelligence signal prominence above the
operational pipeline rows while preserving all existing panel positions
and proportions exactly.

**Role gating:**

The panel is shown only when the session user's roles include at least one
of: System Administrator, HR Director, Workforce Planner.

```typescript
// In DashboardPage (Server Component):
const roles = token ? getSessionRoles(token) : [];
const canSeeIntelligence = roles.some(r =>
  ['System Administrator', 'HR Director', 'Workforce Planner'].includes(r)
);
```

The `serverFetch` for `/api/v1/intelligence/vacancy-risk` must be included
in the `Promise.allSettled` block regardless of role — the 403 response for
non-allowed roles will be handled by the `.catch(() => null)` pattern.
The panel is then conditionally rendered based on `canSeeIntelligence` AND
whether the fetch returned non-null data.

**Panel design:**

The panel must use the existing `CARD` constant from the dashboard page
(white background, `border: 1px solid #e2e8f0`, `borderRadius: 8`,
`boxShadow`). No new design tokens are introduced. The panel must match
the existing `CardHead` component visual treatment (13px 600 weight title,
`View all →` link style, bottom border separator).

Panel elements:
- Card header: title "Vacancy Risk Signals" with no `href` link in M30
  (no dedicated intelligence page exists yet)
- Sub-header line: "Top at-risk open vacancies by deterministic score"
  (11px muted color, matching existing sub-header treatment)
- Per-vacancy row (up to top 5 items by riskScore): positionTitle,
  departmentName (if available), daysOpen label, riskLevel badge, riskScore
- riskLevel badge colors matching existing PriorityBadge treatment:
  CRITICAL → red (#dc2626 bg #fef2f2 border #fecaca)
  HIGH → amber (#d97706 bg #fffbeb border #fde68a)
  MEDIUM → blue (#2563eb bg #eff6ff border rgba(37,99,235,.2))
  LOW → muted (#94a3b8 bg #f8fafc border #e2e8f0)
- Panel footer: "Scores computed from live vacancy data using
  deterministic-v1 formula. Human review required before action."
  This footer is mandatory — it satisfies FR-901 (human review requirement)
  and AI-004 (advisory only) within the dashboard UI.
- If fetch returned null (role denied or error): panel is omitted from
  DOM entirely — no placeholder, no "loading" state, no empty card.
  This matches the existing certifications panel pattern.

The panel must not display any of the following in M30:
- The raw score as a progress bar or animated gauge
- Individual confidence percentages per row (confidence visible in API
  response but not surfaced in the M30 dashboard panel)
- The factors array detail (too granular for a dashboard card)
- Reasoning string per row (the riskLevel badge and daysOpen are sufficient
  at dashboard granularity)

These display enhancements are reserved for a future dedicated intelligence
page that M30 does not create.

**Data fidelity:**

Per GD-M23-1 Decision 2 (Data Fidelity Policy), the panel must show real
scores from real vacancy data only. If no eligible vacancies exist for the
tenant, the panel shows: "No open vacancies scored at this time." No fake
scores, placeholder rows, or hypothetical examples are permitted.

## Decision 11 — Role-Based Intelligence Matrix (Governance Record)

This decision establishes the authoritative Role-Based Intelligence Matrix
for Phase 4. The matrix defines what each role should eventually see, what
each role should not see, what must be backend-enforced, and what can be
frontend-hidden when non-sensitive. M30 implements only the columns marked
as M30 scope. Future milestones must implement their assigned signals
before those signals may be exposed to those roles.

Enforcement classification:

**What must be backend-enforced vs frontend-hidden:**

Any intelligence signal that includes individual-level data (a score linked
to a specific employee, candidate, or named person) must be backend-enforced
— it may not be gated by frontend role-check only. A `roles.includes()`
check in a Server Component is not sufficient protection for PII-linked scores
because BFF routes and direct API calls bypass the frontend entirely.

Aggregated KPI signals (e.g., "agency-wide readiness = 72%", "open vacancy
count = 8") may be frontend-gated because they carry no individual PII and
the backend endpoint already enforces tenant isolation. A role that can call
the endpoint but is not shown the widget in the UI is an acceptable pattern
for aggregated metrics.

## Decision 12 — No External AI/LLM in M30

M30 does not call OpenAI or any external AI/LLM provider under any
circumstance. This is an absolute constraint.

```text
Prohibited in M30:
  OpenAI SDK import or client instantiation
  Any HTTP request to api.openai.com or any AI provider endpoint
  Prompt template execution of any kind
  LLM-generated text in any API response or UI element
  Any runtime cost incurred by an AI provider call
```

Rationale: M30 must be fully self-contained and deployable without any
external API keys or external service availability. The deterministic
scoring formula produces explainable results without LLM support. OpenAI
integration is Phase 4 scope but is deferred to a future milestone (M32 or
later) that will require its own governance decision covering: prompt
management, cost tracking, token limits, failure handling, audit records for
LLM calls, and AI Governance controls per spec/11.

Any future milestone proposing OpenAI integration must:
1. Submit a governance decision before implementation
2. Define prompt versioning (prompt_id, version, author, created, updated)
3. Define cost tracking (model, tokens, estimated cost, tenant)
4. Define failure handling (controlled error on provider unavailability)
5. Confirm that core platform operations remain functional if the AI
   provider is unavailable (per spec/11 AI Failure Handling section)
