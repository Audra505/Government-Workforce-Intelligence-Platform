# Background

## M20 Design Baseline

M20 Recruiting UI established the following platform-wide design decisions
through implementation and browser verification:

```text
Header:
  Navy background #0c2340, full-width, no max-width constraint
  "GWIP" wordmark — not the full platform title
  Domain navigation in header: Workforce | Recruiting
  Inactive domain: text-white/60 with hover transition to text-white
  Active domain: text-white font-semibold
  Logout button at far right — plain text style, no shadcn dependency
  No border-bottom on header

Navigation:
  Inline tab bar below header breadcrumb
  Active tab: 2px solid #2563eb border-bottom, color #0f172a
  Inactive tabs: #475569, hover transition to #0f172a
  Count badge per tab when data is available
  Breadcrumb row: "[Domain] › [Page]"

Canvas:
  Page background #f8fafc
  Content constrained to max-w-[1200px] mx-auto

Color palette (all verified in code):
  NAVY     #0c2340   header background
  CANVAS   #f8fafc   page background
  BORDER   #e2e8f0   all borders
  TEXT     #0f172a   primary text
  SUB      #475569   secondary text
  MUTED    #94a3b8   placeholder / empty states
  BLUE     #2563eb   primary action color
  ERROR    #dc2626   error states

Status indicators:
  Dot + label format (6px filled circle, font-weight 500)
  Pill badges explicitly prohibited on entity lifecycle status

Tables:
  overflow-hidden rounded-md border, borderColor #e2e8f0
  Header: #f8fafc background, uppercase labels, 11px, letterSpacing 0.06em
  Row height: 48px fixed
  Hover: bg-slate-50

Feedback:
  Toast system: bottom-right, success 4s auto-dismiss, error 8s manual dismiss

Typography:
  IBM Plex Sans (body): var(--font-ibm-plex-sans)
  IBM Plex Mono (IDs, codes): var(--font-ibm-plex-mono)
  Self-hosted .woff2 files — no CDN dependency
```

## Workforce UI Current State

The existing Workforce UI was built before M20 and uses a divergent approach:

```text
Header:    White bg-background, full platform title, border-b bottom border
Nav:       Inline dot-separated links inside <main>, no tab bar, no breadcrumb
Status:    Pill badges (rounded-full, Tailwind color classes)
Tables:    Tailwind border token, mixed-case headers, natural row height
Buttons:   shadcn Button component with variants
Forms:     react-hook-form + Zod + shadcn Label components
Inputs:    border-input bg-background (shadcn tokens)
Fonts:     Tailwind font-sans (system stack; IBM Plex CSS vars not applied)
Feedback:  No toast system
```

Key structural issues:
```text
1. No shared Workforce shell — every page inlines its own header and nav
2. getSessionRoles() duplicated in every page file that checks role
3. Two separate logout button implementations (shadcn vs inline)
4. White header creates visual discontinuity when navigating from Recruiting
5. No cross-domain navigation (no path from Workforce to Recruiting or back)
6. Font differs between domains (system stack vs IBM Plex)
```

## Why This Milestone Is Necessary

Every user who navigates between Workforce and Recruiting currently sees a
jarring visual transition: a navy header with IBM Plex fonts and tab navigation
becomes a white header with system fonts and dot-separator navigation. This is
not acceptable in a production government platform. All future milestones
(M22+) will add to this surface — the longer reconciliation is deferred, the
wider the gap becomes and the more expensive the remediation.

# Decisions

## Decision 1 — WorkforceShell Architecture

A shared shell component `WorkforceShell` must be created at:
`apps/web/src/features/workforce/components/workforce-shell.tsx`

```text
WorkforceShell is a Server Component (same as RecruitingShell).
It wraps the content of each Workforce page and renders:

  1. The shared navy application header (see Decision 2)
  2. A domain-specific tab bar: Positions · Vacancies · Employees
  3. A breadcrumb row: "Workforce › [Page]"
  4. A page canvas at #f8fafc
  5. Content constrained to max-w-[1200px] mx-auto

Props:
  activeTab: 'positions' | 'vacancies' | 'employees'
  breadcrumb: string
  children: React.ReactNode
  counts?: { positions?: number; vacancies?: number; employees?: number }

WorkforceShell must NOT contain business logic, data fetching, or RBAC checks.
RBAC checks remain in the page Server Components, passed as props if needed.
```

Rationale:
The existing pattern of each page inlining its own header and nav creates
N copies of the same markup. A shared shell ensures visual consistency,
reduces duplication, and makes future header changes a single-file edit.
GD-M20-1 D4 deferred the shared shell to a future milestone — M21 is that
milestone.

## Decision 2 — Shared Navy Application Header

The platform header is a single shared visual pattern used by both domains:

```text
backgroundColor: #0c2340
padding: px-6 / py-3.5 (matching RecruitingShell)
full-width (no max-width on the outer element)
content row (flex, items-center, justify-between):
  LEFT:
    "GWIP" wordmark — text-base font-semibold tracking-wide color #ffffff
    Domain navigation (see Decision 3)
  RIGHT:
    Logout button (see Decision 8)

Header has no border-bottom.
Header background does not change on scroll.
"Government Workforce Intelligence Platform" as the <title> tag
  (HTML document title, not a visible wordmark).
```

The full platform title "Government Workforce Intelligence Platform" is removed
from the visible header wordmark. "GWIP" is the visible brand. The full title
remains in the HTML `<meta>` title and may appear in the login page heading.

Rationale: "Government Workforce Intelligence Platform" in the current
Workforce header at font-semibold text-lg is visually inconsistent with the
"GWIP" compact wordmark in Recruiting. The Workforce header also lacks domain
navigation. Both issues are resolved by adopting the shared header.

## Decision 3 — Domain Navigation

The header must include domain navigation for all authenticated pages:

```text
Domain links (flex, items-center, gap-6):
  "Workforce" → /workforce/employees (default Workforce landing)
  "Recruiting" → /recruiting/candidates (default Recruiting landing)

Active domain (current user location):
  color: #ffffff
  fontWeight: 600 (font-semibold)
  no underline

Inactive domain:
  color: rgba(255,255,255,0.6) (text-white/60)
  hover: color #ffffff, transition-colors

The active domain is determined by the shell component used:
  WorkforceShell sets Workforce as active.
  RecruitingShell sets Recruiting as active.
No client-side detection is required.
```

Rationale: Currently there is no way for a user on a Workforce page to
navigate to Recruiting (or vice versa) without typing a URL. The domain
nav closes this gap.

## Decision 6 — Platform Typography

IBM Plex Sans and IBM Plex Mono are the platform-wide font family. Both are
already self-hosted as .woff2 files under apps/web/public/fonts/ and
registered in apps/web/src/app/layout.tsx as CSS variables:
  --font-ibm-plex-sans
  --font-ibm-plex-mono

```text
WorkforceShell must apply the font explicitly:
  fontFamily: "var(--font-ibm-plex-sans, 'IBM Plex Sans', system-ui, sans-serif)"
  on the outermost div (same as RecruitingShell).

Monospace content (employee numbers, IDs, codes) must use:
  fontFamily: "var(--font-ibm-plex-mono, 'IBM Plex Mono', monospace)"
  This replaces the current `font-mono` Tailwind class in Workforce tables.

No new font files are required — all files are already present.
No CDN dependency is introduced.
```

Rationale: Workforce pages currently rely on Tailwind font-sans (system
stack). IBM Plex fonts are loaded globally but not applied explicitly in
Workforce. Result: Workforce renders in a different typeface than Recruiting
on most systems. WorkforceShell closes this gap with a single fontFamily prop.

## Decision 10 — Status Indicator Doctrine

The platform adopts a two-pattern status indicator doctrine. The correct
pattern depends on the semantic type of the status value being displayed.

### Pattern A — Dot + Label (entity lifecycle status)

Used when the status represents a single current lifecycle position of an
entity (one and only one status is active at a time).

```text
Format: 6px filled circle + space + label text, font-weight 500
Component: StatusDot from the shared StatusDot primitive (see Decision 14)

Workforce statuses that MUST adopt dot format in M21:

Position lifecycle:
  DRAFT    → gray dot  (#64748b)  "Draft"
  ACTIVE   → green dot (#16a34a)  "Active"
  FROZEN   → blue dot  (#2563eb)  "Frozen"
  CLOSED   → gray dot  (#64748b)  "Closed"

Employment status:
  PENDING_ONBOARDING → yellow dot (#ca8a04) "Pending Onboarding"
  ACTIVE             → green dot  (#16a34a) "Active"
  ON_LEAVE           → blue dot   (#2563eb) "On Leave"
  SUSPENDED          → orange dot (#ea580c) "Suspended"
  SEPARATED          → gray dot   (#64748b) "Separated"

Vacancy status:
  DRAFT          → gray dot   (#64748b) "Draft"
  OPEN           → blue dot   (#2563eb) "Open"
  IN_RECRUITMENT → yellow dot (#ca8a04) "In Recruitment"
  CLOSED         → gray dot   (#64748b) "Closed"
```

### Pattern B — Pill Badge / Severity Indicator (non-lifecycle values)

Used when the value represents severity, priority, or multi-dimensional
information that a dot cannot adequately encode.

```text
Workforce indicators that RETAIN existing pill / text pattern in M21:

VacancyPriority (PriorityBadge):
  CRITICAL / HIGH / MEDIUM / LOW communicate urgency gradients.
  A dot of a single color cannot differentiate these meaningfully.
  PriorityBadge (rounded-full pill) is retained as-is.

AgingCell (age in days + severity):
  Carries two pieces of information: numeric age + risk level.
  Text + color encoding (e.g., "47d ⚠") is not replaceable by a dot.
  AgingCell is retained as-is.

VAC-601 requiresReview badge (Vacancy table):
  The "Review" badge on CRITICAL+OPEN vacancies is retained as-is.
  It is an alert modifier, not a lifecycle status.
```

Rationale: The prohibition on pill badges in GD-M20-1 D16 applies to
Recruiting entity pipeline statuses — it was motivated by the specific
information structure of the recruiting pipeline. Priority badges and
aging cells carry semantic information that pills encode better than dots.
A blanket "no pills anywhere" rule would reduce information density on
Workforce pages without benefit.

## Decision 12 — Toast / Feedback Standard

The platform feedback system uses a bottom-right toast pattern:
- Success: auto-dismisses after 4 seconds
- Error: persists, manual dismiss

```text
Shared toast extraction:
  Current file: apps/web/src/features/recruiting/components/recruiting-toast.tsx
  Target file:  apps/web/src/components/shared/toast.tsx
  Export names: useToast (renamed from useRecruitingToast)

Recruiting components must be updated to import from the new shared path.
Workforce write-action components must add toast feedback in M21C.

Workforce write operations that must show toast on success/failure:
  - Vacancy Open (DRAFT → OPEN)
  - Vacancy Close (any → CLOSED/FILLED)
  - Employee status transitions
  - Position lifecycle actions
```

## Decision 13 — Preservation of Workforce Form Stack

The Workforce form stack (react-hook-form + Zod + zodResolver + shadcn Label)
is explicitly preserved in M21.

```text
Rationale:
  Workforce forms handle more complex validation scenarios than Recruiting
  forms (multi-field schemas, cross-field validation, department selectors).
  Zod schemas provide type-safe validation that raw useState patterns do not.
  Converting Workforce forms to the raw useState pattern used in Recruiting
  would be a regression, not an improvement.

Forms in scope (all preserved exactly as-is):
  create-position-form.tsx
  edit-position-form.tsx
  create-employee-form.tsx
  edit-employee-form.tsx
  create-vacancy-form.tsx
  edit-vacancy-form.tsx

The form internals are not changed in M21. Only the page shell around the
form changes (header/nav via WorkforceShell adoption).
```

# Risks and Controls

## Risk 1 — High File Count

```text
Risk:    M21B touches ~30 files (list pages, detail pages, error.tsx, loading.tsx)
         across 3 Workforce sections. High file count increases regression risk.
Control: Execute batches M21A → M21C in strict sequence.
         Browser-verify each batch before the next begins.
         Do not combine batches into a single commit.
```

## Risk 2 — Loading/Error Shell Consistency

```text
Risk:    If WorkforceShell is adopted in pages but loading.tsx/error.tsx are
         not updated in the same batch, there is a flash of old (white) design
         on page load before data arrives, and on error states.
Control: M21B adopts WorkforceShell across ALL Workforce pages — list, detail,
         new, edit, loading.tsx, and error.tsx — in a single batch.
         No partial adoption within or across sections.
```

## Risk 3 — Status Indicator Visual Change

```text
Risk:    Converting status badges to dots changes the visual presentation
         that existing users may rely on to identify record states.
Control: Pilot on one table first (position-table.tsx), verify in browser
         before converting employee and vacancy tables.
         Priority badges and aging cells are NOT converted — they retain
         all existing visual information.
```

## Risk 4 — Role Gate Preservation

```text
Risk:    Shell refactoring could inadvertently remove or short-circuit the
         canWrite checks that gate action surfaces.
Control: Every page that derives canWrite from JWT must be individually
         verified after M21B that:
         (a) SA/HRD users see action buttons
         (b) Recruiter/CO/WP/EU users do not see restricted buttons
         This is a human browser verification requirement for M21B.
```

## Risk 5 — Recruiting Import Path Breakage (M21A)

```text
Risk:    Moving StatusDot and toast to shared paths requires updating all
         Recruiting import statements. A missed import → build failure.
Control: Run type-check and lint immediately after M21A shared file moves.
         The TypeScript compiler will catch all broken imports.
         M21A must reach CI green before M21B begins.
```
