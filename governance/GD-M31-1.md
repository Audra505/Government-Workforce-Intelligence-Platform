# Background

## Current Intelligence State

The platform has exactly one intelligence signal: deterministic vacancy risk
(M30), visible to System Administrator, HR Director, and Workforce Planner.
Executive User has no Phase 4 intelligence content of any kind today —
Executive User's dashboard shows only the same base operational panels as
every other role, minus anything gated by `canSeeAdmin` or `canSeeIntelligence`
(both of which currently exclude Executive User).

## Workforce Readiness Data Availability

All inputs required for a deterministic workforce readiness score already
exist without any schema change:

```text
workforce.employees        — employmentStatus, positionId, departmentId
workforce.positions        — status (ACTIVE/DRAFT/FROZEN/CLOSED)
workforce.employee_certifications — status (ACTIVE/EXPIRED/REVOKED), expirationDate
workforce.vacancies        — already scored by VacancyRiskService (M30)
```

Scoring can begin the moment M31 is implemented, with no migrations, using
the same PrismaService query pattern already established in M30.

## Why Reuse VacancyRiskService

`VacancyRiskService.score(tenantId, {})` already computes a deterministic,
tenant-scoped, explainable risk score for every eligible vacancy. Workforce
readiness is worse when vacancy risk is high. Recomputing vacancy risk logic
independently inside a second service would duplicate governed formula logic
(GD-M30-1 Decision 5) and risk drift between the two. This decision instead
authorizes `WorkforceReadinessService` to call `VacancyRiskService` directly
as an injected dependency within the same module — the same kind of
in-module service reuse the platform already uses elsewhere (e.g.,
`HireService` composing multiple domain services within a transaction).

## Decision 3 — Tenant Isolation on the Workforce Readiness Endpoint

Identical pattern to GD-M30-1 Decision 3, restated for this endpoint:

```typescript
@Get('workforce-readiness')
@UseGuards(JwtAuthGuard, RolesGuard)
@RequireRoles('System Administrator', 'HR Director', 'Workforce Planner', 'Executive User')
async getWorkforceReadiness(@CurrentUser() actor: RequestUser) {
  return this.workforceReadinessService.score(actor.tenantId);
}
```

`tenantId` is never accepted from the request body, query string, path
parameters, or headers — it is always taken from `actor.tenantId`, resolved
by `JwtAuthGuard` from the JWT. `WorkforceReadinessService` must filter every
Prisma query with `where: { tenantId, deletedAt: null }`, exactly as
`VacancyRiskService` does. This is not optional and cannot be relaxed — a
cross-tenant leak in an aggregate readiness score is still a SEC-003
violation.

## Decision 5 — Deterministic Workforce Readiness Scoring Formula

`WorkforceReadinessService` computes one tenant-wide readiness score per
request, using the following governed formula. This formula version is
identified as `readiness-deterministic-v1`. Any future change to weights,
thresholds, or inputs requires a new governance decision and a new formula
version identifier — exactly the same discipline GD-M30-1 Decision 5
established for vacancy risk.

**Scoring inputs and factor weights (sum of all factors = readinessScore; max = 100):**

```text
Staffing Coverage Factor   (max 30 pts):
  activeRatio = activeEmployees / (activeEmployees + pendingOnboarding +
                onLeave + suspended)
    — excludes SEPARATED employees from the denominator, matching the
      existing dashboard "active current workforce %" calculation
  staffingFactor = round(activeRatio * 30)
  If denominator is 0 (no current workforce): staffingFactor = 15 (neutral
    midpoint — see Decision 7, this also reduces confidence)

Position Capacity Factor   (max 20 pts):
  filledRatio = employeesWithActivePositionAssignment / activePositions
    — employeesWithActivePositionAssignment = count of employees with
      employmentStatus = ACTIVE and a non-null positionId
    — activePositions = count of positions with status = ACTIVE
    — filledRatio capped at 1.0 (more active employees than active
      positions does not exceed the factor cap)
  capacityFactor = round(filledRatio * 20)
  If activePositions is 0: capacityFactor = 10 (neutral midpoint — see
    Decision 7, this also reduces confidence)

Vacancy Pressure Factor   (max 30 pts):
  Calls VacancyRiskService.score(tenantId, {}) — the exact same governed
  call the vacancy-risk endpoint makes — and computes:
    avgVacancyRisk = mean(riskScore) across all items returned
    (0 eligible vacancies → avgVacancyRisk = 0, i.e. no pressure, full points)
  vacancyPressureFactor = round((100 - avgVacancyRisk) / 100 * 30)

Certification Compliance Factor   (max 20 pts):
  complianceRatio = ACTIVE employee_certifications /
                     (ACTIVE + EXPIRED + REVOKED) employee_certifications
                     assigned tenant-wide
  complianceFactor = round(complianceRatio * 20)
  If denominator is 0 (no certifications assigned tenant-wide):
    complianceFactor = 10 (neutral midpoint — see Decision 7, this also
    reduces confidence)
```

**readinessScore:**

```text
readinessScore = min(100, staffingFactor + capacityFactor +
                            vacancyPressureFactor + complianceFactor)
```

The maximum achievable score is 100 (30 + 20 + 30 + 20).

**factors array:**

The service must return a `factors` array itemizing each factor's
contribution, using the same `RiskFactor`-shaped entries GD-M30-1 Decision 8
defined (the interface is named generically — `name` / `contribution` /
`detail` — precisely so it is reusable here without a new type):

```json
[
  { "name": "staffingCoverage", "contribution": 27, "detail": "90% of current workforce active" },
  { "name": "positionCapacity", "contribution": 18, "detail": "90% of active positions filled" },
  { "name": "vacancyPressure", "contribution": 21, "detail": "Moderate vacancy risk across open positions" },
  { "name": "certificationCompliance", "contribution": 17, "detail": "85% of certifications active" }
]
```

Unlike vacancy risk, all four factors are always present in this array
(never omitted for a zero contribution) — a workforce readiness score with
a missing factor would misleadingly look more complete than it is; the
reader must always see all four inputs that composed the score.

## Decision 6 — Score Range and Readiness Classification Thresholds

`readinessScore` is always an integer in the range 0–100 inclusive.

**Readiness classification thresholds:**

| readinessScore range | readinessLevel |
|---|---|
| 0–24 | `CRITICAL` |
| 25–49 | `AT_RISK` |
| 50–74 | `DEVELOPING` |
| 75–100 | `READY` |

These labels are deliberately distinct from the vacancy-risk LOW/MEDIUM/
HIGH/CRITICAL scale — readiness and risk run in opposite directions (a
high readiness score is good; a high risk score is bad), and reusing the
same label set for both would invite misreading. These thresholds are
governed values and must not be adjusted in implementation without a new
governance decision.

**No pagination, no filtering:** the endpoint returns exactly one score per
request — the current tenant-wide readiness snapshot. No `?departmentId`
or similar filter parameter exists in M31 (see Exclusions — FR-411
department/agency breakdown is explicitly deferred).

## Decision 7 — Confidence Behavior and Data Insufficiency Handling

`confidence` is an integer 0–100 representing how reliable the computed
`readinessScore` is given available input data, following the same
philosophy as GD-M30-1 Decision 7.

**Confidence rules:**

```text
Base confidence = 100.

Apply the following reductions, each independently, compounding
multiplicatively (not additively) when more than one applies:

  activePositions == 0
    → confidence × 0.5
    (Position Capacity Factor defaulted to neutral midpoint per Decision 5)

  total certifications assigned tenant-wide == 0
    → confidence × 0.7
    (Certification Compliance Factor defaulted to neutral midpoint)

  current workforce count (activeEmployees + pendingOnboarding + onLeave +
  suspended) < 3
    → confidence × 0.6
    (small-sample statistical unreliability)

Final confidence = max(10, floor(compounded result))
  — confidence is never reported as 0; a floor of 10 signals "score
    exists but is very low confidence" rather than implying no score
    was computed at all.
```

**Data insufficiency rules (defensive, runtime):**

```text
Tenant has zero employees, zero positions, and zero vacancies:
  readinessScore is still computed (all four factors default to their
  neutral midpoints per Decision 5) — the service must never throw.
  confidence will be at its compounded floor in this case.
  This is not an error condition — return the score with low confidence,
  not a 4xx/5xx response.

Any unexpected data shape (e.g., a certification record with a status
outside ACTIVE/EXPIRED/REVOKED):
  Log a warning; exclude that single record from its ratio's numerator
  and denominator; never crash the response — matching GD-M30-1 Decision 7's
  "never throw an unhandled exception" doctrine.
```

## Decision 8 — Explainability Response Contract

Every workforce readiness response must include all of the following
fields, reusing `IntelligenceExplainabilityOutput` from GD-M30-1 Decision 8
verbatim — no new interface is created:

```typescript
// Reused from apps/api/src/intelligence/interfaces/intelligence-explainability.interface.ts
// (GD-M30-1 Decision 8 — unchanged)
export interface IntelligenceExplainabilityOutput {
  riskScore: number;        // repurposed here as readinessScore — see note below
  riskLevel: string;        // repurposed here as readinessLevel
  confidence: number;
  reasoning: string;
  factors: RiskFactor[];
  computedAt: string;
  formulaVersion: string;   // 'readiness-deterministic-v1' for M31
}
```

**Naming note (implementation guidance, not a new interface):** the shared
interface's field names (`riskScore`, `riskLevel`) were written generically
enough to serve as the foundation for future services per GD-M30-1 Decision 8,
but read awkwardly for a "higher is better" signal. The
`WorkforceReadinessResponseDto` returned to callers must expose these as
`readinessScore` and `readinessLevel` in its own field names (the DTO maps
the shared interface's output onto readiness-appropriate field names at the
response boundary) — the shared interface itself is not renamed, to avoid
touching M30's contract.

**Full response shape for GET /api/v1/intelligence/workforce-readiness:**

```json
{
  "success": true,
  "data": {
    "readinessScore": 76,
    "readinessLevel": "READY",
    "confidence": 90,
    "reasoning": "Workforce readiness is READY, driven primarily by strong staffing coverage and low vacancy pressure, offset slightly by certification compliance below target.",
    "factors": [
      { "name": "staffingCoverage", "contribution": 27, "detail": "90% of current workforce active" },
      { "name": "positionCapacity", "contribution": 18, "detail": "90% of active positions filled" },
      { "name": "vacancyPressure", "contribution": 21, "detail": "Moderate vacancy risk across open positions" },
      { "name": "certificationCompliance", "contribution": 17, "detail": "85% of certifications active" }
    ],
    "computedAt": "2026-07-16T14:23:00.000Z",
    "formulaVersion": "readiness-deterministic-v1"
  }
}
```

The `reasoning` field must be dynamically composed from the actual factor
values at request time — it must never be a hardcoded template string
regardless of input, matching GD-M30-1's reasoning-display doctrine
(GD-M30-1 Decision 10 amendment).

**This response shape is identical for every allowed role, including
Executive User** — see Decision 11.

## Decision 9 — AI Audit Event Policy

Every call to `GET /api/v1/intelligence/workforce-readiness` must generate
an audit event, extending the pattern GD-M30-1 Decision 9 established.

**New AuditEventType value to add:**

```typescript
// In apps/api/src/audit/enums/audit-event-type.enum.ts
// Added to the existing Intelligence Events block (GD-M30-1 Decision 9):

INTELLIGENCE_WORKFORCE_READINESS_QUERIED = 'INTELLIGENCE_WORKFORCE_READINESS_QUERIED',
```

**Audit event data to record:**

```typescript
await this.auditService.logEvent({
  tenantId:   actor.tenantId,
  userId:     actor.userId,
  entityType: 'intelligence',
  entityId:   undefined,              // aggregate; no single entity
  action:     AuditEventType.INTELLIGENCE_WORKFORCE_READINESS_QUERIED,
  result:     'SUCCESS',
  metadata: {
    formulaVersion: 'readiness-deterministic-v1',
    readinessLevel: result.readinessLevel,
    confidence:     result.confidence,
  },
});
```

The audit event metadata must never include individual employee data,
vacancy titles, certification holder names, or any row-level detail — only
the aggregate `readinessLevel`, `confidence`, and `formulaVersion`. This is
easier to guarantee here than in M30: because `WorkforceReadinessService`
never queries or returns row-level records to begin with, there is no
individual data available to leak into the audit payload by mistake.

## Decision 11 — Executive User Aggregate-Only Access (Aggregation Guarantee)

This decision satisfies GD-M30-1 Decision 11's requirement that "any future
EU intelligence endpoint must declare its aggregation guarantee in
governance before implementation."

**The aggregation guarantee:**

`GET /api/v1/intelligence/workforce-readiness` returns exactly one
tenant-wide aggregate object per request, for every allowed role, with no
role-conditional field stripping. This guarantee holds structurally, not
just by convention: `WorkforceReadinessService` never queries for or
returns row-level `Employee`, `Vacancy`, `Position`, or
`EmployeeCertification` records — every value it produces is already a
ratio, a count, or a derived score computed across the whole tenant. There
is no individual-level data in the response for *any* role to redact,
because none is ever fetched into memory in individually-identifiable form
in the first place.

**Executive User receives:**

```text
The identical response shape and identical data as System Administrator,
HR Director, and Workforce Planner — readinessScore, readinessLevel,
confidence, reasoning, factors, computedAt, formulaVersion.
```

**Executive User must not receive, in M31 or as a side effect of M31:**

```text
No individual employee record or list
```

## Decision 12 — No External AI/LLM in M31

M31 does not call OpenAI or any external AI/LLM provider under any
circumstance. This is an absolute constraint, restated unchanged from
GD-M30-1 Decision 12:

```text
Prohibited in M31:
  OpenAI SDK import or client instantiation
  Any HTTP request to api.openai.com or any AI provider endpoint
  Prompt template execution of any kind
  LLM-generated text in any API response or UI element
  Any runtime cost incurred by an AI provider call
  Any autonomous staffing decision or recommendation execution
```

Rationale: identical to GD-M30-1 Decision 12 — M31 must remain fully
self-contained and deployable without external API keys or external
service availability. The deterministic readiness formula produces
explainable results without LLM support. Any future milestone proposing
OpenAI integration must submit its own governance decision covering prompt
versioning, cost tracking, failure handling, and confirmation that core
platform operations remain functional if the AI provider is unavailable —
none of that applies to M31, which introduces no such dependency.
