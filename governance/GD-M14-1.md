# Context

GD-M13-2 Decision 1 established the following controller structure:

```text
CertificationController   → /api/v1/certifications         (certification catalog management)
SkillController           → /api/v1/skills                 (skill catalog management)
EmployeeController        → /api/v1/employees/:id/*        (employee-scoped sub-resources)
```

M14 requires a cross-employee query: retrieve all employee-certification assignment
records within the tenant where status is ACTIVE and expirationDate falls within a
requested lookahead window (GD-M13-3 D4; CRT-400). This query spans all employees;
it cannot be expressed as an employee-scoped sub-resource under `/employees/:id/`.

The M14 endpoint returns `workforce.employee_certifications` rows enriched with
employee fields (employeeNumber, firstName, lastName) and certification catalog fields
(certificationName, issuer). It does not return certification catalog records from
`workforce.certifications`.

An architecture review conducted prior to this recording evaluated three candidate
paths and resolved in favor of a collection-level namespace for employee-certification
assignment queries. That decision is recorded here.

---

# Decisions

## Decision 1 — Namespace Reservation: /api/v1/certifications

`/api/v1/certifications` is reserved exclusively for certification catalog management
operations (CRUD on `workforce.certifications`). No cross-employee assignment query
endpoints shall be added under this path.

## Decision 2 — Namespace Reservation: /api/v1/employees/:id/certifications

`/api/v1/employees/:id/certifications` remains the exclusive path for employee-scoped
certification assignment operations (single employee; existing POST and GET endpoints
per GD-M13-2).

## Decision 3 — New Namespace: /api/v1/employee-certifications

`/api/v1/employee-certifications` is approved as the collection-level namespace for
cross-employee queries on the `workforce.employee_certifications` table. This path
represents the employee-certification assignment collection without scoping to a
single employee.

A new `EmployeeCertificationsController` shall serve this path, registered in
`WorkforceModule`. This extends GD-M13-2 Decision 1 — it does not replace it.

## Decision 4 — M14 Endpoint

M14 shall implement:

```text
GET /api/v1/employee-certifications/expiring
```

Query parameters:

```text
withinDays  integer  min 1, max 365, default 30
page        integer  min 1, default 1
pageSize    integer  min 1, max 100, default 20
```

The endpoint returns a paginated list of employee-certification assignment records
where:

```text
employee.tenantId = JWT tenantId
AND employee.deletedAt IS NULL
AND status = 'ACTIVE'
AND expirationDate IS NOT NULL
AND expirationDate <= (today + withinDays days)
```

Results are sorted by expirationDate ASC. ACTIVE records with expirationDate in the
past are included (CRT-400: past-due ACTIVE certifications surface in expiration
tracking queries).

tenantId is derived from the JWT exclusively. It is never accepted from query
parameters, route parameters, or request body (GD-M13-2 D13; SEC-003).

## Decision 5 — RBAC

```text
GET /api/v1/employee-certifications/expiring:
  Authorized:   System Administrator, HR Director, Workforce Planner, Compliance Officer
  Unauthorized: Recruiter, Executive User → HTTP 403
```

Compliance Officer read access is required for FR-153 expiration report generation
(established in GD-M13-2 Decision 12). Executive User is excluded per RBAC-952
(employee-level records; Decision 12).

# Scope Boundary

This decision authorizes one read-only GET endpoint for M14.

It does not authorize:

```text
Automated ACTIVE → EXPIRED status transitions
Notification or alert generation
Scheduled or background jobs
Compliance reports or export endpoints (CSV, PDF)
Dashboard or metrics aggregation endpoints
Position linkage work (GD-PRE-M13-002)
Position certification matching (GD-M13-5)
Competency framework work (FR-152)
Any write endpoint under /api/v1/employee-certifications
Any additional GET endpoint under /api/v1/employee-certifications beyond /expiring
```

Future endpoints under `/api/v1/employee-certifications` (e.g., `/revoked`,
`/recently-renewed`) require a separate governance decision.

# Impact

`CertificationController` is unchanged.
