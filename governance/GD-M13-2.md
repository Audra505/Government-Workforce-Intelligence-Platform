# Context

spec/06_api_contracts.md defines four employee-scoped endpoints for skills and
certifications but does not define catalog management endpoints for creating or
listing skills and certifications as standalone resources. EMP-601 (GD-M12-4)
mandates that M13 deliver Skills catalog CRUD API and Certifications catalog CRUD API.
This decision establishes the authoritative endpoint design, request contracts, RBAC,
and behavioral constraints for the complete M13 API surface. It supplements spec/06
for catalog management endpoints, which are not present in that document.

---

# Decisions

## Decision 1 — Controller Structure

Skills and Certifications catalog management shall be implemented as two separate
NestJS controllers registered in WorkforceModule:

```text
SkillController         — handles /api/v1/skills routes
CertificationController — handles /api/v1/certifications routes
```

Employee-scoped assignment endpoints shall be implemented in EmployeeController:

```text
POST /api/v1/employees/{id}/skills
GET  /api/v1/employees/{id}/skills
POST /api/v1/employees/{id}/certifications
GET  /api/v1/employees/{id}/certifications
```

## Decision 2 — Skills Catalog Endpoints

```text
GET   /api/v1/skills          — list skills (tenant-scoped, paginated)
POST  /api/v1/skills          — create skill
GET   /api/v1/skills/{id}     — get skill by ID
PATCH /api/v1/skills/{id}     — update skill
```

These endpoints are authorized by EMP-601 (GD-M12-4) and supplement spec/06.

## Decision 3 — Certifications Catalog Endpoints

```text
GET   /api/v1/certifications          — list certifications (tenant-scoped, paginated)
POST  /api/v1/certifications          — create certification
GET   /api/v1/certifications/{id}     — get certification by ID
PATCH /api/v1/certifications/{id}     — update certification
```

These endpoints are authorized by EMP-601 (GD-M12-4) and supplement spec/06.

## Decision 4 — POST /api/v1/skills Request Contract

```text
Required: name (string, max 255 characters)
Optional: category (string, max 100 characters)
Optional: description (string, text)
```

tenant_id: derived from JWT; never accepted from request body.

## Decision 5 — POST /api/v1/certifications Request Contract

```text
Required: name (string, max 255 characters)
Required: expirationRequired (boolean)
Optional: issuer (string, max 255 characters)
```

tenant_id: derived from JWT; never accepted from request body.

## Decision 6 — POST /api/v1/employees/{id}/skills Request Contract

```text
Required: skillId (UUID) — must reference an existing skill within the same tenant.
Optional: proficiencyLevel (string) — must be one of the canonical values defined
          in directives/14_skill_management_rules.md if provided.
Optional: verifiedAt (ISO 8601 timestamp) — when null or absent, the assignment is
          self-declared (unverified). When present, the assignment is HR-verified.
          The verifiedAt value is critical for future skill matching weight per
          directives/05_skill_matching_rules.md SKM-402.
```

## Decision 7 — POST /api/v1/employees/{id}/certifications Request Contract

```text
Required: certificationId (UUID) — must reference an existing certification within
          the same tenant.
Optional: status (string) — must be one of the values defined in GD-M13-3.
          When omitted, defaults to ACTIVE per GD-M13-3 Decision 2.
Optional: issueDate (ISO 8601 date).
Optional: expirationDate (ISO 8601 date).
```

When the referenced certification has expirationRequired = true:
  expirationDate is required.
  Omission returns HTTP 422, error code: EXPIRATION_DATE_REQUIRED.

When both issueDate and expirationDate are provided:
  expirationDate must be >= issueDate.
  Violation returns HTTP 422, error code: INVALID_DATE_RANGE.

## Decision 8 — Skill Name Uniqueness

Skill names must be unique within a tenant.

Duplicate skill name within the same tenant:

```text
HTTP 409, error code: SKILL_NAME_CONFLICT
```

## Decision 9 — Certification Name Uniqueness

Certification names must be unique within a tenant.

Duplicate certification name within the same tenant:

```text
HTTP 409, error code: CERTIFICATION_NAME_CONFLICT
```

## Decision 10 — EMP-302 Enforcement on Employee Assignment Endpoints

POST /api/v1/employees/{id}/skills and POST /api/v1/employees/{id}/certifications
must verify that the target employee's employment_status is not SEPARATED before
performing any write operation.

A request targeting a SEPARATED employee:

```text
HTTP 422, error code: EMPLOYEE_SEPARATED
```

This check applies regardless of the requesting user's role. Authority: EMP-302.

## Decision 11 — Upsert Semantics on Assignment Endpoints

Per GD-M13-4, workforce.employee_skills and workforce.employee_certifications use
composite primary keys. Assignment endpoints use upsert semantics:

```text
First assignment:  INSERT — emits ASSIGNED audit event.
Repeat assignment: UPDATE — emits UPDATED audit event.
```

No error is returned for repeat assignments; the record is updated with the new
field values and the appropriate audit event is emitted.

## Decision 12 — RBAC Matrix

```text
POST  /api/v1/skills                              — SA, HR Director
PATCH /api/v1/skills/{id}                         — SA, HR Director
GET   /api/v1/skills                              — SA, HR Director, Workforce Planner, Compliance Officer
GET   /api/v1/skills/{id}                         — SA, HR Director, Workforce Planner, Compliance Officer

POST  /api/v1/certifications                      — SA, HR Director
PATCH /api/v1/certifications/{id}                 — SA, HR Director
GET   /api/v1/certifications                      — SA, HR Director, Workforce Planner, Compliance Officer
GET   /api/v1/certifications/{id}                 — SA, HR Director, Workforce Planner, Compliance Officer

POST  /api/v1/employees/{id}/skills               — SA, HR Director
GET   /api/v1/employees/{id}/skills               — SA, HR Director, Workforce Planner, Compliance Officer

POST  /api/v1/employees/{id}/certifications       — SA, HR Director
GET   /api/v1/employees/{id}/certifications       — SA, HR Director, Workforce Planner, Compliance Officer

Unauthorized for all M13 endpoints: Recruiter, Executive User
Unauthorized response: HTTP 403
```

Compliance Officer read access is required for audit, policy verification, and
FR-153 expiration report generation.

Executive User is excluded from all employee-scoped M13 endpoints per RBAC-952.
RBAC-952 prohibits Executive User access to individual employee records.
Employee skills and certifications are employee-level records.

## Decision 13 — SEC-003 Enforcement

All M13 endpoints derive tenant_id from the JWT exclusively. tenant_id must never
be accepted from a request body, query parameter, or route parameter on any endpoint.
Cross-tenant records return HTTP 404. Response is identical for absent and cross-tenant
records. No tenant enumeration. Authority: GD-M13-1, SEC-003.

## Decision 14 — GET /api/v1/employees/{id}/skills Response Contract
(Added: 2026-06-20 — Step 4 planning gap resolution; approved: Product Owner — M13 Step 4 Governance Resolution)

The GET /api/v1/employees/{id}/skills endpoint returns a flat, non-paginated array of all
skill assignment records for the specified employee within the requesting tenant.

### Response shape

```json
{
  "success": true,
  "data": {
    "skills": [
      {
        "skillId":          "<UUID string>",
        "skillName":        "<string | null>",
        "skillCategory":    "<string | null>",
        "proficiencyLevel": "<BEGINNER|DEVELOPING|PROFICIENT|ADVANCED|EXPERT | null>",
        "verifiedAt":       "<ISO 8601 string | null>"
      }
    ]
  }
}
```

### Field rules

```text
skillId           — always present; the skill catalog UUID
skillName         — workforce.skills.name for the referenced skill;
                    null only if the referenced skill has been soft-deleted (see SKL-004)
skillCategory     — workforce.skills.category; null if category was not set on the skill
proficiencyLevel  — one of the five OPM canonical values (SKL-210), or null if proficiency
                    was not assessed at assignment time
verifiedAt        — ISO 8601 timestamp when HR-verified (SKL-211); null if self-declared
```

### Excluded fields

```text
tenantId     — never in responses (SEC-003, Decision 13)
employeeId   — redundant; present in route URL
description  — prose metadata; available via GET /api/v1/skills/{id}
Timestamps   — not columns on workforce.employee_skills (GD-M13-4 Decision 2)
```

### Pagination

Not required. The response returns the complete set of skill assignments for the specified
employee as a flat array. Employee sub-resource collections are bounded and must be returned
as a complete set to support skill matching computation (directives/05 SKM-402, SKM-700)
and compliance profile review (FR-153). Paginated responses would require multi-call
assembly by the Phase 3 matching engine and are incompatible with the < 5 second matching
performance target in directives/05.

### Catalog data embedding

The service implementation MUST JOIN workforce.skills to resolve skillName and skillCategory.
This follows the established embedding pattern for sub-resource responses (reference:
EmployeeRecord.departmentName). A response containing only junction table UUIDs is
insufficient for all authorized read roles (Workforce Planner, HR Director, Compliance
Officer) and for the Phase 3 skill matching engine. skillDescription is excluded: it is
prose metadata used at catalog management time, not at matching or compliance review time.

### Soft-deleted skill behavior

The service MUST NOT filter the JOIN to workforce.skills by workforce.skills.deleted_at.
Assignments referencing soft-deleted skills are preserved in the response; skillName and
skillCategory will be null for such records. skillId is always returned to support audit
reconciliation. Note: SKL-004 currently prevents soft-delete of skills with active employee
assignments, making this scenario rare in M13 but defined for future correctness.

### Rationale

Embedding skillName and skillCategory is required by:
- FR-113 acceptance criterion "Skill matching supported" — the Phase 3 matching engine
  (directives/05 SKM-402) must identify which skill is held and whether it is HR-verified
  without N+1 secondary API calls
- Workforce Planner and HR Director profile review — UUID-only responses cannot be rendered
  without secondary catalog lookups
- Compliance Officer audit and FR-153 report generation — regulatory output requires
  human-readable skill identifiers and categories
- API consistency — GET /api/v1/employees/{id} embeds departmentName via JOIN; the same
  pattern applies to skill assignment sub-resource responses

## Decision 15 — HTTP Status Differentiation for Assignment Endpoint Upsert Outcomes
(Added: 2026-06-20 — Step 4 planning gap resolution; approved: Product Owner — M13 Step 4 Governance Resolution)

POST /api/v1/employees/{id}/skills and POST /api/v1/employees/{id}/certifications use
upsert semantics (Decision 11). The HTTP response status differentiates INSERT from UPDATE:

```text
First assignment (INSERT path):   HTTP 201 Created
Repeat assignment (UPDATE path):  HTTP 200 OK
```

Rationale: HTTP 201 signals resource creation to API consumers and caching infrastructure.
HTTP 200 on repeat assignment is consistent with REST conventions for endpoint-native upsert
where the resource already exists. The EmployeeSkillService discriminated union communicates
INSERT vs UPDATE path to the controller, which maps to the appropriate HTTP status code.

The same rule applies to POST /api/v1/employees/{id}/certifications (M13 Step 5).

## Decision 16 — GET /api/v1/employees/{id}/certifications Response Contract
(Added: 2026-06-20 — Step 5 planning gap resolution; approved: Product Owner — M13 Step 5 Governance Resolution)

The GET /api/v1/employees/{id}/certifications endpoint returns a flat, non-paginated array of all
certification assignment records for the specified employee within the requesting tenant.

### Response shape

```json
{
  "success": true,
  "data": {
    "certifications": [
      {
        "certificationId":   "<UUID string>",
        "certificationName": "<string | null>",
        "issuer":            "<string | null>",
        "status":            "<ACTIVE | EXPIRED | REVOKED>",
        "issueDate":         "<YYYY-MM-DD string | null>",
        "expirationDate":    "<YYYY-MM-DD string | null>"
      }
    ]
  }
}
```

### Field rules

```text
certificationId    — always present; the certification catalog UUID from the junction table;
                     never null

certificationName  — workforce.certifications.name for the referenced certification;
                     null only if the referenced certification has been soft-deleted

issuer             — workforce.certifications.issuer for the referenced certification;
                     null if issuer was not set at catalog creation time, or if the
                     referenced certification has been soft-deleted

status             — the current certification assignment status; one of the three canonical
                     values defined in GD-M13-3 Decision 1 (ACTIVE, EXPIRED, REVOKED);
                     never null; required by GD-M13-3 Decision 4

issueDate          — the date the employee received or was issued this certification;
                     serialized as YYYY-MM-DD (date-only string, not ISO 8601 datetime);
                     null if not recorded at assignment time

expirationDate     — the date on which this certification expires;
                     serialized as YYYY-MM-DD (date-only string, not ISO 8601 datetime);
                     null if not recorded at assignment time;
                     required in response by GD-M13-3 Decision 4 and CRT-400 for FR-153
```

### Date serialization

issueDate and expirationDate are stored as DATE (not TIMESTAMP) in the database schema
(spec/05, @db.Date in Prisma schema). This differs from verifiedAt on employee skill
assignments, which is @db.Timestamptz. The service implementation must serialize issueDate
and expirationDate as YYYY-MM-DD strings using UTC-based extraction
(toISOString().substring(0, 10)). A response returning a full datetime string (e.g.,
"2026-06-15T00:00:00.000Z") for issueDate or expirationDate is non-conforming.

### Excluded fields

```text
expirationRequired — catalog property, not an assignment property; available via
                     GET /api/v1/certifications/{id}; not required for compliance
                     profile review or Phase 3 matching
tenantId           — never in responses (SEC-003, Decision 13)
employeeId         — redundant; present in route URL
```

### Pagination

Not required. The response returns the complete set of certification assignments for the
specified employee as a flat array. Employee sub-resource collections are bounded.
Paginated responses are incompatible with:
(a) FR-153 expiration compliance review — Compliance Officers must see the complete
    certification profile without multi-call assembly
(b) Phase 3 AI matching engine (directives/05 SKM-302) — matching requires all held
    certifications in a single response to evaluate certification completeness for
    position requirements

### Catalog data embedding

The service implementation MUST JOIN workforce.certifications to resolve certificationName
and issuer for each assignment record. This follows the established embedding pattern for
sub-resource responses (reference: EmployeeRecord.departmentName; Decision 14 skillName).

A response containing only certificationId is insufficient for all authorized read roles
(Workforce Planner, HR Director, Compliance Officer) and for Phase 3 matching.

certificationName is required for:
- Compliance Officer audit and FR-153 report generation (FR-151 "Certification Name")
- Workforce Planner certification profile review without secondary catalog lookups
- Phase 3 matching engine certificate identification

issuer is required for:
- Compliance Officer verification of certification authority (FR-151 "Issuing Authority")
- Regulatory audits distinguishing certifications by issuing body
- API consistency — FR-151 records Issuing Authority as a first-class certification field

### Soft-deleted certification behavior

The service implementation MUST NOT filter the JOIN to workforce.certifications by
workforce.certifications.deleted_at. Certification assignment records referencing
soft-deleted catalog entries are preserved in the response. For such records:
- certificationId is always returned (supports audit reconciliation)
- certificationName will be null
- issuer will be null

Note: CRT-004 currently prevents soft-delete of certifications with active employee
assignments, making this scenario rare in M13. The behavior is defined here for
future correctness when inactive assignments may reference later-deleted catalog entries.

### Rationale

Embedding certificationName and issuer is required by:
- FR-114 acceptance criterion "Certification status visible" — status alone is
  insufficient without the certification name and issuer for compliance display
- FR-151 "Issuing Authority" as a defined certification data field — issuing body is
  a compliance-grade field required in audit contexts
- FR-153 "Expiration Alerts" and "Compliance Notifications" — regulatory output
  requires human-readable certification identifiers with issuing authority
- API consistency — GET /api/v1/employees/{id} embeds departmentName; Decision 14
  embeds skillName and skillCategory; the same pattern applies here
- Phase 3 matching (directives/05 SKM-302) — the matching engine must identify held
  certifications without N+1 secondary catalog calls

# Spec Deviation

Decisions 2 and 3 (catalog management endpoints) are not present in spec/06.
Their addition is authorized by EMP-601 (GD-M12-4) and this decision.
This decision is the authoritative API contract for M13 catalog management endpoints,
supplementing spec/06_api_contracts.md.

Decisions 14 and 15 resolve implementation gaps identified during M13 Step 4 planning
review (2026-06-20). They supplement spec/06, which defines the GET and POST
/employees/{id}/skills endpoint stubs without request/response contract details.

Decision 16 resolves an implementation gap identified during M13 Step 5 planning and
governance review (2026-06-20). It supplements spec/06, which defines the GET and POST
/employees/{id}/certifications endpoint stubs without request/response contract details.
Decision 16 is the authoritative GET contract for the certification assignment
sub-resource endpoint, establishing the complete field set, date serialization format
(YYYY-MM-DD — distinct from ISO 8601 datetime used by verifiedAt in skills),
pagination determination, catalog embedding rationale, and soft-deleted catalog entry
behavior.
