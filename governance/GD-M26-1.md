# Background

## Backend Gap — GET /api/v1/roles

No roles endpoint exists anywhere in the NestJS API. The identity.roles table
is a static seed table containing the 7 authoritative platform roles:

```text
System Administrator
HR Director
Workforce Planner
Recruiter
Hiring Manager
Compliance Officer
Executive User
```

These 7 rows are populated at database seed time. Their UUIDs are generated at
seed time and are environment-specific — they differ between dev, staging, and
production. The frontend cannot safely reference role UUIDs without an API
endpoint to discover them at runtime.

## Backend Gap — Assignable Role Enforcement

The current UsersController applies @RequireRoles('System Administrator', 'HR
Director') at the class level. This correctly restricts who can call user
management endpoints. However, UsersService.createUser() does not inspect the
actor's roles to determine which target roles the actor is permitted to assign
to a new user. Any actor that reaches createUser() — whether SA or HRD — can
currently supply any valid role UUID in the roleIds array, including the System
Administrator role UUID.

This means an HR Director can currently promote a new user to System
Administrator, which violates the principle of least privilege. M26 must patch
this gap in the backend service before the Create User frontend is built.

## Frontend Gap — Create User

As of M25, the Admin Workspace user management surface is read-only:

```text
Exists:
  /admin/users           — user list, paginated (SA + HRD)
  /admin/users/[id]      — user detail, read-only (SA + HRD)

Missing:
  /admin/users/new       — Create User form (does not exist)
  "New User" button      — does not exist on /admin/users
  POST /api/users BFF    — does not exist
```

## Spec Alignment

```text
FR-001  User Registration and account management — partially implemented
        (backend create exists; frontend create missing until M26)
FR-003  Role-Based Authorization management UI — partial
        (role assignment at user creation; role editing deferred to M27+)
```

# Decisions

## Decision 1 — Scope Boundary: Create User Completion

M26 resolves the documented GD-M25-1 Decision 4 blocker via Option A: a new
GET /api/v1/roles endpoint. With role UUIDs dynamically available from the API,
the Create User form can populate a role selector from live backend data without
hardcoding any UUIDs in frontend source.

```text
M26 deliverables:
  Backend:  GET /api/v1/roles (new endpoint)
  Backend:  Assignable role enforcement in UsersService (patch)
  BFF:      POST /api/users (new route)
  Frontend: /admin/users/new (new page)
  Frontend: CreateUserForm component (new component)
  Frontend: "New User" button on /admin/users (modification)

M26 does not deliver:
  PATCH /api/v1/users/:id (user editing/status/role changes — M27)
  Any user write operation other than create (M27+)
  Audit log viewer (Phase 5)
  Agency management write operations (no backend)
```

Rationale: The GD-M25-1 Decision 4 deferral stated exactly two prerequisites
for Create User to proceed: either Option A (GET /api/v1/roles endpoint) or
Option B (CreateUserDto accepts role names instead of UUIDs). Option A is the
lower-risk path — it adds a read-only endpoint that does not change the DTO
contract or require any schema migration. Choosing Option A preserves the
existing POST /api/v1/users contract and its test coverage while adding the
missing discovery mechanism the frontend requires.

## Decision 2 — GET /api/v1/roles Endpoint Definition

A new GET /api/v1/roles endpoint is added to UsersController. The endpoint
returns the set of platform roles that the authenticated actor is permitted to
assign to a new user.

```text
Endpoint:
  GET /api/v1/roles

Placement:
  UsersController — consistent with other user management operations;
  no new controller or module required in M26

RBAC:
  JwtAuthGuard + RolesGuard (class-level guards already on UsersController)
  Route-level @RequireRoles: System Administrator, HR Director
  Workforce Planner, Recruiter, Hiring Manager, Compliance Officer,
  Executive User — receive HTTP 403

Response shape:
  {
    success: true,
    data: {
      roles: Array<{ id: string; name: string }>
    }
  }

Response content — filtered by actor role:
  System Administrator actor:
    Returns all 7 roles in alphabetical order by name:
      Compliance Officer
      Executive User
      Hiring Manager
      HR Director
      Recruiter
      System Administrator
      Workforce Planner

  HR Director actor:
    Returns 6 roles — System Administrator is excluded:
      Compliance Officer
      Executive User
      Hiring Manager
      HR Director
      Recruiter
      Workforce Planner

  (System Administrator is never returned to HRD — this is the same
   restriction enforced by Decision 3 on the create path)

Implementation:
  Service method: RolesService (new injectable in UsersModule) or inline
  in UsersService — implementation choice deferred to the implementing
  engineer; both are valid as long as the behavior matches this spec.
  Query: prisma.role.findMany({ select: { id, name }, orderBy: { name: 'asc' } })
  Filtered for HRD: WHERE name != 'System Administrator'

No pagination:
  The role set is static (7 rows). Pagination would add complexity
  without benefit. The full set is always returned in a single response.

No schema change:
  identity.roles already exists and is populated at seed time. No
  migration is required.
```

Rationale: Returning all roles to SA and a filtered set to HRD at the API
layer is the cleanest approach because it makes the filtering authoritative at
the backend and eliminates the possibility of the frontend accidentally
displaying System Administrator to HRD through a caching or race condition.
The frontend role selector is populated directly from this response — what the
API returns is exactly what can be assigned.

## Decision 3 — Assignable Role Policy and Backend Enforcement

The assignable role policy for user creation is:

```text
System Administrator:
  May assign any of the 7 platform roles to a new user.
  May assign System Administrator to a new user.

HR Director:
  May assign any of the 6 non-privileged platform roles to a new user:
    HR Director, Workforce Planner, Recruiter, Hiring Manager,
    Compliance Officer, Executive User
  May NOT assign System Administrator to a new user.
  Attempting to do so must return HTTP 403.

All other roles:
  Cannot call POST /api/v1/users (HTTP 403 from RolesGuard).
```

Backend enforcement in UsersService.createUser():

```text
If actor.roles does NOT include 'System Administrator':
  1. Resolve the System Administrator role UUID from the database.
  2. If any element of dto.roleIds matches the System Administrator UUID:
     return { outcome: 'FORBIDDEN_ROLE_ASSIGNMENT', forbiddenRoleId: saUUID }

The new discriminated union outcome is:
  | { outcome: 'FORBIDDEN_ROLE_ASSIGNMENT'; forbiddenRoleId: string }

UsersController must map this outcome to HTTP 403:
  throw new ForbiddenException({
    success: false,
    error: {
      code: 'FORBIDDEN_ROLE_ASSIGNMENT',
      message: 'HR Director may not assign System Administrator'
    }
  })

This enforcement is independent of the frontend role selector filtering.
A request that bypasses the frontend (direct API call) must still be
rejected if an HRD actor includes the System Administrator roleId.
This is defense-in-depth.
```

Audit event on FORBIDDEN_ROLE_ASSIGNMENT:

```text
An AUTHZ_ACCESS_DENIED audit event must be emitted when an actor
attempts to assign a role they are not permitted to assign.
This uses the existing AuditEventType.AUTHZ_ACCESS_DENIED enum value.
No new audit event type is required.
```

Rationale: The current backend has no assignable-role enforcement — any HRD
actor can currently escalate a new user to System Administrator via the API.
This is a privilege escalation vulnerability that must be patched before the
Create User form exposes user creation to browser-based actors. Enforcing the
restriction in UsersService rather than UsersController keeps the guard in the
transport-agnostic layer, consistent with the established pattern where
business rules live in the service and HTTP mapping lives in the controller.

## Decision 4 — BFF Route Contract: POST /api/users

A new BFF route is added for the Create User mutation:

```text
New BFF route file:
  apps/web/src/app/api/users/route.ts

Handler: POST

Required behavior:
  1. Parse request body as JSON; return 400 on malformed JSON.
  2. FORBIDDEN_FIELD enforcement (SEC-003 extension):
     If 'tenantId' is present in the request body, return HTTP 400
     immediately and do not forward to NestJS. This is the same
     FORBIDDEN_FIELD pattern established for all M25 BFF routes.
  3. Read JWT from the httpOnly session cookie (SESSION_COOKIE).
     If absent, return HTTP 401.
  4. Forward the request body to POST /api/v1/users with:
       Content-Type: application/json
       Authorization: Bearer <token>
  5. Return NestJS HTTP status and response body unchanged:
       201 Created     — user created; return { success: true, data: { user } }
       400             — validation failure (field errors or ROLE_NOT_FOUND)
       403             — FORBIDDEN_ROLE_ASSIGNMENT (HRD assigning SA)
       409             — EMAIL_CONFLICT
       500             — internal server error
  6. Do not rewrite, suppress, or add fields to any response body.
  7. Do not forward status, appointmentAuthority, or any field other
     than firstName, lastName, email, roleIds, and password to NestJS.
     The BFF must only pass through recognized user creation fields.

Response types:
  apps/web/src/features/admin/types.ts must include:
    CreateUserBffResponse (success true + user data shape)
```

Rationale: The POST /api/users BFF route follows the exact contract policy
established in GD-M25-1 Decision 7 for department BFF routes and GD-M24-1
Decision 8 for skill/certification BFF routes. Consistency in BFF contract
enforcement reduces implementation risk and review surface. The FORBIDDEN_FIELD
check for tenantId is mandatory on all new BFF routes per SEC-003 regardless
of whether the specific mutation would otherwise use it.

## Decision 6 — Password Field Treatment

The password field is a Phase 1 operational substitute for the invitation and
activation email flow. This is the same Phase 1 deviation established in
GD-M25-1 Background (users created as ACTIVE) and documented in
create-user.dto.ts.

```text
Password model:
  The administrator sets the user's initial password at creation time.
  The new user receives no system email or notification (NotificationModule
  not built; FR-700 deferred).
  The new user is created with status ACTIVE (Phase 1 ACTIVE override —
  same as all prior M13/M25 user creation).
  No INVITED status is used for M26-created users.
  No forced password change on first login (no DB support in Phase 1).

The following are explicitly deferred until NotificationModule is built:
  Email-based invitation / activation flow
  Force-change-on-first-login flag
  Temporary password expiration
  Password reset flow (no endpoint, no email infrastructure)

The Create User form must not include:
  A "Send Invitation Email" checkbox
  A "Generate Password" button
  A "Reset Password" link or option
  Any affordance that implies notification delivery is happening

This is consistent with the no-placeholder-panel rule extended to M26
by this document (see Decision 9).
```

Rationale: The Phase 1 password model was established at M13 and does not
change in M26. Documenting it here prevents the implementing engineer from
adding invitation-flow stubs under the assumption that such infrastructure
exists. It also provides a clear hook for a future governance decision when
NotificationModule is implemented.

## Decision 7 — Error Mapping Policy

All server errors from POST /api/v1/users must be surfaced to the operator
with enough information to take corrective action. No error is silently
swallowed or shown as a generic failure.

```text
Error mapping table:

  HTTP 409 from NestJS (EMAIL_CONFLICT)
    BFF forwards as-is (409)
    Form: red bordered banner above form fields
    Text: "A user with this email address already exists in this tenant."
    The email field should not scroll out of view; the banner appears
    near the top of the form

  HTTP 400 from NestJS (ROLE_NOT_FOUND)
    BFF forwards as-is (400)
    Form: red bordered banner above form fields
    Text: "One or more selected roles could not be found. Please reload
    and try again."
    Cause: a selected roleId is no longer in the database; should not
    occur in practice if roles are fetched fresh on page load

  HTTP 400 from NestJS (validation: field-level)
    BFF forwards as-is (400)
    Form: banner with the backend validation message
    These are normally caught by Zod before submission; banner is
    defensive for any gap between frontend and backend validation rules

  HTTP 403 from NestJS (FORBIDDEN_ROLE_ASSIGNMENT)
    BFF forwards as-is (403)
    Form: red bordered banner
    Text: "You do not have permission to assign one or more of the
    selected roles."
    Cause: HRD submitted System Administrator roleId; indicates
    a frontend bug or direct API call bypass

  HTTP 500 from NestJS
    BFF forwards as-is (500)
    Form: red bordered banner
    Text: "An unexpected error occurred. Please try again."

  Network failure (no response)
    Form: same red bordered banner
    Text: "Unable to reach the server. Please check your connection
    and try again."
```

Rationale: Truthful error propagation is a core requirement of the BFF
contract (GD-M25-1 Decision 7 extended to M26). The administrator must
know what went wrong and how to correct it. Generic "something failed"
messages are not acceptable on a government HR platform where user
provisioning has direct operational consequences.

## Decision 8 — Audit Events

M26 introduces no new AuditEventType enum values. The events emitted on
user creation are already defined and used by M13 UsersService:

```text
On successful user creation (already implemented in UsersService):
  IDENTITY_USER_CREATED   — one event per created user
  AUTHZ_ROLE_ASSIGNED     — one event per role assigned to the user

On FORBIDDEN_ROLE_ASSIGNMENT attempt (new in M26):
  AUTHZ_ACCESS_DENIED     — one event emitted by UsersService when
  an actor attempts to assign a forbidden role; already defined in
  AuditEventType enum (AUD-300)
```

No changes to audit-event-type.enum.ts are required in M26. The
AUTHZ_ACCESS_DENIED event type was defined in the AUD-300 block
at M4 and has not been used in the user domain until now.

Rationale: Reusing existing AuditEventType values wherever the semantics
match is the correct approach — expanding the enum unnecessarily creates
maintenance burden and compliance reporting complexity. AUTHZ_ACCESS_DENIED
precisely describes an attempt to assign a role the actor is not permitted
to assign.

```text
UsersController — E2E tests (additional to M13 coverage):
  POST /api/v1/users with HRD token + SA roleId → 403
  POST /api/v1/users with SA token + SA roleId → 201

Existing M13 tests must not be broken by M26 changes.
```

Rationale: The GET /roles endpoint and the assignable-role restriction are
new behaviors with no existing test coverage. Both must be verified at the
unit and E2E layers before the frontend is built against them. The frontend
relies on GET /roles returning exactly the correct filtered set — a wrong
implementation at the backend would silently expose System Administrator as
an assignable role to HRD.
