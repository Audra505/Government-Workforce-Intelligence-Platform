# Background

## M21 Status Indicator Doctrine (Superseded)

GD-M21-1 Decision 10 established a two-pattern status indicator doctrine:

```text
Pattern A — Dot + Label: entity lifecycle status (one active state at a time)
Pattern B — Pill badge: severity, priority, multi-dimensional values

Under D10, all entity lifecycle statuses (Position, Employment, Vacancy,
Candidate, Application, Interview, Offer) must use the dot+label format.
Pill badges were prohibited on lifecycle status values.
```

This doctrine was correct for the initial Workforce UI reconciliation pass.
However, during M23 platform review, filled status pills were determined to be
superior for production platform use:

```text
Dot+label limitations observed in use:
- Small colored circle is hard to distinguish at a glance on 48px rows
- Color alone carries all visual weight — accessible contrast is harder to
  guarantee on the canvas background
- Dot+label is common in utility/admin tools but not in government platform UX

Filled status pill advantages:
- Pill shape + color + label text provide three independent recognition cues
- Clearly distinguishable across employment status values at a glance
- Matches the visual register of production government HR platforms
- Consistent with how recruiting pipeline statuses have always been displayed
  in commercial HR tools
```

M23 supersedes GD-M21-1 D10 for lifecycle status indicators across all platform
list and detail table views.

## Dashboard Maturity Gap

The M22 dashboard shows four global count KPI cards. WF-D6 in GD-M21-1
classified the dashboard as "Placeholder only; no real content yet."

M23 closes WF-D6 by delivering a full real-data dashboard summary with:
- KPI cards: workforce status, open vacancies, active positions, total candidates
- Workforce status panel: stacked proportional bar + rate calculation
- Open vacancies panel: top-3 by priority
- Recruiting pipeline panel: application counts by current status
- Interview and offer summary statistics
- Expiring certifications panel (RBAC-gated — shown only to roles with access)

All data is fetched in parallel via `Promise.allSettled` — no single failed
fetch can crash the dashboard. Widgets that depend on inaccessible data are
silently omitted. No fake data is ever shown.

# Decisions

## Decision 1 — Status Indicator Doctrine Supersession

**This decision supersedes GD-M21-1 Decision 10 for entity lifecycle status
indicators across all platform list and detail table views.**

The platform adopts filled status pills as the single pattern for entity
lifecycle status display. The dot+label Pattern A from GD-M21-1 D10 is
retired for lifecycle statuses.

```text
New canonical pattern — Filled Status Pill:
  Shape: rounded-full inline-flex with min-width
  Content: label text only (no dot)
  Typography: font-weight 500, font-size 11-12px, uppercase or title case
  Color encoding: background + text color pair per status value

Vacancy priority indicator (PriorityBadge) is NOT affected by this change.
Priority is severity-grade data, not lifecycle status. PriorityBadge continues
to use the pill format established in GD-M21-1 D10 Pattern B.

AgingCell is NOT affected. It carries numeric + severity data.

VAC-601 requiresReview badge is NOT affected. It is an alert modifier.
```

Rationale: Filled pills provide three independent recognition cues (shape,
color, text) versus the dot+label two-cue pattern. This improves at-a-glance
status scanning on high-density government HR data tables.

The StatusDot primitive and domain wrapper components (PositionStatusDot,
EmploymentStatusDot, VacancyStatusDot, etc.) created in M21 are superseded
by StatusPill and its domain wrappers for lifecycle status display. StatusDot
may remain in the codebase for other uses but must not be used for new
lifecycle status rendering.

## Decision 2 — Dashboard Data Fidelity Policy

The dashboard must only display data obtained from real, supported API
endpoints at request time.

```text
Rule 1 — No fake data:
  Static/hardcoded values are prohibited as KPI card content.
  Every numeric value on the dashboard must come from a serverFetch call.

Rule 2 — No placeholder panels:
  Dashboard sections that depend on data not yet supported by the backend
  must be omitted entirely. They must NOT appear as:
    - "Coming soon" banners
    - Mockup panels with fake or illustrative data
    - Greyed-out placeholder UI
  Omission is the correct behavior until backend support exists.

Rule 3 — Graceful degradation:
  Dashboard data is fetched via Promise.allSettled. A failed fetch for one
  widget must not crash the page or suppress other widgets.
  If a required fetch fails or returns an error, the affected widget is either
  omitted or shown with a clear unavailable indicator — never shown with
  stale, cached, or fake data.

Rule 4 — RBAC-gated widgets:
  If a widget depends on data from an RBAC-restricted endpoint (e.g.,
  GET /api/v1/employee-certifications/expiring), and the current user's
  role does not have access (403 response), the widget is silently omitted.
  The dashboard must not crash or display an error for a 403 on a widget fetch.
```

Rationale: Government HR platforms must not mislead operators with dashboard
data that does not reflect real system state. A dashboard showing fake or
illustrative values creates a false impression of system health and could
result in incorrect operational decisions.

## Decision 3 — Platform Navigation Consistency Standard

The platform header navigation must be visually identical across all three
domain shells: Dashboard page, WorkforceShell, and RecruitingShell.

```text
Active nav item treatment:
  - Rounded pill: rounded-[5px]
  - Active background: rgba(255,255,255,0.12) — inline style
  - Text: text-white, font-medium, text-[13px]
  - Padding: px-[13px] py-[6px]

Inactive nav item treatment:
  - Same pill shape and padding
  - Text: text-white/50, font-medium, text-[13px]
  - Hover: bg-white/[0.08] text-white/[0.85], transition-all

Nav item spacing: gap-0.5 between items

Active item implementation note:
  The Dashboard page uses a <span> element for the active "Dashboard" item
  (not a <Link>) because the user is already on the dashboard page.
  WorkforceShell and RecruitingShell use <Link> for their active item.
  Both approaches produce the same visual output. Both are correct.

GWIP wordmark:
  - Dashboard: text-white Tailwind class (resolves to #ffffff)
  - WorkforceShell: inline style={{ color: '#ffffff' }}
  - RecruitingShell: inline style={{ color: '#ffffff' }}
  Both approaches render identically. No change required.
```

Rationale: Users navigating between Dashboard, Workforce, and Recruiting must
perceive a single consistent navigation system. Token-for-token consistency
eliminates visual discontinuity at domain transitions.

## Decision 4 — Font Application Rule

IBM Plex Sans must be applied explicitly on the outermost div of every shell
component and standalone page that renders the platform header.

```text
Required fontFamily value:
  fontFamily: "var(--font-ibm-plex-sans, 'IBM Plex Sans', system-ui, sans-serif)"

Applied to:
  - WorkforceShell outer div (already in place from M21)
  - RecruitingShell outer div (already in place from M21)
  - Dashboard page.tsx outer div (added in M23)

Reason this is required:
  apps/web/src/app/layout.tsx loads IBM Plex Sans as CSS variable
  --font-ibm-plex-sans only. The <body> element receives font-sans
  (Tailwind system stack) — not the IBM Plex variable. Without an explicit
  fontFamily on the shell/page outer div, the browser renders in the
  system stack (e.g., Segoe UI on Windows), creating a visual mismatch
  between the Dashboard and Workforce/Recruiting pages.
```

Rationale: A government platform must have consistent typography across all
authenticated surfaces. The system stack vs. IBM Plex San discrepancy is
imperceptible in screenshots but immediately visible to users navigating
between the Dashboard and other pages.

## Decision 5 — Shared StatusPill Component

A shared `StatusPill` component is introduced at:
`apps/web/src/components/shared/status-pill.tsx`

```text
Component: StatusPill
Props:
  color:   string  — text color hex
  bgColor: string  — background color hex
  label:   string  — display text

Usage:
  Domain-specific wrappers in workforce and recruiting components pass the
  correct color/bgColor/label for each status value.
  The shared primitive is the single rendering contract for filled status pills.

Replaces:
  - PositionStatusBadge (workforce)
  - EmploymentStatusBadge (workforce)
  - VacancyStatusBadge (workforce, in vacancy-badges.tsx)
  (The dot+label wrappers from M21 — PositionStatusDot, EmploymentStatusDot,
  VacancyStatusDot — are superseded for lifecycle status rendering.)

Retained (not replaced):
  - PriorityBadge (vacancy-badges.tsx) — severity-grade pill, not lifecycle
  - AgingCell (vacancy-badges.tsx) — numeric + severity display
  - VAC-601 requiresReview badge — alert modifier
```
