  entities record when a row last changed, not what any computed score
  was at a past moment. A trend requires point-in-time history that does
  not exist anywhere in this schema today.

  real computed history over time. Building that accumulation mechanism
  now — without displaying a single trend or forecast from it until real
  history exists — is the only honest way to move toward FR-412/FR-400
  without fabricating data.

The drilldown's conclusion, adopted by this decision: combine the one
capability that is fully ready today (Executive Intelligence Expansion)
with the minimal, honest infrastructure prerequisite for the two that are
not (a workforce signal snapshot table), because the same requests that
serve Executive Intelligence Expansion's metrics are also useful
write-on-query opportunities to begin snapshotting the tenant's existing
signals. Neither FR-412 nor FR-400 is claimed complete by this decision.

# Background

## Current Intelligence State

Four deterministic signals exist (Vacancy Risk, Workforce Readiness,
Attrition Risk, Department Gap), all computed fresh on every request, none
persisted. Executive User currently receives two aggregate cards
(Workforce Readiness, Attrition Risk) plus the dashboard's ungated 4-card
KPI row (Active Workforce, Active Positions, Unfilled Vacancies, Critical
Vacancies — raw counts, no rates or percentages).

## What FR-404 Actually Requires, And What Is Genuinely Missing

FR-404 names five metrics: Vacancy rate, Time to fill, Workforce capacity,
Coverage percentage, Attrition risk. Attrition risk is done (M32).
Workforce capacity is effectively covered by the existing Workforce
Readiness score. The remaining four — Vacancy Rate %, Coverage Rate %,
Time To Fill, Hiring Velocity — are the genuine gap, and every input each
one needs already exists as live, queryable data.

## Why the Snapshot Table Is Foundation, Not Feature

This decision does not implement Workforce Trend Analysis (FR-412) or
Demand Forecasting (FR-400). It creates exactly one additive table whose
sole purpose is to begin accumulating real computed history, so that a
future milestone can build trend/forecast features from real data instead
of choosing between fabricating history or shipping nothing. No endpoint,
UI element, chart, or claim of trend/forecast capability is authorized by
this decision. The table may exist, accumulate rows, and be queried by no
one, for the entire duration of M34 — that is the intended, honest state.

# Decisions

## Decision 1 — Scope Boundary: Two Things, Combined Because They Share a Trigger, Not Because They Share a Purpose

M34 authorizes exactly two things:

```text
1. Executive Intelligence Expansion — four new deterministic, explainable
   metrics (Vacancy Rate %, Coverage Rate %, Time To Fill, Hiring
   Velocity), visible today, computed from existing data, closing the
   FR-404 gap GD-M30-1 Decision 11 named for "M34" by role.
2. Workforce Snapshot Foundation — one additive table that begins
   accumulating real computed history from the five existing/new
   Intelligence signals, as the shared prerequisite for a future FR-412
   (Trend Analysis) and/or FR-400 (Demand Forecasting) milestone.
```

These are combined into one milestone not because they answer the same
product question (they do not — one is a visible metric, the other is
invisible infrastructure) but because the same request path that computes
Executive Intelligence Expansion's metrics is also a natural, already-
happening moment to persist a snapshot row, avoiding the need to invent a
separate trigger mechanism just to seed the snapshot table. Building the
snapshot foundation in total isolation, with nothing else to show for the
milestone, would be a harder milestone to verify and demo; building
Executive Intelligence Expansion without at least starting the snapshot
foundation would mean this milestone's request volume is lost as
historical seed data for no reason.

Neither sub-scope authorizes the other's future capability: Executive
Intelligence Expansion does not become a trend feature by having its
values snapshotted, and the snapshot table does not become a forecast
feature by existing.

## Decision 3 — Tenant Isolation and No Caller-Supplied Parameters

Identical tenant-isolation pattern to every prior Phase 4 endpoint:

```typescript
@Get('executive-metrics')
@UseGuards(JwtAuthGuard, RolesGuard)
@RequireRoles('System Administrator', 'HR Director', 'Workforce Planner', 'Executive User')
async getExecutiveMetrics(@CurrentUser() actor: RequestUser) {
  return this.executiveMetricsService.getByTenant(actor.tenantId);
}
```

`tenantId` is never accepted from the request body, query string, path
parameters, or headers — always taken from `actor.tenantId`. The endpoint
accepts no query parameters, consistent with `GD-M32-1` Decision 3 and
`GD-M33-1` Decision 3's shared rationale that a caller-suppliable filter
on an aggregate-only endpoint invites drill-down misuse even where, as
here, there is no individual-level data to drill down into — the "no
parameters" rule is restated for consistency, not because a new specific
risk was identified for this endpoint.

## Decision 5 — Formula: Vacancy Rate %

```text
vacancyRatePercent = (openVacancyCount / totalActivePositionCount) * 100,
  rounded to 1 decimal place

openVacancyCount:
  Vacancy.count({ tenantId, deletedAt: null,
                   status: { in: ['OPEN', 'IN_RECRUITMENT'] } })
  — identical status-value convention VacancyRiskService already uses
    (GD-M30-1 Decision 5)

totalActivePositionCount:
  Position.count({ tenantId, deletedAt: null, status: 'ACTIVE' })
  — identical convention WorkforceReadinessService's Position Capacity
    factor already uses (GD-M31-1 Decision 5)

If totalActivePositionCount == 0: vacancyRatePercent is null (division
undefined), and this factor's confidence contribution is reduced per
Decision 9. This is not clamped to 0 — a null result must be rendered as
"insufficient data," never as a fabricated 0%.

No trailing window — this is a point-in-time ratio of two live counts, not
a rate over a time period.
```

## Decision 6 — Formula: Coverage Rate %

```text
coverageRatePercent = (employeesWithActivePosition / totalActivePositionCount)
  * 100, rounded to 1 decimal place

employeesWithActivePosition:
  Employee.count({ tenantId, deletedAt: null, employmentStatus: 'ACTIVE',
                    positionId: { not: null } })
  — identical query WorkforceReadinessService's Position Capacity factor
    already runs (GD-M31-1 Decision 5); this decision exposes the same
    ratio independently rather than only as one input buried inside the
    readiness score

totalActivePositionCount: identical to Decision 5

If totalActivePositionCount == 0: coverageRatePercent is null, confidence
reduced per Decision 9, never fabricated as 0% or 100%.

No trailing window — point-in-time ratio, same rationale as Decision 5.
```

## Decision 7 — Formula: Time To Fill

```text
Governed constant: TIME_TO_FILL_WINDOW_DAYS = 365 (identical trailing
  window length to AttritionRiskService's governed window, GD-M32-1
  Decision 5 — reused for consistency, not derived from it)

timeToFillDays = average of (filledAt - createdAt) in days, across every
  vacancy filled within the trailing window, rounded to 1 decimal place

Input query:
  Vacancy.findMany({
    where: { tenantId, deletedAt: null, filledAt: { gte: windowStart } },
    select: { createdAt: true, filledAt: true },
  })

If zero vacancies were filled in the trailing window: timeToFillDays is
null, confidence floored per Decision 9, never fabricated as 0.

filledVacancyCount (the sample size) is reported alongside the value in
the factor detail text (Decision 10) so a reader can judge how much the
average reflects — this is explainability, not a second metric.
```

## Decision 8 — Formula: Hiring Velocity

```text
Governed constant: HIRING_VELOCITY_WINDOW_DAYS = 90 (deliberately shorter
  than Time To Fill's 365-day window — "velocity" is a recent-pace signal,
  not a long-run average; both windows are governed constants, not
  independently meaningful beyond "the window this decision assigns them")

hiringVelocityCount = count of employees whose hireDate falls within the
  trailing 90-day window, tenant-wide

Input query:
  Employee.count({
    tenantId, deletedAt: null,
    employmentStatus: { in: ['ACTIVE', 'ON_LEAVE', 'PENDING_ONBOARDING', 'SUSPENDED'] },
    hireDate: { gte: windowStart },
  })
  — reuses AttritionRiskService's CURRENT_WORKFORCE_STATUSES constant
    (GD-M32-1 Decision 5) to exclude hires who have since separated within
    the same window, keeping this a "currently-employed recent hires"
    count, not a lifetime hire-event count

Reported as a raw count with its window explicit in the factor detail
("N hires in the last 90 days"), not normalized into a per-day or
per-month rate — normalizing a small count invites a misleadingly precise
decimal (e.g., "0.078 hires/day") that adds no explanatory value over the
plain count. A count of 0 is a fully valid, non-null result — unlike
Decisions 5-7, this metric has no undefined-denominator case, so it is
never null.
```

## Decision 9 — Confidence Rules

Each of the four metrics reports its own `confidence` (0-100), following
the same multiplicative-reduction, floor-at-10 pattern `GD-M31-1` Decision
7 and `GD-M32-1` Decision 7 established — restated per metric because the
insufficiency conditions differ:

```text
Vacancy Rate % / Coverage Rate % (Decisions 5, 6):
  confidence = 100 if totalActivePositionCount > 0
  confidence = 10  if totalActivePositionCount == 0 (floor, not zero)

Time To Fill (Decision 7):
  Let n = filledVacancyCount in the trailing window.
  confidence = 100 if n >= 10
  confidence = 70  if 5 <= n < 10
  confidence = 40  if 1 <= n < 5
  confidence = 10  if n == 0 (floor, not zero; value is null per Decision 7)

Hiring Velocity (Decision 8):
  confidence = 100 always — a count-based metric with no undefined-
  denominator case has no data-insufficiency condition to model; a result
  of 0 hires is itself a fully confident, fully explainable answer.
```

No confidence value below 10 is ever returned, matching the floor already
governed for every existing Phase 4 signal.

## Decision 10 — Explainability Response Contract

`ExecutiveMetricsService.getByTenant(tenantId)` returns an
`executive-metrics-response.dto.ts`-shaped payload. This decision
introduces a metric-level shape alongside (not replacing)
`IntelligenceExplainabilityOutput` (`GD-M30-1` Decision 8), because these
four values are plain rates/counts, not classified risk scores with a
LOW/MEDIUM/HIGH/CRITICAL level — forcing a risk-level classification onto
"Time To Fill = 34 days" would invent a threshold this decision does not
govern and the product requirement does not ask for.

```json
{
  "success": true,
  "data": {
    "vacancyRate": {
      "value": 12.0, "unit": "PERCENT", "confidence": 100,
      "detail": "6 of 50 active positions are currently vacant.",
      "windowDays": null
    },
    "coverageRate": {
      "value": 82.0, "unit": "PERCENT", "confidence": 100,
      "detail": "41 of 50 active positions are filled by an active employee.",
      "windowDays": null
    },
    "timeToFill": {
      "value": 34.2, "unit": "DAYS", "confidence": 70,
      "detail": "Average 34.2 days to fill, based on 8 vacancies filled in the last 365 days.",
      "windowDays": 365
    },
    "hiringVelocity": {
      "value": 7, "unit": "COUNT", "confidence": 100,
      "detail": "7 employees hired in the last 90 days.",
      "windowDays": 90
    },
    "computedAt": "ISO-8601 timestamp",
    "formulaVersion": "executive-metrics-deterministic-v1"
  }
}
```

Any metric whose value is `null` (Decisions 5-7's undefined-denominator or
zero-sample cases) must still include its `confidence` (floored at 10) and
a `detail` string explaining why — never an omitted field, never a
fabricated numeric value in its place. `formulaVersion:
'executive-metrics-deterministic-v1'` identifies this decision's formula

## Decision 14 — Snapshot Foundation: Purpose and Explicit Non-Goals

This is the decision that governs what the snapshot table is *for*, and —
more importantly — what it is explicitly **not** for in M34.

```text

  (Vacancy Risk, Workforce Readiness, Attrition Risk, Department Gap,
  Executive Metrics) so that a FUTURE, separately-governed milestone can
  build Workforce Trend Analysis (FR-412) and/or a Demand Forecasting
  foundation (FR-400) from real accumulated data, never from fabricated
  or backfilled history.

Explicit non-goals of M34 (restated from the Not In Scope section for
emphasis, because this is the decision most likely to be misread as
authorizing more than it does):
  This decision does NOT implement Workforce Trend Analysis. No trend is

    dashboard or workspace surface may communicate to a user. The
    Capability Maturity language for both, when PROGRESS.md is updated in

    "in progress" and not "partially implemented" — because no feature

  No historical data is backfilled at any point by this decision.
    Snapshot accumulation begins the moment M34 ships and not one moment
    earlier — there is no synthetic seed data, no reconstruction of past
    scores from historical entity timestamps, and no estimate presented
```

## Decision 15 — Snapshot Table Schema

A new Prisma schema, `intelligence`, is added to the datasource `schemas`
list (already anticipated by `schema.prisma`'s own header comment). This
is the only schema-level change this decision authorizes, and the only
new table is the one below — additive only, no change to any existing
model.

```prisma
// datasource db { schemas = [..., "intelligence"] }  — one addition to the existing array

model WorkforceSignalSnapshot {
  id             String    @id @default(uuid()) @db.Uuid
  tenantId       String    @map("tenant_id") @db.Uuid
  signalType     String    @map("signal_type") @db.VarChar(50)
  // 'VACANCY_RISK' | 'WORKFORCE_READINESS' | 'ATTRITION_RISK' |
  // 'DEPARTMENT_GAP' | 'EXECUTIVE_METRICS' — VARCHAR per platform
  // convention (no Prisma enums), governed value set, not user input
  scopeType      String    @map("scope_type") @db.VarChar(20)
  // 'TENANT' | 'DEPARTMENT'
  scopeId        String?   @map("scope_id") @db.Uuid
  // departmentId when scopeType = 'DEPARTMENT' (Department Gap signal
  // only); null when scopeType = 'TENANT'. Storing this is not a new
  // exposure — the same departmentId is already returned to the same
  // allowed roles by the existing department-gap endpoint (see Decision 18)
  snapshotDate   DateTime  @map("snapshot_date") @db.Date
  // UTC calendar day derived from computedAt — the deduplication key
  // component (Decision 17), not a free-standing timestamp
  score          Decimal?  @db.Decimal(6, 2)
  // the single headline numeric value only (riskScore / readinessScore /
  // attritionScore / a department-gap entry's score / one executive
  // metric's value) — nullable, matching Decisions 5-7's null cases
  level          String?   @map("level") @db.VarChar(50)
  // riskLevel / readinessLevel / attritionRiskLevel where applicable; null
  // for Executive Metrics, which have no level classification (Decision 10)
  confidence     Int       @db.SmallInt
  formulaVersion String    @map("formula_version") @db.VarChar(100)
  computedAt     DateTime  @map("computed_at") @db.Timestamptz(6)
  createdAt      DateTime  @default(now()) @map("created_at") @db.Timestamptz(6)

  @@unique([tenantId, signalType, scopeType, scopeId, snapshotDate],
           map: "idx_workforce_signal_snapshots_dedup")
  @@index([tenantId, signalType], map: "idx_workforce_signal_snapshots_tenant_signal")
  @@map("workforce_signal_snapshots")
  @@schema("intelligence")
}
```

```text
Deliberately excluded from this table:
  No factors/reasoning columns — factor-level detail is not needed to
    accumulate a trend line or a forecast input series; storing it for
    every snapshot would multiply storage for no stated future use and
    would need its own privacy review this decision does not undertake
  No employee-level, candidate-level, or any row-level identifier of any
    kind — every row is an aggregate or department-scoped aggregate value,
    matching the aggregation guarantee every source signal already carries
  No deletedAt / soft-delete column — snapshot rows are immutable,
    insert-or-update-only historical records, matching the audit_events
    table's own immutability principle; a snapshot is never deleted, only
    superseded by a later day's row (Decision 17)
  No updatedAt column in the general sense — the row IS updated in place
    exactly once per calendar day per Decision 17's upsert rule, but this
    is deliberately not exposed as a generic "last modified" concept; the
    row's createdAt reflects its first write for that day, and
    computedAt reflects the most recent upsert's computation time

No migration is created by this decision. This code block describes the
schema an implementation step will apply; per this task's explicit
instruction, no migration file is generated as part of authoring this
governance decision.
```

## Decision 16 — Snapshot Persistence Rule (Write-On-Query)

```text
No scheduler, cron job, or worker process is introduced (confirmed absent
from this codebase today — no services/worker directory, no
@nestjs/schedule, no node-cron anywhere in apps/api).

Instead, snapshot persistence rides on requests that already happen:
every time one of the five governed signal services completes a
successful computation — VacancyRiskService.score(), WorkforceReadiness
Service.score(), AttritionRiskService.score(), DepartmentGapService.
getByTenant() (once per non-suppressed department entry), or
ExecutiveMetricsService.getByTenant() (once per metric) — it calls
SnapshotWriterService with the computed headline value(s). This is
entirely internal: no field is added to any of these methods' existing
return types, no caller of these methods observes any behavioral
difference, and no new HTTP round-trip is introduced.

Suppressed Department Gap entries (GD-M33-1 Decision 6) are NOT
snapshotted — there is no score to snapshot, and snapshotting the fact
that a department was suppressed on a given day would itself begin
accumulating a form of small-population history this decision's own
non-goals (Decision 14) do not authorize building toward.

If the snapshot write fails for any reason (database error, constraint
violation), the triggering request's own response is NOT affected — the
signal computation and its return to the caller succeed independently of
snapshot persistence. Snapshot writes are logged-and-continue, never
blocking, matching this platform's general failure-handling doctrine
(spec/11_ai_architecture.md's AI Failure Handling section: core operations
continue when a non-critical subsystem fails).
```

## Decision 17 — Snapshot Deduplication Rule

**This is the safeguard that makes write-on-query safe.** Without it, a
dashboard or workspace refreshed repeatedly within a single day — which
this platform's existing usage pattern makes routine — would create one
snapshot row per HTTP request, growing the table unboundedly with no
added historical value (a signal recomputed five times in one hour has
not actually changed five times).

```text
Deduplication key: (tenantId, signalType, scopeType, scopeId,
                     snapshotDate)
  — enforced as a database-level UNIQUE constraint (Decision 15), not
    merely an application-level check, so the guarantee holds even under
    concurrent requests

snapshotDate = the UTC calendar date derived from the moment of
  computation (DATE-typed, not a timestamp) — granularity is one row per
  signal per scope per tenant per day, maximum

Write path: UPSERT, not INSERT
  If no row exists for today's (tenantId, signalType, scopeType, scopeId)
    combination: insert one.
  If a row already exists for today: UPDATE it in place with the latest
    computed score/level/confidence/formulaVersion/computedAt — the
    snapshot always reflects the MOST RECENT computation of that day, not
    the first.
  This bounds growth to at most (number of distinct signal/scope
    combinations for the tenant) rows PER TENANT PER DAY — for a tenant
    with N departments, at most 3 (Vacancy Risk, Workforce Readiness,
    Attrition Risk, tenant-scoped) + 1 (Executive Metrics, tenant-scoped)
    + up to 2*N (Department Gap readiness + attrition, department-scoped,
    non-suppressed departments only) rows per day — a small, predictable,
    linear-in-department-count bound, never unbounded by request volume.

This rule is implemented exactly once, in SnapshotWriterService (Decision
2), and called identically by all five signal services — no service
re-implements or varies the dedup logic.
```

## Decision 18 — Snapshot Privacy and No-New-Exposure Guarantee

```text
No read endpoint, query API, export, or admin view of any kind exposes
WorkforceSignalSnapshot data to any user, role, or system in M34. The
table is written to and read by nothing except the SnapshotWriterService
that writes it. This decision authorizes zero new information becoming
visible to any user as a result of this table's existence.

Every value stored in a snapshot row is already, independently, visible
to the same allowed roles through the existing (or this decision's new)
signal endpoint at the moment it was computed — a snapshot never contains
a value more granular, more identifying, or more sensitive than what the
originating endpoint already returned to an allowed caller. Department
Gap's suppression rule (GD-M33-1 Decision 6) is preserved: a suppressed
department contributes no snapshot row at all (Decision 16), so the
suppression guarantee is not weakened by this table's existence.

Any future decision proposing to read this table back into an endpoint or
UI (the eventual FR-412/FR-400 milestone) must independently re-verify
this no-new-exposure guarantee at that time, because reading ACROSS
snapshot rows (a trend) is a materially different exposure question than
writing one row at a time — this decision's guarantee covers write-time
exposure only and does not pre-authorize any future read pattern.
```
