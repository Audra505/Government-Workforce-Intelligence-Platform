# Background

## Backend Domain Status — Users (M13 Identity Layer — Complete)

The user management backend is implemented and CI-confirmed. Three endpoints
are live in the NestJS API:

```text
Users:
  POST   /api/v1/users         — create user (SA + HRD)
  GET    /api/v1/users         — list users, paginated (SA + HRD)
  GET    /api/v1/users/:id     — get user by ID (SA + HRD)
```

No additional user management endpoints exist:

```text
Not implemented:
  PATCH  /api/v1/users/:id     — edit user (does not exist)
  DELETE /api/v1/users/:id     — delete user (does not exist)
  PATCH  /api/v1/users/:id/roles — role reassignment (does not exist)
  POST   /api/v1/users/:id/reset-password — password reset (does not exist)
  GET    /api/v1/roles         — list available roles (does not exist)
```

The `POST /api/v1/users` endpoint requires `roleIds: UUID[]`. Role UUIDs are
environment-specific seed values. Because no `GET /api/v1/roles` endpoint
exists, the frontend cannot dynamically populate a role selector to obtain
the required UUIDs. This blocks Create User in M25 (see Decision 4).

Phase 1 deviation: users are created with `status: 'ACTIVE'` directly,
bypassing the INVITED flow. The NotificationModule (D-011 / FR-700) is not
yet built; activation email is deferred. The `password` field is required at
creation (admin-sets-password-at-creation model) as an operational substitute.

## Backend Domain Status — Departments (M13 Organization Layer — Complete)

The department management backend is implemented and CI-confirmed. Five
endpoints are live:

```text
Departments:
  POST   /api/v1/departments        — create (SA + HRD)
  GET    /api/v1/departments        — list, paginated (SA + HRD + WP)
  GET    /api/v1/departments/:id    — detail (SA + HRD + WP)
  PATCH  /api/v1/departments/:id    — update / deactivate (SA + HRD)

Agency:
  GET    /api/v1/agencies/current   — current agency (all authenticated users)
```

Department deactivation is performed via PATCH with `{ status: 'INACTIVE' }`.
The backend enforces three 422 guard outcomes when active dependents exist:

```text
DEPARTMENT_HAS_ACTIVE_EMPLOYEES   — active employees must be reassigned first
DEPARTMENT_HAS_ACTIVE_POSITIONS   — N active position(s) must be closed or
                                    reassigned first (count included)
DEPARTMENT_HAS_ACTIVE_DEPENDENTS  — N employee(s) and M position(s) must be
                                    resolved first (both counts included)

Guard checks:
  Active employee statuses: PENDING_ONBOARDING, ACTIVE, ON_LEAVE, SUSPENDED
  Active position statuses: DRAFT, ACTIVE, FROZEN
  SEPARATED employees and CLOSED/soft-deleted positions do NOT block deactivation
```

Department reactivation (INACTIVE → ACTIVE) is technically accepted by the
`UpdateDepartmentDto` but is explicitly not supported in Phase 1 per DEP-004.
It must not be exposed in the M25 UI (see Decision 2).

# Decisions

## Decision 1 — Scope Boundary: Admin Workspace Definition

M25 delivers an Admin Workspace: a new top-level platform section for
platform administration. Two sub-domains are delivered:

```text
Department Management:
  Full CRUD: list, create, view detail, edit, deactivate.
  Backend is complete — all four endpoints with full guard logic are live.
  No backend work required.

User Management (read-only):
  List users, view user detail.
  Write operations (create, edit, deactivate, delete, password reset,
  role reassignment) are deferred — backend endpoints do not exist for
  any of them. See Decision 4 for the Create User deferral reason.

Excluded admin surfaces:
  Agency Management (no write endpoints in Phase 1)
  Audit Log viewer (no backend read endpoint)
  System Settings (no backend support)
  AI/matching configuration (Phase 4)
  Competency framework (GD-PRE-PHASE3-004 — Phase 4)
  Notification preferences (NotificationModule not built)
  User import / bulk operations (no backend support)
```

Rationale: The backend domain coverage for departments is complete;
for users it is partial (read-only endpoints exist; write endpoints
do not). M25 scope is bounded by what the current backend fully
supports. Extending scope to operations that have no backend would
require creating stub UX that cannot be fulfilled, which violates
the data fidelity and no-placeholder-panel rules established in
GD-M23-1 and GD-M24-1.

## Decision 2 — Department Management Scope

M25 delivers full CRUD for department management within the Admin Workspace.

```text
In-scope department operations:
  list departments   — paginated table, search by name/code/description,
                       filter by status (ACTIVE | INACTIVE)
  create department  — form with name (required), code (required), description
                       (optional); CODE_CONFLICT (409) mapped to form-level error
  view detail        — department card with name, code, description, status,
                       createdAt; deactivate action available to SA and HRD
  edit department    — pre-filled form with name, code, description; PATCH
                       semantics; CODE_CONFLICT (409) mapped to form-level error
  deactivate         — sets status to INACTIVE via PATCH; triggers guard error
                       display when dependents exist (see Decision 8)

Explicitly excluded department operations:
  delete department  — no DELETE endpoint; no delete actions policy (absent)
  reactivate         — DEP-004 Phase 1 restriction; INACTIVE → ACTIVE must not
                       be exposed in the UI even though UpdateDepartmentDto
                       technically accepts status: 'ACTIVE'; the BFF PATCH route
                       for M25 must only send status: 'INACTIVE' for deactivation
                       and must not forward a status: 'ACTIVE' payload
  fake data          — all department data must come from live API responses
  bypass guards      — deactivation guard errors must be surfaced to the admin;
                       the frontend must never suppress or silently retry on 422
```

The department PATCH BFF route handles two operations via a single endpoint:
edit (name/code/description changes) and deactivate (status change). These
may be sent in the same PATCH request (e.g., rename + deactivate) or
separately. The BFF must forward the request body faithfully to NestJS and
return NestJS responses unchanged, including 422 guard errors.

Rationale: The department backend is complete with full guard logic. M25
closes the entire admin gap for departments. Excluding reactivation is
required both by DEP-004 and by the principle that unsupported Phase 1
operations must not be exposed in the UI, even when the DTO technically
permits them.

## Decision 3 — User Management Scope (Read-Only)

M25 delivers read-only User Management: list users and view user detail.
No write operations may be implemented for users in M25.

```text
In-scope user operations:
  list users       — paginated table; search by email/firstName/lastName;
                     filter by status (ACTIVE | INVITED | SUSPENDED |
                     DEACTIVATED); columns: name, email, status, roles, createdAt
  view user detail — user card with firstName, lastName, email, status,
                     roles (role names as strings), createdAt, lastLoginAt

Response fields available (from GET /api/v1/users and GET /api/v1/users/:id):
  id, firstName, lastName, email, status, roles (string[]), createdAt,
  lastLoginAt (nullable)

Excluded fields (never returned by backend):
  passwordHash, tenantId, failedLoginAttempts, lockedUntil, deletedAt
  (These fields are excluded at the response DTO layer and will never
  appear in the frontend — no special handling required.)

Explicitly excluded user operations (backend absent):
  create user        — deferred (see Decision 4)
  edit user          — no PATCH /api/v1/users/:id endpoint
  deactivate user    — no PATCH /api/v1/users/:id endpoint
  suspend user       — no PATCH /api/v1/users/:id endpoint
  delete user        — no DELETE endpoint; no delete actions policy
  reset password     — no endpoint; NotificationModule not built
  change roles       — no endpoint
  invitation flow    — NotificationModule not built
  activation email   — NotificationModule not built

No affordances for excluded operations:
  The user list and detail pages must not include disabled buttons,
  greyed-out actions, "coming soon" labels, or any UI element that
  implies a write operation is imminent or planned.
  Absent operations are simply absent.
```

Read operations use `serverFetch` directly in Server Components — the
same pattern established for all other read-only pages in the platform.
No new BFF routes are required for user reads.

Rationale: The user read endpoints are complete and functional. The write
endpoints do not exist. Showing disabled or stub write affordances where
no backend exists violates the no-placeholder-panel rule (GD-M23-1 D2
extended in GD-M24-1 D9). This document extends that rule to M25: absent
means absent, not greyed-out.

## Decision 4 — Create User Deferral (Explicit)

Create User is explicitly deferred from M25 and from any future milestone
until the required backend prerequisite is resolved.

```text
Deferral reason:
  POST /api/v1/users requires roleIds: UUID[] in the request body.
  Role UUIDs are environment-specific values set at database seed time.
  No GET /api/v1/roles endpoint exists anywhere in the NestJS API.
  The frontend cannot dynamically obtain role UUIDs without this endpoint.

What is forbidden:
  Hardcoding role UUIDs in frontend source code is explicitly prohibited.
  Role UUIDs are not stable across environments (dev, staging, production
  each have independent seed runs that may produce different UUIDs).
  Any UUID hardcoded in frontend source would be environment-specific,
  fragile, and a maintenance liability.

What Create User requires before it can be built:
  Option A: A new GET /api/v1/roles endpoint is added to NestJS, returning
            at minimum { id: UUID, name: string } for each seeded role.
  Option B: The CreateUserDto is changed to accept role names (strings)
            instead of role UUIDs, with NestJS resolving names to IDs
            internally.
  Either option requires a backend change, which is out of M25 scope.

Future milestone action required:
  When either Option A or Option B is implemented and CI-confirmed,
  a new governance decision (GD-M26-1 or similar) must authorize the
  Create User form, defining: the role selector behavior, the password
  field treatment (Phase 1 operational substitute), password policy
  display (12-char min; uppercase + lowercase + digit + special char),
  email uniqueness error mapping (409 CONFLICT), and whether the INVITED
  flow remains deferred.
```

Authoritative role set (from apps/api/src/platform-roles.spec.ts):

```text
System Administrator
HR Director
Workforce Planner
Recruiter
Hiring Manager
Compliance Officer
Executive User
```

These 7 roles exist in the database as static seed data. Their UUIDs
are not available to the frontend without a roles API endpoint.

Rationale: The roleIds UUID blocker is a hard technical constraint, not
a scheduling choice. Proceeding with hardcoded UUIDs would create a class
of environment-specific bugs that are invisible in development but
destructive in production (user creation fails silently or with a
ROLE_NOT_FOUND 400 error against production role UUIDs). Deferral is the
only correct response.

## Decision 5 — RBAC Policy

Access to Admin Workspace surfaces is governed by the following role matrix,
derived from the user and department endpoint RBAC established in the NestJS
controller layer:

```text
Role                   | Users list | User detail | Dept list | Dept create | Dept edit | Dept deactivate
-----------------------|------------|-------------|-----------|-------------|-----------|----------------
System Administrator   | Yes        | Yes         | Yes       | Yes         | Yes       | Yes
HR Director            | Yes        | Yes         | Yes       | Yes         | Yes       | Yes
Workforce Planner      | No (403)   | No (403)    | Yes       | No (403)    | No (403)  | No (403)
Recruiter              | No (403)   | No (403)    | No (403)  | No (403)    | No (403)  | No (403)
Hiring Manager         | No (403)   | No (403)    | No (403)  | No (403)    | No (403)  | No (403)
Compliance Officer     | No (403)   | No (403)    | No (403)  | No (403)    | No (403)  | No (403)
Executive User         | No (403)   | No (403)    | No (403)  | No (403)    | No (403)  | No (403)

canAdminWrite  = System Administrator OR HR Director
canAdminRead   = System Administrator OR HR Director
canDeptRead    = System Administrator OR HR Director OR Workforce Planner
```

Frontend implementation rules:

```text
Admin nav link:
  The "Admin" top-nav link (alongside Dashboard · Workforce · Recruiting)
  is visible only to System Administrator and HR Director.
  Workforce Planner may navigate directly to /admin/departments via URL
  or other surface links; they do not see the top-level Admin nav item
  because it would imply broad admin access across all Admin tabs.
  All other roles do not see the Admin nav link.

AdminShell tab visibility:
  Users tab: rendered for canAdminRead roles (SA + HRD) only.
  Departments tab: rendered for canDeptRead roles (SA + HRD + WP).

Write action visibility:
  Create, Edit, and Deactivate affordances on department pages:
    rendered only when canAdminWrite (SA + HRD).
  User Management has no write affordances in M25.

Graceful 403 handling:
  If a role without department access navigates to /admin/departments
  and serverFetch receives a 403 from NestJS, the page must render
  gracefully (empty state or role-appropriate message, not a crash).
  If a role without user access navigates to /admin/users and serverFetch
  receives a 403, same graceful treatment applies.
  No page in the Admin Workspace may crash, throw an unhandled error,
  or display a React error boundary for a role-based 403.

Backend enforcement:
  Frontend role gating is UX only, not a security boundary. NestJS
  guards enforce RBAC independently. A request reaching NestJS without
  the correct role will be rejected regardless of frontend gating.
  The frontend must never rely on its own role check as a security
  control.
```

Rationale: The role matrix is derived directly from the NestJS controller
RBAC decorators, not from a frontend assumption. Workforce Planner has
explicit read access to departments in the backend (SA + HRD + WP on
GET endpoints) and must have a path to that data. However, a top-level
Admin nav item visible to WP would mislead them into expecting user
management access they do not have. The cleanest UX boundary is: Admin
nav for canAdminRead (SA + HRD), with WP accessing departments through
a direct route link rather than a top-level Admin tab.

## Decision 7 — BFF Route Contract Policy

Two new BFF route files are introduced in M25. Both must follow the
established BFF contract pattern from GD-M24-1 Decision 8 and
GD-M13-1 (SEC-003).

```text
New BFF route files:
  apps/web/src/app/api/departments/route.ts       — POST (create)
  apps/web/src/app/api/departments/[id]/route.ts  — PATCH (update / deactivate)

Required for all new BFF handlers:
  1. Read JWT from the httpOnly session cookie (same as all existing handlers).
  2. Forward JWT as Authorization: Bearer <token> to NestJS.
  3. SEC-003 enforcement: if tenantId is present in any incoming request body,
     return HTTP 400 immediately and do not forward the request to NestJS.
     tenantId is always derived from the JWT at the NestJS service layer.
     It must never be accepted from the client.
  4. Forward NestJS HTTP status codes and response bodies unchanged.
     Do not rewrite 4xx/5xx errors. Do not suppress validation errors.
     Do not suppress 422 guard errors (see Decision 8).
  5. Do not add, remove, or rewrite response body fields.
  6. The PATCH route for departments must not forward status: 'ACTIVE' in
     the request body (reactivation is not supported — see Decision 2).
     If the incoming client request body contains status: 'ACTIVE', the
     BFF must return HTTP 400 and not forward to NestJS.

Read operations:
  Department reads (GET /api/v1/departments, GET /api/v1/departments/:id)
  and user reads (GET /api/v1/users, GET /api/v1/users/:id) use serverFetch
  directly in Server Components — no BFF route is needed for these calls.
  This is consistent with the existing platform pattern for all read-only
  server component pages.
```

Rationale: Uniform BFF contract enforcement is the mechanism by which
SEC-003 tenant isolation is preserved at the frontend boundary. The
reactivation rejection at the BFF layer (status: 'ACTIVE' → 400) adds
an explicit guard that prevents the DEP-004 unsupported operation from
ever reaching NestJS, even if a client were to construct a request
manually. Defense-in-depth for a documented Phase 1 restriction.

## Decision 8 — Department Deactivation Guard Error Display

The three 422 guard errors from PATCH /api/v1/departments/:id must be
rendered as informational inline banners on the department detail page,
not as generic toast error messages.

```text
Backend guard errors and their required UI treatment:

  DEPARTMENT_HAS_ACTIVE_EMPLOYEES
    Message: "Cannot deactivate department: active employees must be
    reassigned first."
    UI: Inline banner on department detail page. Instructs the admin
    to navigate to Workforce and reassign or transition the listed
    employees before returning to deactivate.

  DEPARTMENT_HAS_ACTIVE_POSITIONS
    Message: "Cannot deactivate department: N active position(s) must be
    closed or reassigned first."
    UI: Inline banner displaying the position count from the error message.
    Instructs the admin to navigate to Workforce and close or reassign
    the listed positions.

  DEPARTMENT_HAS_ACTIVE_DEPENDENTS
    Message: "Cannot deactivate department: N employee(s) and M position(s)
    must be resolved first."
    UI: Inline banner displaying both counts. Instructs the admin to
    resolve both employees and positions before returning to deactivate.

Banner placement:
  Inline alert banner on the department detail page, rendered below the
  department card and above (or replacing) the deactivate button area.
  Must persist until the admin takes an action or navigates away.

Banner vs. toast distinction:
  Toast is appropriate for transient confirmation (successful create,
  successful edit). Toast is not appropriate for guard errors that
  require the admin to take action in another part of the system.
  Guard errors must be persistent inline banners because the admin
  needs to read, understand, and act before they can return to complete
  the deactivation.

NestJS error response shape:
  { success: false, error: { code: string, message: string } }
  The error.message field contains the human-readable text including
  the numeric counts. The frontend may use error.message directly
  in the banner or parse counts from it for richer display.
```

Rationale: Toasts disappear after a fixed duration (4–8 seconds per
the platform standard). A toast for a guard error that requires the
admin to open another page, perform multiple operations, then return
would disappear before the admin has context for what to do. An inline
banner is the correct persistent affordance for errors that communicate
a multi-step resolution path.

## Decision 9 — Status Treatment

Status pills for users and departments must use the StatusPill shared
component established in GD-M23-1 and consistently applied across M24.

```text
Department status:
  ACTIVE   — green    (background: #dcfce7, text: #15803d)
  INACTIVE — gray     (background: #f1f5f9, text: #475569)

User status:
  ACTIVE      — green  (background: #dcfce7, text: #15803d)
  INVITED     — blue   (background: #dbeafe, text: #2563eb)
  SUSPENDED   — amber  (background: #fef9c3, text: #a16207)
  DEACTIVATED — gray   (background: #f1f5f9, text: #475569)
```

User status notes:
In Phase 1, users are created directly as ACTIVE. INVITED and SUSPENDED
statuses may not occur in the current runtime but are valid backend values
(status filter on GET /api/v1/users accepts all four values). The frontend
must render all four statuses correctly whenever they appear — no special
"this status never occurs" handling is warranted. DEACTIVATED records
may exist if a future backend update adds status-change capability.

Department status notes:
INACTIVE departments are returned by the backend when `?status=INACTIVE`
is included in the query. The list page status filter allows toggling
between ACTIVE and INACTIVE views. Both states must be presented with
the correct pill color at all times.

Rationale: StatusPill is the established platform status component since
GD-M23-1. Consistent status color semantics (green=active/healthy,
gray=inactive/deactivated, amber=suspended/at-risk, blue=invited/pending)
allow operators to scan status at a glance without decoding label text.

## Decision 10 — Data Fidelity Policy

GD-M23-1 Decision 2 (Dashboard Data Fidelity Policy), extended to M24
pages by GD-M24-1 Decision 9, extends to all Admin Workspace pages
introduced in M25.

```text
Rule 1 — No fake data:
  Every value displayed on any M25 page must come from a live API
  response at request time. Static or hardcoded user counts, department
  names, department codes, or user role lists are prohibited.

Rule 2 — No placeholder panels:
  Sections for deferred capabilities (Create User, edit user, deactivate
  user, role reassignment, department reactivation, agency management,
  audit log) must be omitted entirely. They must not appear as:
    - "Coming soon" banners
    - Mockup panels with illustrative data
    - Greyed-out or locked UI sections
    - Disabled buttons that imply future functionality
  Omission is the correct behavior until backend support and a governing
  decision exist.

Rule 3 — Graceful degradation:
  Non-fatal fetch failures on any Admin Workspace page must not crash
  the page. Server Component reads that fail must render gracefully
  (empty state or role-appropriate message, not an error boundary crash).

Rule 4 — Role-gated content omission:
  If a fetch fails with 403 (Recruiter or other unauthorized role
  navigating to an Admin page), the content area must be omitted or
  shown with a role-appropriate message. The page must not error.
```

Rationale: Consistent data fidelity across all platform areas is
non-negotiable for a government HR platform where operators rely on
displayed information for workforce decisions. Placeholder UI that is
visible but non-functional erodes trust in the platform and violates
the no-stub-workflows rule established across earlier milestones.

## Decision 12 — UI Consistency Policy

M25 Admin Workspace must visually align with the existing M23/M24 platform
design system. It must not introduce a new visual style, new component
patterns, new spacing system, or new design tokens. Every visual surface
in the Admin Workspace — AdminShell, Users pages, Departments pages, tables,
forms, status pills, buttons, breadcrumbs, page headers, empty states,
loading states, error states, and pagination — must match the established
platform treatment exactly.

### Required alignment

```text
Shell and chrome:
  AdminShell header         — navy background #0c2340; identical structure
                              to WorkforceShell and RecruitingShell
  GWIP wordmark             — same treatment as WorkforceShell and
                              RecruitingShell
  Top-nav pill styling      — rounded-pill active nav treatment token-for-token
                              identical to Dashboard / Workforce / Recruiting /
                              Admin (GD-M23-1 D3 Navigation Consistency Standard)
  Active nav state          — same underline/pill highlight as all other shells
  Hover nav state           — same hover transition as all other shells
  Sign Out pill             — same treatment as WorkforceShell and RecruitingShell
  Font                      — IBM Plex Sans self-hosted, applied on the outer div
                              of AdminShell and every Admin page component
                              (GD-M23-1 D4 Font Application Rule); no new font
                              family, weight, or @font-face declaration
  Canvas background         — #f8fafc on all Admin Workspace pages
  Content max-width         — max-w-[1200px] as in WorkforceShell/RecruitingShell

Page structure:
  Breadcrumb                — same breadcrumb pattern as Workforce/Recruiting pages
  Page title/subtitle       — same heading and subtitle spacing as established
                              M23/M24 pages
  Card border/radius/shadow — same card treatment as Workforce/Recruiting pages:
                              border #e2e8f0, no drop shadow on content cards

Tables (department list, user list):
  Header row                — uppercase 11px, letter-spacing, color #94a3b8
                              (same as M20/M21/M23/M24 tables)
  Row height                — 48px minimum (same as established platform tables)
  Table border color        — #e2e8f0 (same as established platform tables)
  Entity links              — blue #2563eb for clickable department names and
                              user names (same as existing entity links on
                              Workforce and Recruiting tables)

Status indicators:
  Component                 — shared StatusPill (components/shared/status-pill.tsx)
                              established in GD-M23-1; no dot+label indicators
                              for any M25 status fields

Forms (create department, edit department):
  Field/label/input spacing — same layout as M24 skill and certification forms
  Primary action button     — blue #2563eb background (same as M24 submit buttons)
  Cancel button             — outline style with border #e2e8f0 (same as M24
                              cancel buttons)
  Required field marker     — same treatment as existing platform forms
  Error state               — same inline red text below field as M24 forms
  Server error banner       — same red bordered alert (border #fca5a5,
                              background #fef2f2, text #dc2626)

Shared components (mandatory reuse — no equivalent components may be created):
  Pagination                — components/shared/pagination.tsx
  EmptyState                — components/shared/empty-state.tsx
  Toast / error handling    — components/shared/toast.tsx (useToast +
                              ToastContainer); success auto-dismiss 4s,
                              error auto-dismiss 8s

Deactivation guard banners (Decision 8):
  Same bordered informational alert pattern used for non-fatal errors
  elsewhere in the platform; must not be styled as a new component

Loading states:
  Same skeleton or neutral loading indicator pattern as Workforce and
  Recruiting pages; no new loading component introduced
```

### Forbidden in M25 Admin Workspace

```text
Sidebar navigation       — no sidebar; tab bar is the navigation pattern
New card styling         — no new card border-radius, shadow, or background
New table styling        — no new header font, row height, or border color
New button styling       — no new button variants, colors, or border-radius values
New status badge styling — no new status colors, shapes, or badge components
New spacing system       — no new spacing scale, margin, or padding units
New colors               — no hex values outside the established platform palette:
                           NAVY #0c2340, CANVAS #f8fafc, BORDER #e2e8f0,
                           TEXT #0f172a, SUB #475569, MUTED #94a3b8,
                           BLUE #2563eb, ERROR #dc2626
New fonts                — IBM Plex Sans is the only permitted typeface; no
                           new font family, weight, or @font-face declaration
New navigation behavior  — no drawer, dropdown, accordion, or flyout nav;
                           tab bar is the only sub-navigation pattern
Placeholder widgets      — no illustrative or mockup panels for deferred
                           features (GD-M23-1 D2 extended to M25 by Decision 10)
Disabled "coming soon"   — no disabled buttons implying future capability
```

Rationale: The platform has an established visual language set in GD-M20-1
(platform design system), GD-M21-1 (WorkforceShell), and GD-M23-1 (status
pill doctrine, font rule, navigation consistency standard). Admin Workspace
is a new domain but not a new product. Introducing new visual patterns in
M25 would create visual fragmentation across the platform and undermine the
design coherence achieved since M20. Every component and treatment required
above has a working reference implementation already in the codebase.
Reuse is both the correct design decision and the lowest-risk implementation
path.
