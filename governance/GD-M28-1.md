Milestone Context:
M27 User Lifecycle Management is complete and CI-confirmed (implementation

strictly limited: add a compact, non-interactive identity chip to authenticated

# Background

## Current Header State

Every authenticated header in the platform follows this right-side pattern:

```text
[VDIV] [Sign out]
```

The vertical divider and Sign Out button are present in all four shells:
WorkforceShell, RecruitingShell, AdminShell, and the Dashboard page header.
No user name, email, or role is displayed at any point in the authenticated
experience.

## JWT Payload Gap

The current JwtPayload interface in auth.service.ts carries:

```typescript
export interface JwtPayload {
  sub: string;       // userId
  tenantId: string;
  email: string;
  roles: string[];
}
```

`firstName` and `lastName` are absent. They are available in the user record
returned by `IdentityService.validateCredentials()` — the query at identity.
service.ts line 43 uses `include: { userRoles: { include: { role: true } } }`
with no custom `select`, which returns all columns of the `identity.users`
table, including `firstName` and `lastName`. Adding these fields to the JWT
requires only extending the interface and the payload object in
`AuthService.login()`. No changes to `IdentityService` are required.

## Session Utility Duplication

`getSessionRoles()` in apps/web/src/lib/session.ts is the canonical JWT decode
utility on the frontend. It is imported by all three shell components and
approximately 20 additional page components. However, several pages in
apps/web/src/app/(dashboard)/recruiting/ define their own inline copy of the
same base64 decode logic rather than importing the shared utility.

M28 adds `getSessionUser()` to the same file, returning the full available
identity from the JWT. This eliminates the need for any new inline decode
logic. It does not remove existing `getSessionRoles()` call sites — backward
compatibility is mandatory (see Decision 5).

# Decisions

## Decision 1 — Scope Boundary: User Identity Header Only

M28 delivers visible user identity in authenticated headers only. Its entire
deliverable is:

```text
What changes:
  Backend  — JwtPayload + RequestUser extended with firstName + lastName
  Backend  — AuthService.login() encodes firstName + lastName in JWT
  Backend  — JwtStrategy.validate() extracts firstName + lastName
  Frontend — getSessionUser() utility added to lib/session.ts
  Frontend — UserIdentityChip server component created
  Frontend — UserIdentityChip placed in all 4 authenticated headers

What does not change:
  All existing API endpoints and response shapes
  All existing page routes and navigation
  All existing RBAC logic and guards
  All existing BFF routes
  All existing shell tab bars, breadcrumbs, and domain navigation
  PROGRESS.md (updated only after implementation is CI-confirmed)
```

Rationale: The user identity gap in the header is a foundational UX issue
that affects all users across all delivered phases. Resolving it before Phase
4 begins ensures that the Intelligence domain's UI surfaces will be built on
a foundation where users can always confirm who they are. The change is
architecturally minimal — it does not introduce new routing, new API endpoints,
or new RBAC layers.

## Decision 2 — JWT Payload Extension: firstName and lastName

`firstName` and `lastName` shall be added to the `JwtPayload` interface in
`apps/api/src/identity/auth.service.ts` and encoded at token signing time.

```typescript
// Before (current):
export interface JwtPayload {
  sub: string;
  tenantId: string;
  email: string;
  roles: string[];
}

// After (M28):
export interface JwtPayload {
  sub: string;
  tenantId: string;
  email: string;
  firstName: string;
  lastName: string;
  roles: string[];
}
```

The payload construction in `AuthService.login()` shall be updated:

```typescript
// Before (current, login() SUCCESS case, lines 46-50):
const payload: JwtPayload = {
  sub: result.user.id,
  tenantId: result.user.tenantId,
  email: result.user.email,
  roles: result.user.userRoles.map((ur) => ur.role.name),
};

// After (M28):
const payload: JwtPayload = {
  sub: result.user.id,
  tenantId: result.user.tenantId,
  email: result.user.email,
  firstName: result.user.firstName,
  lastName: result.user.lastName,
  roles: result.user.userRoles.map((ur) => ur.role.name),
};
```

`result.user.firstName` and `result.user.lastName` are available without
query changes. The `IdentityService.validateCredentials()` query at line 43
uses `findFirst` with `include` and no custom `select`, which returns all
columns of `identity.users` including these fields.

Backward compatibility: Existing tokens issued before M28 deployment do not
contain `firstName` or `lastName` fields. JWT tokens have a 1-hour maxAge
(`JWT_ACCESS_EXPIRES_IN_SECONDS` in identity.constants.ts), so all pre-M28
tokens will expire within 60 minutes of deployment. The frontend handles
missing name fields gracefully via the fallback display logic in Decision 8.
The backend handles missing fields via `?? ''` defaults in JwtStrategy.validate()
(see Decision 3). No server-side validation or token blocklist is required.

## Decision 3 — RequestUser Extension and Backward-Safe validate()

`RequestUser` in `apps/api/src/identity/jwt.strategy.ts` shall be extended:

```typescript
// Before (current):
export interface RequestUser {
  userId: string;
  tenantId: string;
  email: string;
  roles: string[];
}

// After (M28):
export interface RequestUser {
  userId: string;
  tenantId: string;
  email: string;
  firstName: string;
  lastName: string;
  roles: string[];
}
```

`JwtStrategy.validate()` shall use empty-string defaults for name fields to
handle any pre-M28 token in the 1-hour transition window:

```typescript
// After (M28):
validate(payload: JwtPayload): RequestUser {
  return {
    userId: payload.sub,
    tenantId: payload.tenantId,
    email: payload.email,
    firstName: payload.firstName ?? '',
    lastName: payload.lastName ?? '',
    roles: payload.roles,
  };
}
```

This typing decision (non-optional `string` with empty-string default rather
than `string | undefined`) means all downstream consumers of `RequestUser`
can treat `firstName` and `lastName` as always-strings without null checks.
The empty string is the appropriate representation of "name not available" —
it is falsy and can be used in display fallback logic (`firstName || email`).

No other uses of `RequestUser` in the existing codebase (`GET /auth/me`,
`POST /auth/logout`) expose name fields in their response — those endpoints
remain unchanged. `RequestUser.firstName` and `RequestUser.lastName` are
available on the request context if future endpoints need them, but no
existing endpoint is modified.

## Decision 4 — Session Utility: getSessionUser()

`apps/web/src/lib/session.ts` shall export a new `SessionUser` interface and
`getSessionUser()` function alongside the existing `getSessionRoles()`:

```typescript
export interface SessionUser {
  userId: string;
  email: string;
  firstName: string;
  lastName: string;
  roles: string[];
}

export function getSessionUser(token: string): SessionUser {
  try {
    const raw = JSON.parse(
      atob(token.split('.')[1].replace(/-/g, '+').replace(/_/g, '/')),
    ) as {
      sub?: unknown;
      email?: unknown;
      firstName?: unknown;
      lastName?: unknown;
      roles?: unknown;
    };
    return {
      userId:    typeof raw.sub       === 'string' ? raw.sub       : '',
      email:     typeof raw.email     === 'string' ? raw.email     : '',
      firstName: typeof raw.firstName === 'string' ? raw.firstName : '',
      lastName:  typeof raw.lastName  === 'string' ? raw.lastName  : '',
      roles:     Array.isArray(raw.roles)           ? (raw.roles as string[]) : [],
    };
  } catch {
    return { userId: '', email: '', firstName: '', lastName: '', roles: [] };
  }
}
```

All fields default to empty string (or empty array for roles) when absent.
The function never throws. Empty string is falsy and allows display fallback
logic (`user.firstName || user.email`) without null-check ceremony.

`getSessionUser()` is the preferred utility for all new code that needs more
than role names. The existing `getSessionRoles()` is preserved unchanged —
its ~25 call sites are not modified in M28.

## Decision 5 — getSessionRoles() Backward Compatibility

`getSessionRoles()` in `apps/web/src/lib/session.ts` is preserved exactly as
it exists at M27. Its implementation, signature, and return type are not
changed in M28. All existing import sites continue to work without modification.

Rationale: ~25 files across the codebase import and call `getSessionRoles()`.
Modifying or removing it would require touching every one of those files,
expanding M28 scope dramatically. M28's only session-utility deliverable is
the additive `getSessionUser()` function.

Recommendation for future milestones: Files that currently define their own
inline copy of the JWT base64-decode logic (observed in several files under
apps/web/src/app/(dashboard)/recruiting/) should be migrated to import
`getSessionRoles()` from `@/lib/session`. This cleanup is not in M28 scope
but is the recommended first step when any of those files are next touched.

## Decision 6 — Role Abbreviation Authority

The following canonical abbreviation map is authoritative for all platform
features that display role names in compact form. This map is defined in the
`UserIdentityChip` component and must not be duplicated or diverged from in
other components.

```typescript
const ROLE_ABBREVIATIONS: Record<string, string> = {
  'System Administrator': 'SA',
  'HR Director':          'HRD',
  'Workforce Planner':    'WP',
  'Recruiter':            'REC',
  'Hiring Manager':       'HM',
  'Compliance Officer':   'CO',
  'Executive User':       'EU',
};
```

Fallback for unknown role names: take the first 3 characters of the role
name, uppercased. This handles any role name that might be added in a future
milestone without requiring a `UserIdentityChip` update.

This abbreviation map shall be exported from the `UserIdentityChip` module so
that future features (e.g., a Phase 5 audit log viewer's role column) can
import it rather than defining their own.

## Decision 7 — UserIdentityChip Component Design

A new Server Component shall be created at:

```text
apps/web/src/components/shared/user-identity-chip.tsx
```

Component contract:

```typescript
// Props: none. Reads JWT from session cookie internally.
// Returns null when: no session cookie, roles array is empty.
// Is a Server Component — no 'use client' directive.
// Reference: GD-M28-1 Decisions 4, 6, 8, 9
export function UserIdentityChip(): React.ReactElement | null
```

The component reads the session cookie via `cookies()` from `next/headers`,
decodes the JWT via `getSessionUser()`, and returns a compact inline span.
It must not accept the token or user as props — doing so would require every
consumer to perform their own decode; the chip should self-contain that logic.

The chip is non-interactive:
- No `onClick` handler
- No hover state beyond the default cursor
- No dropdown, tooltip, or popover of any kind
- Cursor: `default` (not `pointer`)

Visual treatment shall match the existing header right-side element style:
- Font: inherit from shell (IBM Plex Sans via fontFamily on outer div)
- Text color: `rgba(255, 255, 255, 0.75)` for name and role text
- Role badge: small rounded pill with `background: rgba(147, 197, 253, 0.15)`,
  `color: #93c5fd` (the existing header nav active treatment), font-size 10px,
  font-weight 700, letter-spacing 0.04em, border-radius 4px, padding 1px 5px
- No border
- Display: `inline-flex`, `align-items: center`, `gap: 6px`

The component uses only inline styles (no Tailwind class names). This is
consistent with the existing shell headers, which use inline styles for
elements in the nav and header-right sections to avoid utility class specificity
conflicts.

## Decision 8 — Display Format and Canonical Examples

The chip displays in this order: `{displayName} · {primaryRole} [{badge}]`

Where:
- `displayName` = `user.firstName` when non-empty; `user.email` otherwise
- `primaryRole` = `user.roles[0]` (the first element in the roles array)
- `badge` = role abbreviation (single role) or `+N` (multiple roles)
- The `·` separator is the middle dot character (U+00B7), consistent with the
  existing domain nav separator treatment in the header

Canonical display examples:

```text
Single role, name available:
  Adaora · System Administrator    [SA]
  Adaora · HR Director             [HRD]
  Adaora · Workforce Planner       [WP]
  Marcus · Recruiter               [REC]
  Elena · Hiring Manager           [HM]
  David · Compliance Officer       [CO]
  Sarah · Executive User           [EU]

Multiple roles, name available:
  Adaora · System Administrator    [+1]  (has 2 roles; badge shows extra count)
  Marcus · HR Director             [+2]  (has 3 roles; badge shows extra count)

Name missing (firstName is empty string):
  adaora@dev.gov · System Administrator    [SA]
  marcus@dev.gov · Recruiter               [REC]
```

The `+N` badge counts additional roles beyond the primary. If a user has 3
roles, the badge shows `+2`. The role names for the non-primary roles are not
displayed in the chip — they are visible on the user's own record in the Admin
Workspace.

The chip must not exceed a reasonable horizontal width. If a role name is
unusually long (beyond the 7 defined platform roles), the badge abbreviation
fallback (first 3 chars, uppercased) keeps it compact.

## Decision 10 — Role Switcher Explicit Deferral

A role switcher — any affordance that allows a user to select, filter, or
temporarily adopt a single active role from their set of assigned roles —
is explicitly prohibited in M28 and deferred indefinitely.

Prohibition rationale:

```text
1. Backend RBAC evaluates RequestUser.roles (the full assigned roles array),
   not a selected active role. A frontend role filter would not reduce the
   user's actual permissions — they would still receive all role-gated content
   for all assigned roles. The UI would be misleading.

2. A frontend-only active-role filter is insecure. If the page appears to show
   only HR Director content but the user can still POST to SA-gated endpoints
   (because the JWT still carries both roles), the UI creates a false sense
   of permission scoping that has no backend enforcement.

3. True role switching requires backend infrastructure:
   - A session-override endpoint (e.g., POST /auth/switch-role)
   - A modified JWT that carries only the selected role (or a role-selection
     field read by all guards)
   - Audit events for role-switch initiation and termination
   - A governance decision authorizing the endpoint, its RBAC rules, and its
     audit surface
   None of this infrastructure exists.

4. No functional requirement for role switching exists in the current platform
   user base. All 7 platform roles have distinct, non-overlapping workflows.
   A user assigned both HRD and WP roles sees all HRD content plus WP content,
   which is correct and expected — they are authorized for all of it.
```

Any future milestone that proposes role switching must submit a governance
decision that addresses all four items above before any implementation.
