# Decisions

## Decision 1 — spec/05 Deviation: appointment_authority Column

`workforce.employees` is amended to include:

```sql
appointment_authority VARCHAR(50) NOT NULL
```

This column is not present in `spec/05_database_schema.md` as written.
Its addition is authorized by GD-PRE-M13-001 Decision 2, which requires the field
on all Path A employee records. This decision records the formal spec/05 deviation
following the GD-M13-1 precedent.

The complete `appointment_authority` value domain at the database level:

```text
LATERAL_TRANSFER
EMERGENCY_APPOINTMENT
SCHEDULE_A
SCHEDULE_C
REINSTATEMENT
SENIOR_EXECUTIVE
ADMINISTRATIVE
COMPETITIVE_APPOINTMENT   — stored value for Phase 3 VAC-401 hires; never a valid
                            API input (GD-PRE-M13-001 Decision 3)
```

`COMPETITIVE_APPOINTMENT` is permitted in the database column but rejected at the
service layer for any API-initiated write. Its presence in the domain allows Phase 3
VAC-401 to populate the field system-side without a schema migration.

## Decision 2 — spec/05 Deviation: position_id Column

`workforce.employees` is amended to include:

```sql
position_id UUID NULL
```

With the following FK constraint:

```sql
ALTER TABLE workforce.employees
ADD CONSTRAINT fk_employee_position
FOREIGN KEY (position_id) REFERENCES workforce.positions(id);
```

And the following index:

```sql
CREATE INDEX idx_employees_position
ON workforce.employees(position_id)
WHERE position_id IS NOT NULL;
```

The column is nullable, consistent with GD-PRE-M13-002 Decision 2 (positionId is
nullable at employee creation). The partial index covers only assigned rows; unassigned
employees are excluded from the index.

`position_id` is not present in `spec/05_database_schema.md` employees table as
written. GD-PRE-M13-002 explicitly governs "Future position linkage milestone schema
design" and is the design authority for this column. This decision records the
formal spec/05 deviation.

No database-level UNIQUE constraint is placed on `position_id`. The 1:1 FTE Slot
Model constraint (GD-PRE-M13-002 Decision 1) is enforced at the service layer, not
at the database level, per the explicit instruction in GD-PRE-M13-002 Decision 1:
"Enforcement method: Service layer. No database-level unique constraint is required."

## Decision 3 — Migration Backfill Authorization for appointment_authority

All existing `workforce.employees` records receive:

```sql
appointment_authority = 'ADMINISTRATIVE'
```

as the permanent backfill value when the M15 migration adds the
`appointment_authority NOT NULL` column.

### Authorization Rationale

GD-PRE-M13-001 Decision 4 states that appointmentAuthority is permanent and immutable
on the employee record. This backfill assigns an immutable value to records that
predate the field's existence. Such a retroactive assignment requires explicit
governance authorization rather than implementation discretion.

Authorization is granted on the following basis:

```text
1. All pre-M15 employee records were created via POST /api/v1/employees (Path A),
   the Non-Competitive and Administrative Appointment path (GD-PRE-M13-001 D1).

2. ADMINISTRATIVE is the explicitly designated catch-all value for non-competitive
   appointments not covered by a more specific authority code
   (GD-PRE-M13-001 Decision 2).

3. No creation-time appointmentAuthority data exists for pre-M15 records. In the
   absence of specific authority data, ADMINISTRATIVE is the least-specific and
   therefore the safest permanent assignment.

4. The repository operates in a development environment. No pre-M15 record represents
   a real employee with a legally binding appointment authority record. ADMINISTRATIVE
   is operationally sufficient for all development and testing purposes.
```

### One-Time Exception

This backfill constitutes a one-time authorized deviation from the GD-PRE-M13-001
Decision 4 immutability rule. The exception applies exclusively to records created
before the M15 migration executes. Records created after the migration must supply
`appointmentAuthority` at creation time; no backfill applies to future records.

### Migration Technique

The M15 migration must:

```text
1. Add the column as nullable: ALTER TABLE ... ADD COLUMN appointment_authority VARCHAR(50)
2. Backfill all existing rows: UPDATE workforce.employees SET appointment_authority = 'ADMINISTRATIVE'
3. Add the NOT NULL constraint: ALTER TABLE ... ALTER COLUMN appointment_authority SET NOT NULL
```

This three-step approach prevents migration failure on existing rows and is the
authoritative M15 migration technique for this column.

## Decision 4 — Create Employee Request Body Amendment

`POST /api/v1/employees` request body is amended as follows:

```json
{
  "employeeNumber": "EMP001",
  "firstName": "John",
  "lastName": "Smith",
  "departmentId": "uuid",
  "appointmentAuthority": "ADMINISTRATIVE",
  "positionId": "uuid"
}
```

### appointmentAuthority Field Rules

```text
Required at creation on Path A.

Valid values (Path A):
  LATERAL_TRANSFER
  EMERGENCY_APPOINTMENT
  SCHEDULE_A
  SCHEDULE_C
  REINSTATEMENT
  SENIOR_EXECUTIVE
  ADMINISTRATIVE

Rejected value:
  COMPETITIVE_APPOINTMENT — system-only (GD-PRE-M13-001 D3)
  Any other string — not a recognized authority code

On missing field:       HTTP 422, error code: APPOINTMENT_AUTHORITY_REQUIRED
On rejected value:      HTTP 422, error code: INVALID_APPOINTMENT_AUTHORITY
On COMPETITIVE_APPOINTMENT submitted via API: HTTP 422, error code: COMPETITIVE_APPOINTMENT_SYSTEM_ONLY
```

### positionId Field Rules

```text
Optional at creation (GD-PRE-M13-002 D2).

When positionId is provided at creation:
  — Target position must be in ACTIVE status.
    Violation: HTTP 422, error code: POSITION_NOT_ACTIVE_FOR_ASSIGNMENT
  — Target position must have no other non-SEPARATED incumbent.
    Violation: HTTP 422, error code: POSITION_ALREADY_OCCUPIED
  — On success: the employee record is created with positionId set.
    Audit event emitted: WORKFORCE_EMPLOYEE_POSITION_ASSIGNED (Decision 9).

When positionId is omitted:
  — Employee record is created with position_id = NULL.
  — positionId may be assigned via POST /employees/{id}/assign-position (Decision 5).
  — The employee cannot transition to ACTIVE status until positionId is set
    (GD-PRE-M13-002 D2; error code: POSITION_REQUIRED_FOR_ACTIVATION).
```

### GET Response Extension

After M15, `GET /api/v1/employees` and `GET /api/v1/employees/{id}` responses include
`appointmentAuthority` and `positionId` fields. This is an additive, non-breaking
extension. spec/06 does not define response body shapes for GET employee endpoints;
no spec deviation is required for this addition.

## Decision 5 — Position Assignment Endpoint

M15 introduces a dedicated position assignment action endpoint:

```text
POST /api/v1/employees/{id}/assign-position
```

### Rationale for Dedicated Endpoint

The repository uses dedicated `POST` action endpoints for all named operations that
change lifecycle state and produce distinct audit events:

```text
POST /api/v1/positions/{id}/close     — position lifecycle transition
POST /api/v1/positions/{id}/activate  — position lifecycle transition
POST /api/v1/vacancies/{id}/close     — vacancy lifecycle transition
```

Position assignment is a named occupancy state change that:
- Triggers three service-layer guards (slot model, status, occupancy)
- Produces distinct audit events (POSITION_ASSIGNED vs. POSITION_REASSIGNED vs. POSITION_CLEARED)
- Operates on a distinct error domain (POSITION_NOT_ACTIVE_FOR_ASSIGNMENT, POSITION_ALREADY_OCCUPIED, POSITION_REQUIRED_FOR_ACTIVATION)

A dedicated endpoint unambiguously identifies the operation in audit logs, separates
position occupancy changes from general employee metadata updates, and is consistent
with the existing lifecycle action pattern.

`PATCH /api/v1/employees/{id}` with an optional `positionId` body field is rejected
as the assignment mechanism. Mixing occupancy state changes into the general update
endpoint conflates two distinct operation classes and prevents clean audit event
differentiation.

### Request Contract

```text
POST /api/v1/employees/{id}/assign-position

Body:
{
  "positionId": "uuid | null"
}

positionId is required in the request body.
positionId: uuid — assigns or reassigns the employee to the specified position.
positionId: null — clears the position assignment (subject to Decision 6 restrictions).
```

### Response

```text
HTTP 200 — success (both initial assignment and reassignment)
Body: updated employee record
```

### Validation Rules (assignment and reassignment — positionId is a UUID)

```text
1. Employee must not be SEPARATED.
   Violation: HTTP 422, error code: EMPLOYEE_SEPARATED

2. Target position must be in ACTIVE status.
   Violation: HTTP 422, error code: POSITION_NOT_ACTIVE_FOR_ASSIGNMENT

3. Target position must have no other incumbent in PENDING_ONBOARDING, ACTIVE,
   ON_LEAVE, or SUSPENDED status (the 1:1 FTE Slot Model, GD-PRE-M13-002 D1).
   The requesting employee is excluded from this check (allowing reassignment
   to the same position, which is a no-op; no change is written).
   Violation: HTTP 422, error code: POSITION_ALREADY_OCCUPIED

4. Tenant isolation: position must belong to the same tenant as the employee.
   Cross-tenant position ID: HTTP 404 (SEC-003).
```

## Decision 6 — Position Clearance and Reassignment Behavior

This decision defines the complete behavioral contract for the `positionId: null`
case and the `positionId: different-uuid` case on `POST /employees/{id}/assign-position`.

### Reassignment (positionId A → positionId B)

Reassignment is supported for all non-SEPARATED employees via the same endpoint
as initial assignment. The target position B must be ACTIVE and unoccupied by any
other non-SEPARATED employee.

The prior position A is automatically vacated: the 1:1 FTE Slot Model occupancy
query no longer counts the moved employee against position A. No explicit API call
is required to vacate position A.

Audit event: `WORKFORCE_EMPLOYEE_POSITION_REASSIGNED` (Decision 9).

### Clearance (positionId: null)

Clearance sets `position_id = NULL` on the employee record, removing the employee
from their current position without a replacement assignment.

#### Clearance is permitted only for PENDING_ONBOARDING employees.

Rationale:

```text
1. GD-PRE-M13-002 Decision 2: positionId is nullable at creation and must be set
   at or before the PENDING_ONBOARDING → ACTIVE transition. A PENDING_ONBOARDING
   employee returning to null positionId re-enters the valid nullable-at-creation
   state. No operational invariant is violated.

2. The legitimate HR scenario for clearance on PENDING_ONBOARDING: a position was
   incorrectly assigned before the employee began work and must be corrected before
   the correct position can be assigned. Reassignment (positionId A → positionId B)
   is the preferred resolution when the correct position is known. Clearance addresses
   the case where no correct position is yet available.

3. A PENDING_ONBOARDING employee has not yet transitioned to ACTIVE status. The
   position assignment is not yet operationally load-bearing.
```

#### Clearance is NOT permitted for ACTIVE, ON_LEAVE, or SUSPENDED employees.

```text
Violation: HTTP 422, error code: POSITION_CLEARANCE_NOT_PERMITTED_FOR_STATUS
Error message: "positionId cannot be cleared while employee status is [status].
               Reassign the employee to a different position or separate the
               employee through the standard HR process."
```

Rationale:

```text
1. An ACTIVE, ON_LEAVE, or SUSPENDED employee holds a current operational assignment.
   Clearing positionId would create an employee in an active employment status with
   no position — a state that violates the operational intent of the 1:1 FTE Slot
   Model and produces an indeterminate occupancy count.

2. The correct HR operations for "removing an employee from a position" are:
   (a) Reassign the employee to a different active position, or
   (b) Separate the employee through the standard SEPARATED transition.
   Clearance without one of these operations is not a defined HR workflow.

3. Position closure does not cascade to employee status (GD-PRE-M13-002 D4).
   The corresponding principle applies in the inverse direction: employee occupancy
   clearance without a position-level event is not a supported operation.
```

#### Clearance is NOT permitted for SEPARATED employees.

```text
SEPARATED employees retain their historical positionId reference
(GD-PRE-M13-002 D4). No modifications to a SEPARATED employee's positionId
are permitted. Violation: HTTP 422, error code: EMPLOYEE_SEPARATED.
```

### Future Workforce Movement Milestone

Workforce movement workflows (lateral transfers between positions without SEPARATED
transition, temporary position detachments, acting assignments) are not defined in
M15. If future HR operations require positionId clearance for ACTIVE employees, a
new governance decision must be issued. M15 implements the minimum viable occupancy
mechanism as designed in GD-PRE-M13-002.

## Decision 7 — Position Detail Occupant Response Contract

`GET /api/v1/positions/{id}` is amended to include an `occupant` field in the
response body.

### Response Shape

```json
{
  "id": "uuid",
  "title": "string",
  "status": "DRAFT | ACTIVE | FROZEN | CLOSED",
  "departmentId": "uuid",
  "classification": "string | null",
  "salaryBand": "string | null",
  "createdAt": "ISO 8601 timestamp",
  "updatedAt": "ISO 8601 timestamp",
  "occupant": {
    "id": "uuid",
    "firstName": "string",
    "lastName": "string",
    "employeeNumber": "string",
    "employmentStatus": "PENDING_ONBOARDING | ACTIVE | ON_LEAVE | SUSPENDED",
    "hireDate": "YYYY-MM-DD | null"
  }
}
```

When the position is vacant (no employee in PENDING_ONBOARDING, ACTIVE, ON_LEAVE,
or SUSPENDED status holds a positionId reference to this position):

```json
{
  "occupant": null
}
```

### Occupant Field Rules

```text
id              — employee UUID; always present when occupant is non-null
firstName       — display name; always present
lastName        — display name; always present
employeeNumber  — employee identifier; always present
employmentStatus — canonical employment status; always present
hireDate        — serialized as YYYY-MM-DD (date-only, consistent with
                  GD-M13-2 D16 date serialization pattern); null if not recorded
```

### Excluded Fields

```text
email              — PII; excluded per RBAC-952 pattern; available via full employee record
appointmentAuthority — compliance-sensitive; available via GET /api/v1/employees/{id}
tenantId           — never in responses (SEC-003)
positionId         — redundant; this IS the position being queried
departmentId       — redundant; known from the position's own departmentId field
terminationDate    — not relevant to occupancy display
createdAt/updatedAt/deletedAt — employee lifecycle timestamps; not required for
                                occupancy display context
```

### Implementation Notes

The occupant data is derived by querying `workforce.employees` for the first row
matching:

```text
position_id = this position's id
AND employment_status IN ('PENDING_ONBOARDING', 'ACTIVE', 'ON_LEAVE', 'SUSPENDED')
AND tenant_id = requesting user's tenant_id
AND deleted_at IS NULL
LIMIT 1
```

The 1:1 FTE Slot Model (GD-PRE-M13-002 D1) guarantees at most one row is returned
for this query. The `LIMIT 1` is a defensive guard, not an operational assumption.

The list endpoint `GET /api/v1/positions` does NOT include the occupant object.
List items link to the detail page per GD-PHASE2-CLOSURE-002 D3. Occupant data
is loaded at detail page level only.

### Rationale

Embedding occupant in `GET /api/v1/positions/{id}` follows the established
sub-resource embedding pattern in this repository:

```text
GET /api/v1/employees/{id}        — embeds departmentName from JOIN
GET /api/v1/employees/{id}/skills — embeds skillName, skillCategory from JOIN
                                    (GD-M13-2 D14)
```

A separate `GET /api/v1/positions/{id}/occupant` endpoint is rejected: it would
require two API calls on every position detail page load (one for position fields,
one for occupant data), with no caching benefit. GD-PHASE2-CLOSURE-002 D3 specifies
the occupant display as part of the position detail page context — not as an
independently queryable resource.

## Decision 8 — appointmentAuthority Immutability Enforcement

Attempts to modify `appointmentAuthority` after employee creation are rejected.

### Enforcement Rule

A `PATCH /api/v1/employees/{id}` request body that includes an `appointmentAuthority`
field — regardless of whether the submitted value matches the stored value — is
rejected:

```text
HTTP 422
Error code: APPOINTMENT_AUTHORITY_IMMUTABLE
Message: "appointmentAuthority is immutable and cannot be changed after creation."
```

### Pattern Reference

This follows the GD-M12-6 precedent for `employeeNumber` immutability:

```text
GD-M12-6: employeeNumber immutable after creation; PUT with employeeNumber field
           returns HTTP 422 + EMPLOYEE_NUMBER_IMMUTABLE
```

`APPOINTMENT_AUTHORITY_IMMUTABLE` is the M15 equivalent error code for the
appointmentAuthority field, following the same pattern.

### Rejection Scope

The immutability check applies to:

```text
PATCH /api/v1/employees/{id}  — explicit rejection when appointmentAuthority present
POST  /api/v1/employees/{id}/assign-position — appointmentAuthority is not in the
                                               request body for this endpoint;
                                               no check required
```

The `POST /api/v1/employees` (Create Employee) endpoint is the only write surface
where `appointmentAuthority` is a valid field. Any other write path must reject it.

## Decision 9 — Audit Event Types for M15 Position Linkage Operations

The following `AuditEventType` values must be added to the `AuditEventType` enum
before M15 schema migration and service implementation begins. This follows the
GD-M13-4 Decision 4 precedent.

```text
Position assignment operations on employees:
  WORKFORCE_EMPLOYEE_POSITION_ASSIGNED    — positionId set from null to a UUID
                                            (initial assignment, including at creation)
  WORKFORCE_EMPLOYEE_POSITION_REASSIGNED  — positionId changed from one UUID to another
  WORKFORCE_EMPLOYEE_POSITION_CLEARED     — positionId set from UUID to null
                                            (PENDING_ONBOARDING only; Decision 6)
```

### Event Selection Rules

```text
Prior positionId is null,  new positionId is UUID     → POSITION_ASSIGNED
Prior positionId is UUID,  new positionId is UUID (≠) → POSITION_REASSIGNED
Prior positionId is UUID,  new positionId is null      → POSITION_CLEARED
Prior positionId == new positionId (no change)         → no audit event emitted
```

### Event Metadata Requirements

WORKFORCE_EMPLOYEE_POSITION_ASSIGNED:
```text
employee_id
new_position_id
new_position_title
appointment_authority       — record the employee's authority at assignment time
```

WORKFORCE_EMPLOYEE_POSITION_REASSIGNED:
```text
employee_id
prior_position_id
prior_position_title
new_position_id
new_position_title
```

WORKFORCE_EMPLOYEE_POSITION_CLEARED:
```text
employee_id
prior_position_id
prior_position_title
employment_status_at_clearance   — for audit context; clearance is only valid on
                                   PENDING_ONBOARDING per Decision 6
```

## Decision 10 — RBAC for Position Assignment Endpoint

```text
POST /api/v1/employees/{id}/assign-position:
  Authorized:    System Administrator, HR Director
  Unauthorized:  Workforce Planner, Compliance Officer, Recruiter, Executive User
                 → HTTP 403
```

### Rationale

Position assignment is a write operation on the employee record that modifies occupancy
state. It carries the same authorization as all other employee write operations:
System Administrator and HR Director (EMP-AUTH-001 and EMP-AUTH-002, GD-M12-3).

Executive User is excluded from all employee write operations per RBAC-952 (prohibition
on Executive User access to individual employee records). This prohibition extends to
position assignment.

Workforce Planner and Compliance Officer hold read-only access to employee records
under EMP-AUTH-003 and EMP-AUTH-004. Position assignment write access is not granted
to these roles.
