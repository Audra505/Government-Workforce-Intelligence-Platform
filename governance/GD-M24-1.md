# Background

## Backend Domain Status (M13 + M14 — Complete)

The Skills and Certifications backend domain has been live since M13/M14. Nine
endpoint groups are implemented and CI-confirmed:

```text
Skill catalog:
  POST   /api/v1/skills                              — create (SA + HRD)
  GET    /api/v1/skills                              — list, paginated (SA+HRD+WP+CO)
  GET    /api/v1/skills/:id                          — detail (SA+HRD+WP+CO)
  PATCH  /api/v1/skills/:id                          — update (SA + HRD)

Certification catalog:
  POST   /api/v1/certifications                      — create (SA + HRD)
  GET    /api/v1/certifications                      — list, paginated (SA+HRD+WP+CO)
  GET    /api/v1/certifications/:id                  — detail (SA+HRD+WP+CO)
  PATCH  /api/v1/certifications/:id                  — update (SA + HRD)

Employee skill assignments (upsert):
  POST   /api/v1/employees/:id/skills                — assign or update (SA + HRD)
  GET    /api/v1/employees/:id/skills                — list, non-paginated (SA+HRD+WP+CO)

Employee certification assignments (upsert):
  POST   /api/v1/employees/:id/certifications        — assign or update (SA + HRD)
  GET    /api/v1/employees/:id/certifications        — list, non-paginated (SA+HRD+WP+CO)

Cross-employee expiring certification tracking:
  GET    /api/v1/employee-certifications/expiring    — paginated (SA+HRD+WP+CO)
```

No DELETE endpoints exist for employee skill or certification assignments.
There is no aggregate skill coverage endpoint. There is no position-skill
endpoint.

# Decisions

## Decision 1 — Scope Boundary: Tenant-Catalog and Employee-Level Only

M24 operates at the tenant skill/certification catalog level and at the
individual employee assignment level exclusively.

```text
In-scope domain surfaces:
  Skill catalog (tenant-scoped): create, list, edit
  Certification catalog (tenant-scoped): create, list, edit
  Employee skill assignments: view, assign, update proficiency/verifiedAt
  Employee certification assignments: view, assign, update status/dates
  Expiring certifications: cross-employee read view (no write from this page)

Explicitly excluded from all M24 surfaces:
  position_skills / position_certifications (GD-M13-5 — schema absent)
  Position skill requirements or coverage UI (no backend; no schema)
  FR-152 Competency Framework (GD-PRE-PHASE3-004 — Phase 4)
  Candidate-level skills (no schema; not a Phase 3 requirement)
  Bulk import of any skill or certification data
  AI skill-matching or competency inference
  Skills coverage analytics or aggregate gap reporting
  Fake or placeholder sections for any deferred capability
```

Rationale: The backend domain is complete. The frontend gap is complete
absence. M24 closes that gap for the surfaces the backend already supports
without introducing any design or data model assumptions for capabilities
that have no backend and no governed spec path.

## Decision 2 — No Removal Endpoints

Neither employee skill assignments nor employee certification assignments
may be removed through any M24 frontend surface.

```text
Backend state:
  POST /api/v1/employees/:id/skills          — upsert (assign or update)
  POST /api/v1/employees/:id/certifications  — upsert (assign or update)
  DELETE endpoints for either: do not exist.
  No service method for skill or certification removal exists.

Required frontend behavior:
  Assignment forms must not expose a "remove" or "delete" action.
  Skill and certification rows in the employee detail page must not
  include a delete button, trash icon, or any affordance implying removal.
  Skill catalog and certification catalog list rows may not expose
  a delete/archive action (no endpoint; no soft-delete exposed to UI).

Future capability:
  Any "remove skill from employee" or "archive catalog entry" feature
  requires a new governed backend endpoint and a new governance decision
  before frontend work may proceed.
```

Rationale: Exposing a removal affordance for which no backend exists creates
a broken UX contract. UI actions must only be offered when the backend
endpoint exists to fulfill them. The upsert model is the complete backend
contract for M24 assignment management.

## Decision 3 — RBAC Policy

Access to skills and certifications surfaces is governed by the following
role matrix, derived from GD-M13-2 and GD-PRE-PHASE3-003:

```text
Role                   | Catalog read | Catalog write | Employee assign | Employee read | Expiring view
-----------------------|--------------|---------------|-----------------|---------------|---------------
System Administrator   | Yes          | Yes           | Yes             | Yes           | Yes
HR Director            | Yes          | Yes           | Yes             | Yes           | Yes
Workforce Planner      | Yes          | No            | No              | Yes           | Yes
Compliance Officer     | Yes          | No            | No              | Yes           | Yes
Recruiter              | No (403)     | No (403)      | No (403)        | No (403)      | No (403)
Executive User         | No (403)     | No (403)      | No (403)        | No (403)      | No (403)
Hiring Manager         | No (403)     | No (403)      | No (403)        | No (403)      | No (403)

canWrite = System Administrator OR HR Director
canRead  = System Administrator OR HR Director OR Workforce Planner OR Compliance Officer
```

Frontend implementation rules:

```text
Tab visibility:
  Skills and Certifications tabs in WorkforceShell are visible to all
  authenticated users. Role restriction is enforced at the BFF and backend
  layers, not through tab hiding. Hiding tabs is not a security control.

Role-gated actions:
  "New Skill" / "New Certification" buttons: rendered only when canWrite
  "Edit" action on catalog rows: rendered only when canWrite
  "Assign Skill" / "Assign Certification" buttons: rendered only when
    canWrite AND !isSeparated (see Decision 7)

Graceful 403 handling:
  If a Recruiter or Executive User navigates to a Skills or Certifications
  page and the BFF receives a 403 from NestJS, the page must render
  gracefully. Acceptable responses: empty state, "not available for your
  role" message. The page must not crash, throw an unhandled error, or
  display a React error boundary for a role-based 403.

verifiedAt:
  The verifiedAt field may only be meaningfully set by SA or HRD
  (GD-M13-2 D13; SKL-211). Since assignment write access is already
  restricted to SA+HRD by canWrite, verifiedAt will always be processed
  correctly at the backend. No additional role check is needed in the
  frontend for verifiedAt.
```

Rationale: RBAC is enforced by the NestJS guards and BFF JWT validation.
The frontend's role is to provide a coherent, non-crashing experience for
all authenticated users — not to be the enforcement layer. Graceful
degradation on 403 is required to prevent broken page states for
lower-privileged users who navigate via URL or bookmark.

## Decision 5 — Expiring Certifications Route and Navigation

The expiring certifications management view is a sub-route under the
Certifications catalog, not a top-level Workforce tab.

```text
Route: /workforce/certifications/expiring

ActiveTab value: 'certifications'
  The Certifications tab remains visually active on this sub-route.
  This is consistent with the recruiting pattern where /recruiting/
  candidates/[id] keeps the Candidates tab active.

Page behavior:
  Fetches from GET /api/v1/employee-certifications/expiring directly
  via serverFetch (same as dashboard — no new BFF route required).
  Paginated using the shared Pagination component.
  withinDays filter controlled by URL search parameter (default: 30).
  Table rows link to /workforce/employees/[employeeId] (employee detail).
  Page is accessible to: SA, HRD, Workforce Planner, Compliance Officer.
  Recruiter and Executive User receive 403 from serverFetch and must be
  handled gracefully (empty state or role message, not an error crash).

Column set:
  Employee, Employee #, Certification, Issuer, Status, Issue Date,
  Expiration Date, Days Until Expiry (or overdue)

Sort order:
  Ascending by expirationDate (soonest first) — matches backend default.
```

Rationale: A standalone tab for "Expiring Certifications" would fragment
the certifications domain into two disconnected tabs. As a sub-route of
the Certifications catalog, it is discoverable from the Certifications
tab and from the dashboard cert widget link (Decision 11). The operator's
mental model stays coherent: certifications are one domain, and the
expiry view is a filter within it.

## Decision 6 — Certification Status Transition Rules in Assignment Forms

The employee certification assignment form must enforce backend status
transition rules to prevent predictable 422 errors.

```text
Initial assignment (INSERT path — no prior assignment for this certificationId):
  Status field: must be ACTIVE or absent (defaults to ACTIVE).
  EXPIRED and REVOKED must not appear as selectable options.
  The form may omit the status field entirely on initial assignment;
  the backend will default to ACTIVE.

Update path (record already exists for this certificationId):
  Allowed transitions (from GD-M13-3 and employee-certification.service.ts):
    ACTIVE  → ACTIVE   (renewal/refresh with updated dates)
    ACTIVE  → EXPIRED  (manual expiration)
    ACTIVE  → REVOKED  (revocation — terminal)
    EXPIRED → ACTIVE   (renewal — triggers CERT_RENEWED audit event)
    EXPIRED → EXPIRED  (date update only)
    EXPIRED → REVOKED  (revocation from expired state — terminal)
    REVOKED → [none]   (REVOKED is terminal; no outbound transitions)

REVOKED rendering:
  A certification assignment in REVOKED status must be rendered as
  read-only. No edit button, no form trigger, no status selector.
  Display the REVOKED StatusPill (red). The form must not be invocable
  for REVOKED assignments.

expirationRequired enforcement:
  When the selected certification has expirationRequired = true, the
  expirationDate field is required in the form (Zod schema conditional).
  The form must not allow submission without expirationDate when
  expirationRequired is true.

Date validation:
  expirationDate must not precede issueDate when both are provided.
  This matches the INVALID_DATE_RANGE backend guard (CRT-203).
```

Rationale: Frontend validation that mirrors the backend rules reduces
user-facing error cycles. Errors that the form can prevent locally —
submitting EXPIRED on initial assignment, submitting without
expirationDate when required, triggering a REVOKED assignment — should
be caught before the network request. REVOKED being visually read-only
prevents user confusion over why updates are rejected.

## Decision 7 — SEPARATED Employee Guard for Skill and Certification Assignment

SEPARATED employees cannot receive new skill or certification assignments
(EMP-302 enforced at the NestJS service layer). The frontend must gate
assignment affordances using the already-computed `isSeparated` flag.

```text
isSeparated flag:
  Already computed in apps/web/src/app/(dashboard)/workforce/employees/
  [id]/page.tsx as a derived boolean from employee.employmentStatus.

Required behavior:
  If isSeparated is true:
    "Assign Skill" button must not render.
    "Assign / Update Certification" button must not render.
    Skill and certification sections remain visible (read-only display
    of existing assignments is still informative for auditors and
    HR Directors reviewing a separated employee's record).

  If isSeparated is false:
    Button visibility is gated by canWrite as per Decision 3.

Behavior of existing assignments:
  Existing skill and certification assignments on a SEPARATED employee
  remain visible and readable. The isSeparated guard applies to write
  affordances only, not to display.
```

Rationale: EMP-302 is a hard backend constraint. Showing an assignment
button that the backend will reject with EMPLOYEE_SEPARATED is a broken
UX contract. The isSeparated flag is already present in the page and
requires no additional fetch.

## Decision 8 — BFF Route Contract Policy

Six new BFF route files are introduced in M24. All must follow the
established BFF contract pattern.

```text
New BFF route files:
  apps/web/src/app/api/skills/route.ts
  apps/web/src/app/api/skills/[id]/route.ts
  apps/web/src/app/api/certifications/route.ts
  apps/web/src/app/api/certifications/[id]/route.ts
  apps/web/src/app/api/employees/[id]/skills/route.ts
  apps/web/src/app/api/employees/[id]/certifications/route.ts

Required for all new BFF handlers:
  1. Read JWT from the httpOnly session cookie (same as all existing handlers).
  2. Forward JWT as Authorization: Bearer <token> to NestJS.
  3. Do not forward tenantId from any request body (SEC-003). If tenantId
     is present in an incoming request body, return HTTP 400 immediately
     and do not forward the request to NestJS.
  4. Forward NestJS HTTP status codes and response bodies unchanged.
     Do not rewrite 4xx/5xx errors. Do not suppress validation errors.
  5. Do not add, remove, or rewrite response body fields.
  6. serverFetch for read-only page data (Server Components) continues
     to call NestJS directly — no BFF route needed for serverFetch calls.

Method coverage per file:
  skills/route.ts:                GET (list with query params), POST (create)
  skills/[id]/route.ts:           PATCH (update)
  certifications/route.ts:        GET (list with query params), POST (create)
  certifications/[id]/route.ts:   PATCH (update)
  employees/[id]/skills/route.ts:         GET (list), POST (assign/upsert)
  employees/[id]/certifications/route.ts: GET (list), POST (assign/upsert)

tenantId source:
  tenantId is always derived from the JWT at the NestJS service layer.
  It is never accepted from the client in a request body or query param.
```

Rationale: Uniform BFF contract enforcement is the mechanism by which
SEC-003 tenant isolation is preserved at the frontend boundary. New BFF
routes must apply the same contract as all existing routes — there are
no exceptions for the skills and certifications domain.

## Decision 9 — Data Fidelity Policy Applies to All M24 Pages

GD-M23-1 Decision 2 (Dashboard Data Fidelity Policy) extends to all
Skills and Certifications frontend pages introduced in M24.

```text
Rule 1 — No fake data:
  Every value displayed on any M24 page must come from a live API
  response at request time. Static or hardcoded skill or certification
  counts, names, or statuses are prohibited.

Rule 2 — No placeholder panels:
  Sections for deferred capabilities (skills coverage analytics, position
  skill requirements, competency framework) must be omitted entirely.
  They must not appear as:
    - "Coming soon" banners
    - Mockup panels with illustrative data
    - Greyed-out or locked UI sections
  Omission is the correct behavior until backend support and a governing
  decision exist.

Rule 3 — Graceful degradation:
  Non-fatal fetch failures (skills list fetch fails, employee skills
  fetch returns error) must not crash the page. Skill and certification
  section cards on the employee detail page must use .catch(() => null)
  on their serverFetch calls. A null result renders an empty state,
  not an error boundary.

Rule 4 — Role-gated content omission:
  If a fetch fails with 403 (Recruiter or Executive User navigating to
  a Skills or Certifications page), the content area must be omitted or
  shown with a role-appropriate message. The page must not error.
```

Rationale: Government HR platforms must not mislead operators with
data that does not reflect real system state. This applies equally to
catalog pages, employee detail sub-sections, and the expiring
certifications view. The same principle that governs the dashboard
governs all M24 surfaces.

## Decision 10 — verifiedAt Field Handling

The `verifiedAt` field on employee skill assignments conveys HR-verified
proficiency (versus self-declared) as governed by GD-M13-2 D13 (SKL-211).

```text
Backend behavior:
  verifiedAt is silently dropped by the NestJS controller for any role
  that is not SA or HR Director, even if provided in the request body.
  Since M24 write access is already restricted to SA+HRD by canWrite,
  verifiedAt will always be processed correctly for the roles that can
  reach the assignment form.

Frontend behavior:
  The assign skill form may include an optional verifiedAt date input.
  If included, it must be labeled clearly (e.g., "Verified Date" or
  "HR-Verified As Of").
  If not included in M24A, it may be added in M24B without a new
  governance decision — it is within the governed scope of the
  assign-skill form.
  The form must never make verifiedAt required; the field is optional
  per AssignSkillDto.

Display:
  When verifiedAt is non-null on a skill assignment row, a visual
  indicator (label, date, or checkmark) distinguishes it from
  self-declared (verifiedAt null = proficiency not HR-verified).
  The presence or absence of verifiedAt must not be hidden.
```

Rationale: verifiedAt is a data quality signal used by HR Directors
and Compliance Officers to distinguish organization-verified competencies
from employee self-assessments. The frontend must surface this distinction
to make the data meaningful without requiring an extra governance decision
for each incremental form improvement.
