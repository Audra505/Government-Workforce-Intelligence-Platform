## Planning Inspection Findings

The M17 planning inspection (completed 2026-06-29) found:

```text
1. GD-PRE-PHASE3-002 D4 defines the canonical recruiting.applications table
   with all columns, indexes, partial unique constraint, and FK constraints.
   This is authoritative — implementation must not deviate.

5. RECRUITING_APPLICATION_SUBMITTED already exists in AuditEventType (line 97
   of audit-event-type.enum.ts). Four additional AUD-500 values must be added.

6. The existing CandidateService stub (GD-M16-1 D7) explicitly documents the
   M17 replacement query. M17 must implement it.

7. ApplicationService should use PrismaService directly for VAC-301 vacancy
   status updates rather than importing VacancyService, avoiding a cross-module
   dependency that does not currently exist.
```

## Decision 4 — POS-301 and VAC-601 Disposition

POS-301 and VAC-601 are formally deferred from M17.

**POS-301 — position close cascade to open vacancies:**

POS-301 appears in GD-PRE-PHASE3-001 D6 in the context of M17 scope items.
Inspection finds:

```text
No governing spec exists for POS-301 cascade behavior.
No behavioral rules directive defines the cascade rule.
No schema change is defined for the cascade side effect.
The existing VacancyService closeVacancy method does not implement POS-301.
Implementing POS-301 without governing spec would introduce unverified
behavior into the position close path.
```

POS-301 is deferred. It must not be silently implemented in M17. A future
governance decision must define:

```text
Trigger: which position lifecycle transitions trigger the cascade
Scope: which vacancy statuses are affected (OPEN only, or OPEN + IN_RECRUITMENT)
Behavior: auto-close vs notification vs manual confirmation
Atomicity: whether the cascade must be transactional
Audit events: what audit trail is generated
```

POS-301 deferral does not block M17.

**VAC-601 — vacancy-level applicant count / statistics:**

VAC-601 appears near M17 scope in GD-PRE-PHASE3-001 D6. Inspection finds:

```text
No governing spec defines what VAC-601 covers.
No schema change is defined.
No API contract is defined.
The identifier "VAC-601" appears without behavioral definition.
```

VAC-601 is deferred. It must not be silently implemented in M17. A future
governance decision must define VAC-601 scope before implementation.

VAC-601 deferral does not block M17.

## Decision 5 — Schema Scope

The canonical `recruiting.applications` table definition from GD-PRE-PHASE3-002
Decision 4 is authoritative. M17 must implement this exact column set.

### Required Columns

```sql
id             UUID         PRIMARY KEY
tenant_id      UUID         NOT NULL
candidate_id   UUID         NOT NULL
vacancy_id     UUID         NOT NULL
status         VARCHAR(50)  NOT NULL DEFAULT 'APPLIED'
submitted_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW()
current_stage  VARCHAR(100) NULL
notes          TEXT         NULL
created_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW()
updated_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW()
deleted_at     TIMESTAMPTZ  NULL
```

### Required Partial Unique Index (hand-edited migration)

```sql
CREATE UNIQUE INDEX idx_applications_candidate_vacancy_active
  ON recruiting.applications(tenant_id, candidate_id, vacancy_id)
  WHERE deleted_at IS NULL
    AND status NOT IN ('REJECTED', 'WITHDRAWN');
```

This index is required at the database layer for duplicate prevention.
It must be hand-edited into the migration SQL after `prisma migrate dev`
generates the initial CREATE TABLE statement. Prisma's `@@unique` directive
cannot express a WHERE predicate. Do not add a `@@unique` in schema.prisma
for this constraint — it would block legitimate re-application after terminal
states.

### Required FK Constraints

```sql
ALTER TABLE recruiting.applications
  ADD CONSTRAINT fk_application_candidate
  FOREIGN KEY (candidate_id)
  REFERENCES recruiting.candidates(id);

ALTER TABLE recruiting.applications
  ADD CONSTRAINT fk_application_vacancy
  FOREIGN KEY (vacancy_id)
  REFERENCES workforce.vacancies(id);
```

### Required Supporting Indexes

```sql
CREATE INDEX idx_applications_tenant
  ON recruiting.applications(tenant_id);

CREATE INDEX idx_applications_candidate
  ON recruiting.applications(tenant_id, candidate_id);

CREATE INDEX idx_applications_vacancy
  ON recruiting.applications(tenant_id, vacancy_id);

CREATE INDEX idx_applications_status
  ON recruiting.applications(tenant_id, status)
  WHERE deleted_at IS NULL;
```

### Prisma Model

The Prisma model must follow the Prisma multi-schema pattern established
in M16 for the Candidate model. The Application model uses `@@schema("recruiting")`.

The Vacancy model (workforce schema) has a relation to Position. Application
has a FK to Vacancy (cross-schema). Use a one-directional Prisma relation
from Application → Vacancy and Application → Candidate. Do not add a
back-relation `applications Application[]` on the Vacancy or Candidate
models if WorkforceModule and the CandidateService have no need to load
applications via those models. Confirm with Prisma schema validation during
implementation (Step 2).

### Canonical Application Status Values (GD-PRE-PHASE3-002 D8)

```text
APPLIED      — initial status; set on create; not accepted from caller
SCREENING    — active workflow state
INTERVIEW    — active workflow state
EVALUATION   — active workflow state
OFFER        — active workflow state
HIRED        — terminal; reserved for M19; not reachable in M17
REJECTED     — terminal; implemented in M17
WITHDRAWN    — terminal; implemented in M17
```

HIRED must not be reachable through any M17 endpoint. The advance endpoint
must not accept HIRED as a target status. Advance from OFFER must return
APPLICATION_AWAITING_HIRE, not transition to HIRED.

## Decision 6 — Application Relationship Rules

### Candidate rules:

```text
The submitted candidateId must resolve to a candidate that:
  - exists (not soft-deleted; deleted_at IS NULL)
  - belongs to the same tenant as the authenticated user (SEC-003)
  - has status = 'ACTIVE' (not ARCHIVED)

A soft-deleted or cross-tenant candidate returns APPLICATION_NOT_FOUND or
CANDIDATE_NOT_FOUND, not a tenant-leaking error. Service-layer outcomes:
  - candidate absent or deleted_at IS NOT NULL → CANDIDATE_NOT_FOUND
  - candidate.tenantId ≠ actor.tenantId → CANDIDATE_NOT_FOUND (not 403)
  - candidate.status = 'ARCHIVED' → CANDIDATE_ARCHIVED
```

### Vacancy rules:

```text
The submitted vacancyId must resolve to a vacancy that:
  - exists (not soft-deleted; deleted_at IS NULL)
  - belongs to the same tenant as the authenticated user (SEC-003)
  - has status in ('OPEN', 'IN_RECRUITMENT')

A soft-deleted or cross-tenant vacancy returns VACANCY_NOT_FOUND.
A vacancy in any other status returns VACANCY_NOT_OPEN.

Vacancy statuses that must reject new applications:
  DRAFT, CLOSED, CANCELLED, FILLED, and any future non-open status.
```

### Tenant rules (SEC-003):

```text
Cross-tenant candidate or vacancy references must return not-found behavior,
not authorization error. This prevents tenant enumeration — callers must not
be able to distinguish "this record doesn't exist" from "this record exists
in another tenant."
```

## Decision 7 — Duplicate Application Prevention

### Rule:

A candidate may have at most one active (non-terminal) application for the
same vacancy within the same tenant at any point in time.

### Scope of "active" for this constraint:

```text
Active means: deleted_at IS NULL AND status NOT IN ('REJECTED', 'WITHDRAWN')
```

HIRED is not included in the NOT IN list because the partial unique index
(Decision 5) is designed for the M17 terminal states. If a HIRED application
exists, re-application would need governance review — but HIRED is not
reachable in M17, so this edge case does not arise until M19.

### Re-application allowed:

```text
A candidate may re-apply to the same vacancy after REJECTED.
A candidate may re-apply to the same vacancy after WITHDRAWN.
```

### Enforcement layers:

```text
Database layer: partial unique index (Decision 5 — idx_applications_candidate_vacancy_active)
Service layer:  pre-check query before INSERT
  SELECT id FROM recruiting.applications
  WHERE tenant_id = :tenantId
    AND candidate_id = :candidateId
    AND vacancy_id = :vacancyId
    AND deleted_at IS NULL
    AND status NOT IN ('REJECTED', 'WITHDRAWN')

Pre-check returns APPLICATION_ALREADY_EXISTS.
Catch Prisma P2002 as race-condition fallback → also return APPLICATION_ALREADY_EXISTS.
```

## Decision 8 — VAC-301 Vacancy Interaction

### Rule:

When the first valid application is submitted to a vacancy in OPEN status,
ApplicationService must transition that vacancy's status to IN_RECRUITMENT.

### "First" semantics:

If `vacancy.status = 'OPEN'`, transition to `'IN_RECRUITMENT'` on successful
application insert. If `vacancy.status = 'IN_RECRUITMENT'` already (a prior
application triggered the transition), no status change is required — the
vacancy is already in the correct state.

### Implementation:

```text
Use Prisma updateMany with a conditional WHERE to make the transition idempotent:

  await this.prisma.vacancy.updateMany({
    where: { id: vacancyId, tenantId, status: 'OPEN' },
    data:  { status: 'IN_RECRUITMENT' },
  });

updateMany returns a count. If count > 0, the vacancy was actually transitioned.
Emit WORKFORCE_VACANCY_UPDATED audit event only if count > 0.
If count = 0 (vacancy was already IN_RECRUITMENT or not found), no audit event.
```

### Module dependency:

ApplicationService must use PrismaService directly for the VAC-301 vacancy
update. ApplicationService must not import or inject VacancyService. Reason:
RecruitingModule does not currently import WorkforceModule. Adding that
dependency for a single conditional update is unnecessary coupling. PrismaService
is @Global() and available to ApplicationService without a new module import.

### Write atomicity:

The application INSERT and the VAC-301 vacancy UPDATE are two separate Prisma
calls, consistent with the EMP-700 pattern used throughout this codebase.
They are not wrapped in a database transaction.

Recoverable inconsistency risk: if the application INSERT succeeds but the
vacancy UPDATE fails (e.g., infrastructure transient error), the application
exists but the vacancy remains OPEN. The next successful application to the
same vacancy will trigger VAC-301 again and correct the vacancy status.
This inconsistency window is brief and self-healing. It is accepted as a
known risk consistent with the EMP-700 pattern. The risk is recorded here
for audit trail completeness.

### Audit event for VAC-301:

Use the existing `WORKFORCE_VACANCY_UPDATED` AuditEventType. No new enum
value is required for VAC-301.

```text
metadata: {
  previousStatus: 'OPEN',
  newStatus: 'IN_RECRUITMENT',
  trigger: 'VAC-301',
  vacancyId: <id>
}
```

No candidate PII in this metadata. No application notes or candidate names.

## Decision 9 — Candidate Archive Active-Application Guard

### Rule:

A candidate with one or more active (non-terminal) applications may not be
archived. The `archiveCandidate` method in CandidateService must return
`CANDIDATE_HAS_ACTIVE_APPLICATIONS` when active applications exist.

### Active application definition for this guard:

```text
Active means:
  tenant_id = :tenantId
  AND candidate_id = :candidateId
  AND deleted_at IS NULL
  AND status NOT IN ('REJECTED', 'WITHDRAWN')
```

HIRED intentionally blocks candidate archive and is therefore NOT included in
the notIn exclusion list. A HIRED application represents a completed hire —
the candidate has become an employee. Archiving a candidate who has been hired
is semantically incoherent and must be blocked. Note: HIRED is not reachable
in M17; this clause is forward-compatible with M19. It is consistent with the
stub contract in GD-M16-1 D7.

Only REJECTED and WITHDRAWN are excluded from the blocking set — they
represent closed applications with no ongoing workflow.

### Implementation:

Replace the stub comment in `CandidateService.archiveCandidate()` with:

```typescript
const activeApplicationCount = await this.prisma.application.count({
  where: {
    tenantId,
    candidateId,
    deletedAt: null,
    status: { notIn: ['REJECTED', 'WITHDRAWN'] },
  },
});
if (activeApplicationCount > 0) {
  return { outcome: 'CANDIDATE_HAS_ACTIVE_APPLICATIONS' };
}
```

This query uses the Application Prisma model, which does not exist until M17
Step 2 (schema migration). The stub replacement must occur after the schema
migration is applied (Step 4 of the implementation sequence).

## Decision 10 — Application Status Transition Rules

### Create:

```text
status is always set to 'APPLIED' by ApplicationService.createApplication.
status is never accepted from the request body (CreateApplicationDto must not
  contain a status field).
```

### Advance (POST /api/v1/applications/{id}/advance):

```text
Permitted forward transitions:
  APPLIED    → SCREENING
  SCREENING  → INTERVIEW
  INTERVIEW  → EVALUATION
  EVALUATION → OFFER

Blocked transitions:
  Advance from OFFER   → APPLICATION_AWAITING_HIRE (403 is not appropriate here;
    this is a workflow error — use 422)
  Advance from HIRED   → APPLICATION_IN_TERMINAL_STATE
  Advance from REJECTED → APPLICATION_IN_TERMINAL_STATE
  Advance from WITHDRAWN → APPLICATION_IN_TERMINAL_STATE
  Any non-sequential skip → INVALID_APPLICATION_TRANSITION

HIRED must not be reachable through advance in M17 under any circumstances.
```

### Advance endpoint semantics — Option B (target status):

The advance endpoint accepts a request body with `targetStatus`. This decision
chooses Option B (explicit target status) over Option A (no body, always next
stage) for the following reasons:

```text
1. Auditability: the caller's intent is explicit in the request and captured
   in audit metadata.
2. Idempotency clarity: a caller advancing to SCREENING twice receives
   INVALID_APPLICATION_TRANSITION on the second call rather than silently
   advancing to INTERVIEW.
3. Test clarity: test assertions reference the intended target state, not
   a position-in-sequence assumption.
4. Consistency with the platform's explicit-intent design pattern (compare:
   POST /api/v1/employees/:id/assign-position which always accepts explicit
   positionId rather than inferring it).
```

`targetStatus` must be one of: SCREENING, INTERVIEW, EVALUATION, OFFER.
APPLIED, HIRED, REJECTED, and WITHDRAWN are not valid targetStatus values.
Service must verify that targetStatus is exactly the next sequential state
from the current status. A request for INTERVIEW when status is APPLIED returns
INVALID_APPLICATION_TRANSITION (not a skip-ahead).

### Reject (POST /api/v1/applications/{id}/reject):

```text
Transitions any non-terminal status (APPLIED, SCREENING, INTERVIEW,
EVALUATION, OFFER) to REJECTED.

Reject from a terminal state (HIRED, REJECTED, WITHDRAWN) returns
APPLICATION_IN_TERMINAL_STATE.

No request body required. Empty body or absent body accepted.

On success, returns 200 with the updated ApplicationResponse (status = REJECTED).
tenantId and deletedAt must not appear in the response.
The service must return the updated ApplicationRecord so the controller can
project it through toApplicationShape(). Callers must not need a second GET
to observe the resulting REJECTED status.
```

### Withdraw (POST /api/v1/applications/{id}/withdraw):

```text
Transitions any non-terminal status (APPLIED, SCREENING, INTERVIEW,
EVALUATION, OFFER) to WITHDRAWN.

Withdraw from a terminal state returns APPLICATION_IN_TERMINAL_STATE.

No request body required.

On success, returns 200 with the updated ApplicationResponse (status = WITHDRAWN).
tenantId and deletedAt must not appear in the response.
The service must return the updated ApplicationRecord so the controller can
project it through toApplicationShape(). Callers must not need a second GET
to observe the resulting WITHDRAWN status.

Note: WITHDRAWN represents staff-initiated withdrawal on behalf of a
candidate. There is no candidate-portal or self-service role in the current
architecture. All withdrawal actions are performed by SA, HRD, or Recruiter.
```

### Update (PUT /api/v1/applications/{id}):

```text
Only `notes` and `currentStage` are editable via PUT.
`status`, `candidateId`, `vacancyId`, `tenantId` are not editable via PUT.
`submittedAt`, `createdAt` are not editable.
Updating a terminal-state application is blocked: APPLICATION_IN_TERMINAL_STATE.
```

## Decision 12 — DTO Scope

### CreateApplicationDto:

```text
candidateId   UUID     required
vacancyId     UUID     required
notes         string   optional; if present, max 5000 characters
currentStage  string   optional; max 100 characters

EXCLUDED (must be stripped or rejected, not passed through):
tenantId      — derived from JWT only (SEC-003)

```

### UpdateApplicationDto:

```text
notes         string   optional; max 5000 characters if present
currentStage  string   optional; max 100 characters if present

EXCLUDED:
status        — not editable via PUT
tenantId      — not accepted
candidateId   — not editable after creation
vacancyId     — not editable after creation
submittedAt   — not editable
```

### AdvanceApplicationDto:

```text
targetStatus  string   required
Allowed values: SCREENING, INTERVIEW, EVALUATION, OFFER

NOT allowed as targetStatus: APPLIED, HIRED, REJECTED, WITHDRAWN
```

### ListApplicationsQueryDto (query parameters):

```text
page          number   optional; default 1; min 1
pageSize      number   optional; default 20; max 100
status        string   optional; filter by status value
candidateId   UUID     optional; filter to one candidate's applications
vacancyId     UUID     optional; filter to one vacancy's applications

No free-text search in M17.
```

### ApplicationResponseDto (HTTP response shape):

```text
id            string   UUID
candidateId   string   UUID
vacancyId     string   UUID
status        string
submittedAt   string   ISO 8601
currentStage  string | null
notes         string | null
createdAt     string   ISO 8601
updatedAt     string   ISO 8601

EXCLUDED from all HTTP responses:
tenantId      — SEC-003; must not appear in response body
deletedAt     — soft-deleted records appear as NOT_FOUND; field never exposed
```

The controller must implement a `toApplicationShape()` function (equivalent
to M16's `toCandidateShape()`) that projects the ApplicationRecord from the
service into the HTTP response object, explicitly omitting `tenantId` and
`deletedAt`.

All mutation and action endpoints (create, update, advance, reject, withdraw)
must return the updated ApplicationResponse on success. This is consistent
across the domain: callers always receive the current application state after
a write and do not need a second GET request to observe the result. The
ApplicationService must therefore return an ApplicationRecord on the SUCCESS
path of rejectApplication and withdrawApplication — not merely { outcome:
'SUCCESS' } — so the controller can project it through toApplicationShape().

### List response shape:

```text
{
  applications: ApplicationResponseDto[],
  total: number,
  page: number,
  pageSize: number
}
```

## Decision 13 — Error Codes

M17 defines the following error code vocabulary for application endpoints.
These string values must appear in error response bodies using the existing
platform error response style.

```text
APPLICATION_NOT_FOUND            — application absent, soft-deleted, or cross-tenant
APPLICATION_ALREADY_EXISTS       — active application for this candidate+vacancy exists
APPLICATION_IN_TERMINAL_STATE    — operation attempted on HIRED, REJECTED, or WITHDRAWN
APPLICATION_AWAITING_HIRE        — advance from OFFER blocked; hire is M19 scope
INVALID_APPLICATION_TRANSITION   — advance targetStatus is not the next sequential state
CANDIDATE_NOT_FOUND              — candidate absent, deleted, or cross-tenant
CANDIDATE_ARCHIVED               — candidate exists but status = ARCHIVED
VACANCY_NOT_FOUND                — vacancy absent, deleted, or cross-tenant
VACANCY_NOT_OPEN                 — vacancy status is not OPEN or IN_RECRUITMENT
CANDIDATE_HAS_ACTIVE_APPLICATIONS — candidate has active applications; archive blocked
UNAUTHORIZED                     — no authenticated user (401)
FORBIDDEN                        — authenticated but insufficient role (403)
INTERNAL_ERROR                   — unexpected server error (500)
```

### Hiring Manager scoped access — deferred from M17:

GD-PRE-PHASE3-003 D3 designates Hiring Manager access as "S" (scoped read —
limited to vacancies in their department). Inspection of the identity schema
finds:

```text
UserRole model: fields are userId and roleId only. No departmentId.
User model: no departmentId field.
Vacancy model: no direct departmentId field.
Department reachable from Vacancy only via: vacancy → position → department (two hops).
No governed manager-to-department assignment data model exists.
```

Scoped Hiring Manager application access cannot be implemented without a
governed data model that associates a Hiring Manager user with one or more
departments. Implementing unscoped (unrestricted) HM read access would
violate GD-PRE-PHASE3-003 D3 intent. Implementing an ad-hoc scoping
mechanism without governance would introduce an ungoverned pattern.

**M17 must deny all Hiring Manager application access (403 on all application
endpoints for HM role).**

A future governance decision must define:

```text
1. The data model for manager-to-department assignment (e.g., an explicit
   user_department_assignments table, or a departmentId field on UserRole,
   or manager-vacancy assignment at the vacancy level).
2. The exact query filter for HM scoped list and detail access.
3. Whether HM scoping applies to all 7 application endpoints or only list/detail.
```

Until that decision is issued and implemented, Hiring Manager application
access remains forbidden. This deferral is recorded in governance_history.md.

## Decision 15 — Tenant Isolation (SEC-003)

The following rules apply to all M17 application endpoints and service methods:

```text
1. tenant_id on recruiting.applications is derived from the authenticated
   user's JWT token only. It is never accepted from the request body, URL
   parameters, or query parameters.

2. All application service read operations include:
     WHERE tenant_id = :actorTenantId AND deleted_at IS NULL

3. Cross-tenant application records return APPLICATION_NOT_FOUND (404), not
   403 Forbidden. This prevents tenant enumeration.

4. Cross-tenant candidate references during application create return
   CANDIDATE_NOT_FOUND (404), not 403.

5. Cross-tenant vacancy references during application create return
   VACANCY_NOT_FOUND (404), not 403.

6. tenantId must not appear in any HTTP response body.

7. deletedAt must not appear in any HTTP response body.

8. Soft-deleted applications (deleted_at IS NOT NULL) are invisible to all
   application reads and return APPLICATION_NOT_FOUND.

9. No error difference must allow a caller to distinguish "record doesn't
   exist" from "record exists in another tenant."
```
