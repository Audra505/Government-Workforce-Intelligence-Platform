# Context

spec/01_requirements.md FR-113 states that employee skill history must be retained.
FR-114 implies certification renewal tracking through expiration monitoring.
spec/05_database_schema.md defines composite primary keys:

```text
workforce.employee_skills:          PRIMARY KEY (employee_id, skill_id)
workforce.employee_certifications:  PRIMARY KEY (employee_id, certification_id)
```

These composite PKs permit exactly one record per employee-skill and
employee-certification pair, preventing storage of multiple records per pair.
This conflicts with a literal reading of "history retained" as a separate history table.

This decision resolves the conflict by establishing the audit event trail as the
authoritative, compliance-grade history mechanism, and confirming the composite PK
schema as implemented.

---

# Decisions

## Decision 1 — History Mechanism

The immutable audit event trail in audit.audit_events is the authoritative history
mechanism for employee skill and certification records.

Every create and update operation on workforce.employee_skills and
workforce.employee_certifications must generate a corresponding immutable audit event.
The audit event record is the durable, queryable, and legally admissible history record
for all changes to employee skill and certification assignments.

FR-113 "history retained" is satisfied by this mechanism. The audit trail provides
complete change history: who assigned the skill, when, at what proficiency, when
proficiency was updated, and who made each change.

## Decision 2 — Junction Table Schema

spec/05 composite primary keys are implemented exactly as specified:

```text
workforce.employee_skills:          PRIMARY KEY (employee_id, skill_id)
workforce.employee_certifications:  PRIMARY KEY (employee_id, certification_id)
```

No surrogate PKs. No history tables. No additional columns beyond spec/05.
No spec deviation on table structure.

## Decision 3 — Upsert Semantics

Assignment endpoints use upsert semantics against the composite PK.

A second POST /api/v1/employees/{id}/skills for a skill the employee already holds
updates the existing record and emits WORKFORCE_EMPLOYEE_SKILL_UPDATED.

A second POST /api/v1/employees/{id}/certifications for a certification the employee
already holds updates the existing record and emits WORKFORCE_EMPLOYEE_CERT_UPDATED,
WORKFORCE_EMPLOYEE_CERT_RENEWED, or WORKFORCE_EMPLOYEE_CERT_REVOKED depending on
which fields changed (see Decision 4 for event selection rules).

## Decision 4 — Audit Event Types for M13

The following AuditEventType values must be added to the AuditEventType enum before
M13 Step 1 (Schema Foundation) implementation begins:

```text
Skills catalog:
  WORKFORCE_SKILL_CREATED               — new skill entry created in catalog
  WORKFORCE_SKILL_UPDATED               — skill catalog entry updated

Certifications catalog:
  WORKFORCE_CERTIFICATION_CREATED       — new certification type created in catalog
  WORKFORCE_CERTIFICATION_UPDATED       — certification catalog entry updated

Employee skill assignments:
  WORKFORCE_EMPLOYEE_SKILL_ASSIGNED     — skill first assigned to an employee
  WORKFORCE_EMPLOYEE_SKILL_UPDATED      — existing employee skill record updated

Employee certification assignments:
  WORKFORCE_EMPLOYEE_CERT_ASSIGNED      — certification first assigned to an employee
  WORKFORCE_EMPLOYEE_CERT_UPDATED       — generic cert record update (issueDate, etc.)
  WORKFORCE_EMPLOYEE_CERT_RENEWED       — status changed from EXPIRED to ACTIVE with
                                          new issueDate and/or expirationDate; prior
                                          expirationDate recorded in event metadata
  WORKFORCE_EMPLOYEE_CERT_REVOKED       — status changed to REVOKED; revocation
                                          context recorded in event metadata
```

Event selection for certification upsert:

```text
If status changes to ACTIVE and prior status was EXPIRED: CERT_RENEWED
If status changes to REVOKED:                             CERT_REVOKED
Otherwise:                                                CERT_UPDATED
```

## Decision 5 — Audit Event Metadata for Key Events

WORKFORCE_EMPLOYEE_SKILL_UPDATED must include in metadata:

```text
prior_proficiency_level, new_proficiency_level
prior_verified_at, new_verified_at (where changed)
```

WORKFORCE_EMPLOYEE_CERT_RENEWED must include in metadata:

```text
prior_expiration_date, new_expiration_date
prior_issue_date (if changed), new_issue_date (if changed)
```

WORKFORCE_EMPLOYEE_CERT_REVOKED must include in metadata:

```text
certification_id, certification_name
prior_status, revocation_timestamp
```

## Decision 6 — Future Evolution Path

If a future milestone requires separate history tables (e.g., a legal mandate for
immutable per-issuance records), a new governance decision must be issued. The
implementation path is additive: employee_skill_history and employee_cert_history
tables may be added alongside the existing junction tables without modifying the
current schema. The composite PKs and upsert semantics on the current tables are
not affected by such an addition.

---

# Rationale

The AuditService is globally injectable and operational since Milestone 4. Immutable
audit events satisfy all regulatory government HR audit requirements: they record who
made each change, when, from what previous state, and to what new state. This is
superior to a history table for compliance purposes because audit events are write-once
(not updateable), indexed by tenant and timestamp, and exportable per FR-500 compliance
requirements.

The composite PK design correctly models the business reality that an employee holds
one current skill record per skill type — not multiple simultaneous records. History
of how that current record evolved lives in the audit trail.

---

# Spec Deviation

None.

spec/05 composite PKs are implemented exactly as written.

FR-113 "history retained" is satisfied by Decision 1 (audit event trail). The audit
mechanism is more robust than a history table for regulatory purposes: audit events
are immutable, searchable, and exportable per FR-500.
