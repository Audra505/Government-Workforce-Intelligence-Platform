M20 Recruiter Workspace UI is:

```text
Phase 3 Recruiting & Staffing — fifth implementation milestone
Frontend-only — no new backend endpoints, no Prisma schema changes,
  no new migrations, no backend service or controller changes
Recruiter Workspace — delivers the first user-visible surface for the
  Phase 3 Recruiting & Staffing domain

  and serverFetch reads

Not an AI milestone (Phase 4)
Not a resume file storage milestone (infrastructure decision pending;
  GD-PRE-PHASE3-001 D7)
Not a Hiring Manager scoped workspace milestone
Not a dashboards or analytics milestone (Phase 5)
```

M20 makes the Phase 3 backend workflow user-visible for the first time.
After M20, authorized users can manage the full recruiting lifecycle
through the UI: create candidates, track applications, schedule interviews,
manage offers, and initiate the hire action.

M20 does not extend or modify the backend domain. All business rules,
RBAC enforcement, and data integrity guarantees remain at the NestJS API
layer (M16–M19). M20 delivers only the UI surface that calls those
existing endpoints.

### BFF handler security constraints (mandatory):

```text
1. tenantId is NEVER accepted from the browser request body.
   BFF handlers must not forward any tenantId field to the NestJS API.
   tenantId is derived from the JWT at the NestJS layer (SEC-003).

2. Each BFF handler reads the session cookie for the JWT token using
   SESSION_COOKIE from @/lib/auth, following the existing vacancy route
   handler pattern in apps/web/src/app/api/vacancies/route.ts.

3. BFF handlers must not implement business logic. They proxy the request
   body to NestJS and proxy the response status and error codes back to
   the browser. No data transformation beyond stripping tenantId.

4. API_URL is read from process.env.API_URL (default 'http://localhost:3001'),
   consistent with the existing BFF pattern.

5. All BFF handlers must handle these error cases:
   — Invalid JSON request body → 400 VALIDATION_ERROR
   — Missing session cookie → 401 UNAUTHORIZED
   — NestJS unavailable (fetch throws) → 503 INTERNAL_ERROR
   — NestJS error response → proxy NestJS status and error code verbatim

6. Response type alias: BffResponse = { success: true } |
   { success: false; error: { code: string; message: string } }
   Use the same BffResponse pattern as the existing vacancy handler.
```

## Decision 5 — RBAC and UX Gates

GD-PRE-PHASE3-003 is the authoritative RBAC source. This decision
translates the API-layer RBAC into UI surface restrictions.

The UI surface does not enforce RBAC — all enforcement remains at the
NestJS API. The UI applies these rules to determine which action
surfaces to render, to avoid presenting buttons that will return 403.

### Role-to-surface mapping:

```text
System Administrator (SA):
  Full workspace visibility
  All action surfaces rendered: archive candidate, create application,
    advance/reject/withdraw application, schedule interview, manage
    interview status, create offer, submit/approve/send offer, record
    offer response, withdraw offer
  Hire action surface: RENDERED (SA is authorized; GD-PRE-PHASE3-003 D3)

HR Director (HRD):
  Full workspace visibility
  All action surfaces rendered (same as SA)
  Hire action surface: RENDERED (HRD is authorized; GD-PRE-PHASE3-003 D3)

Recruiter:
  Full workspace visibility
  Action surfaces rendered: archive candidate, create application,
    advance/reject/withdraw application, schedule interview, manage
    interview status, create offer, submit offer, withdraw offer
  Approve offer surface: NOT RENDERED (Recruiter cannot approve offers;
    GD-PRE-PHASE3-003 D16)
  Send offer surface: NOT RENDERED (Recruiter cannot send offers;
    GD-PRE-PHASE3-003 D16)
  Hire action surface: NOT RENDERED (Recruiter is denied; GD-PRE-PHASE3-003 D3)

Compliance Officer (CO):
  Full workspace visibility — READ ONLY
  No action surfaces rendered (CO has read-only access to Phase 3
    endpoints; GD-PRE-PHASE3-003 D3)
  No create, advance, approve, send, hire, or any write button rendered

Hiring Manager (HM):
  HM scoped workspace deferred beyond M20 (see Decision 13 Exclusions
    and Unresolved Questions). M20 does not render any HM-specific
    recruiting workspace. If an HM session reaches a recruiting page,
    the page renders read-only data consistent with CO surface above.
    API access for HM follows GD-PRE-PHASE3-003 (HM is denied most
    recruiting endpoints; any visible HM data is limited to read scope
    that HM endpoints allow, or the page will show no data if HM lacks
    API access).

Workforce Planner (WP) / Executive User (EU):
  No action surfaces. If these roles navigate to a recruiting page,
  the page should display a "not authorized" message rather than
  rendering data. WP and EU have no Phase 3 recruiting API access
  (GD-PRE-PHASE3-003 D3); all read requests will return 403, resulting
  in empty or error states. M20 should handle this gracefully.
```

### Role determination:

```text
The role of the current user must be derived from the session JWT or
session context available in Next.js Server Components or BFF handlers.
The UI must not derive RBAC from client-side state alone.

The existing session pattern (SESSION_COOKIE, JWT from cookie) is the
sole source of role information for UI surface gate decisions.
```

## Decision 6 — Tenant Isolation (SEC-003)

```text
1. tenantId is NEVER accepted from or surfaced in the browser UI.
   No form field, hidden input, URL parameter, or query parameter may
   carry tenantId in M20 pages or BFF handlers.

2. BFF handlers must not include tenantId in the body forwarded to
   NestJS. tenantId is derived from the JWT by NestJS (SEC-003).

3. API responses may include tenantId in NestJS responses; BFF handlers
   and Server Components must not forward tenantId fields to the browser.
   This is enforced by the NestJS response projections (toXShape() functions),
   not by BFF-layer stripping. M20 trusts the NestJS projection.

4. Server Components call the NestJS API using the session JWT via
   serverFetch (reads) or via BFF handlers (writes). The tenantId is
   embedded in the JWT and is never passed as a separate parameter by
   M20 code.
```

```text
VAC-601 enforcement (blocking workflow based on requiresReview) is deferred
```

## Decision 9 — BFF and serverFetch Constraint

```text
serverFetch<T>(path) in apps/web/src/lib/api.ts is GET-only and must
remain GET-only after M20. M20 must not modify serverFetch to support
write operations.

Write operations (POST/PATCH/DELETE) must use per-route BFF handler files
under apps/web/src/app/api/recruiting/. This is the existing pattern
used by the workforce domain (see apps/web/src/app/api/vacancies/route.ts)
and must be followed consistently in M20.

Rationale:
— Consistency with the existing BFF pattern avoids introducing two
  write-path patterns into the codebase.
— The per-route BFF pattern provides a clear security boundary: each
  write action is an explicit, auditable, testable proxy.
— Extending serverFetch for writes would require adding HTTP method and
  body parameters to a utility that currently has a simple GET-only
  contract, increasing complexity beyond M20 scope.

Server Components use serverFetch for all GET reads. Forms and action
buttons call BFF handlers via the browser fetch API (or Server Actions
if consistent with the page architecture, but the BFF handler pattern
is preferred for consistency with existing code).
```

## Decision 16 — UI Design System

M20 is the first milestone to apply a formal design system to the
GWIP frontend. The following standards are mandatory for all M20
pages and components. The Workforce section (Employees, Positions,
Vacancies) will be refactored to match in a separately governed
post-M20 milestone. M20 must not begin that refactor.

### Typography

```text
IBM Plex Sans — all UI text (weights: 300, 400, 500, 600)
IBM Plex Mono — IDs, codes, employee numbers, salaries, counts,
  vacancy numbers, and other fixed-width values where monospace
  improves scannability

Load via Google Fonts CDN in the Next.js root layout:
  IBM Plex Sans: wght@300;400;500;600
  IBM Plex Mono: wght@400;500
```

### Color Palette

```text
Navy:        #0c2340  — header background only
Blue accent: #2563eb  — primary action buttons, active tab underline, links
Blue light:  #eff6ff  — RBAC notice / info banner background
White:       #ffffff  — content card and table row surfaces
Background:  #f8fafc  — page canvas, table header rows, form field backgrounds
Border:      #e2e8f0  — card borders, table dividers, input borders
Text:        #0f172a  — primary body text
Subtext:     #475569  — secondary labels, dates, department names, metadata
Muted:       #94a3b8  — placeholder text, table column headers, hints

Status dot colors:
  Green:  #16a34a — Active, Completed, Filled, Accepted, Hired
  Yellow: #ca8a04 — Pending Onboarding, Under Review, Pending Approval,
                    Interview, Evaluation
  Blue:   #2563eb — Scheduled, Sent, Open, Applied, Screening, In Recruitment
  Orange: #ea580c — No Show, In Recruitment (vacancy)
  Purple: #7c3aed — Offer stage (where purple is preferred over green
                    to distinguish pre-acceptance from accepted)
  Red:    #dc2626 — Rejected, Declined, Revoked, Suspended
  Gray:   #64748b — Archived, Separated, Cancelled, Draft, Withdrawn, Closed
```

### Status Indicators — Mandatory Format

```text
Status values must be displayed as:
  6px solid filled circle (dot) + single space + status label text

Example rendering:
  ● Active      (green dot, "Active" text)
  ● Scheduled   (blue dot, "Scheduled" text)
  ● Rejected    (red dot, "Rejected" text)

This format applies to all recruiting entity status values across all
M20 pages: candidates, applications, interviews, offers.

PROHIBITED in M20:
  — Pill badges (background-filled, border-enclosed status chips)
  — Chip components with padding and rounded-rectangle borders
  — Status labels with any background color behind the text
  — Any visual container or enclosed shape around status text

The dot + label is the only authorized status display pattern.
Note: where the word "badge" appears in Decision 7 of this document,
it refers to this dot + label pattern. It does not authorize pill
badge or chip implementation.
```

### Header Navigation

```text
The production application header contains exactly two domain nav items:
  Workforce  — links to the Workforce section
  Recruiting — links to /recruiting/candidates

No additional nav items are authorized in M20. In particular:
  — No "Design Demo" nav item (prototype tool only; see Decision 13)
  — No "Positions" as a top-level nav item (Positions is under Workforce)
  — No prototype, mockup, or placeholder navigation item of any kind

GWIP is the header wordmark placeholder. Final agency branding
replaces it at go-live. No implementation decision depends on
this string.

No role switcher control is implemented in the production header.
The session role is derived from the JWT only. The role switcher
visible in the design reference mockup (tmp/platform-mockup-v2.html)
is a preview tool and must not appear in any production code.
```

### Page Layout

```text
Breadcrumb row below the header (e.g. Recruiting › Candidates)
Inline tab navigation within the Recruiting section
  (Candidates | Applications | Interviews | Offers)
No shared sidebar — inline per-page nav only (Decision 4)
Page canvas background: #f8fafc
Content max-width: 1200px
Content padding: 24px
Table row height: 48px
Table header rows: #f8fafc background, uppercase 11px muted labels
```

### Feedback Pattern

```text
Success and error feedback: bottom-right toast notification
  Success — green-tinted (#f0fdf4 background), bold title + short
            message, auto-dismiss after ~4 seconds
  Error   — red-tinted (#fef2f2 background), bold title + short
            message, auto-dismiss or user-dismissed

Inline field validation: error text below the field in red (#dc2626)
with a small warning icon. No full-page error states for form
validation.
```

### Pagination

```text
All list views: 25 rows per page
Pagination controls: Previous | page numbers | Next, right-aligned
in the table footer row.
```

### Design Reference Mockup

```text
A temporary HTML file (tmp/platform-mockup-v2.html) was created as a
visual design exploration and decision-making tool before M20A.
It is scratch-space only:
  — Must not be committed to the repository (tmp/ is never committed)
  — Must not become a production route, page, component, or artifact

The mockup demonstrates the design language documented in this
```

## Unresolved Questions

```text
1. Hiring Manager scoped recruiting workspace:
   GD-PRE-PHASE3-003 D2 defers HM scoped access because no governing
   identity model links a user to a department as manager. No governing
   spec defines a user-to-department manager relationship. Until this
   model is defined and governed, M20 cannot implement an HM-specific
   recruiting workspace. This remains deferred beyond M20.

   Prerequisite for resolution: a governance decision defining the
   identity schema extension that maps a user (role: Hiring Manager)
   to one or more departments, enabling scoped API access per
   GD-PRE-PHASE3-003 D10–D12.

2. Resume file storage infrastructure:
   GD-PRE-PHASE3-001 D7 records that a resume file storage infrastructure
   decision is required before any resume upload endpoint is created.
   This decision is not required for M20 (resume upload is excluded from
   M20 per Decision 13) but remains open as a prerequisite for Phase 4
   resume features.

3. Shared sidebar vs. inline nav long-term:
   M20 uses inline per-page nav to avoid refactoring existing workforce
   pages (Decision 4). A future milestone may introduce a shared layout
   component that unifies workforce and recruiting navigation. When that
   milestone is planned, it must include refactoring all existing pages
   (workforce + recruiting) to use the shared component.
```
