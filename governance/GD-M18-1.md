## Decision 4 — Interview Schema Scope

The canonical `recruiting.interviews` table definition from GD-PRE-PHASE3-002
Decision 5 is authoritative. M18A must implement this exact column set.

### Required Columns

```sql
id                   UUID         PRIMARY KEY
tenant_id            UUID         NOT NULL
application_id       UUID         NOT NULL
interview_type       VARCHAR(50)  NOT NULL
scheduled_at         TIMESTAMPTZ  NULL
status               VARCHAR(50)  NOT NULL DEFAULT 'SCHEDULED'
interviewer_name     VARCHAR(255) NULL
interviewer_user_id  UUID         NULL
feedback             TEXT         NULL
created_at           TIMESTAMPTZ  NOT NULL DEFAULT NOW()
updated_at           TIMESTAMPTZ  NOT NULL DEFAULT NOW()
deleted_at           TIMESTAMPTZ  NULL
```

### Required Indexes and FK Constraint (hand-edited migration)

```sql
CREATE INDEX idx_interviews_tenant
    ON recruiting.interviews(tenant_id);

CREATE INDEX idx_interviews_application
    ON recruiting.interviews(tenant_id, application_id);

ALTER TABLE recruiting.interviews
    ADD CONSTRAINT fk_interview_application
    FOREIGN KEY (application_id) REFERENCES recruiting.applications(id);
```

### Prisma Model

The Prisma model uses `@@schema("recruiting")` following the pattern established
in M16 (Candidate) and M17 (Application). The Interview model uses a
one-directional relation to Application. Do not add a back-relation
`interviews Interview[]` on the Application model unless ApplicationService
has a direct need to load interviews via the Application model.

### Interviewer Field Rules

Both `interviewer_name` and `interviewer_user_id` are nullable at the database
constraint level. Service-layer enforcement requires that at least one of the
two fields is present when an interview record is created. This allows
flexibility for external interviewers (interviewer_name only) and for
platform users who are the assigned interviewer (interviewer_user_id).

No database-level FK constraint is added for `interviewer_user_id`
(per GD-PRE-PHASE3-002 D5). When `interviewer_user_id` is provided, the
service must confirm the referenced user exists and belongs to the same tenant.

If `interviewer_user_id` is provided but the referenced user does not exist,
is soft-deleted, or belongs to a different tenant, the service returns
INTERVIEWER_NOT_FOUND (404). The response must not reveal whether the user
exists in another tenant — a cross-tenant user reference must return 404
indistinguishable from a genuine not-found response. This prevents user
enumeration across tenant boundaries (SEC-003 extension to user references).

This rule applies at both interview create (POST /interviews) and interview
update (PUT /interviews/:id), since interviewerUserId is settable via both
endpoints.

## Decision 5 — Interview Lifecycle Rules

### Status values (GD-PRE-PHASE3-002 D8 — authoritative):

```text
SCHEDULED  — interview has been scheduled; initial state; mutable state
COMPLETED  — interview took place; terminal for status transitions
CANCELLED  — interview was cancelled; terminal for this interview record
NO_SHOW    — candidate did not appear; terminal for this interview record
```

### State transitions:

```text
SCHEDULED → COMPLETED  (via POST /interviews/:id/complete)
SCHEDULED → CANCELLED  (via POST /interviews/:id/cancel)
SCHEDULED → NO_SHOW    (via POST /interviews/:id/no-show)
```

No transitions are permitted from COMPLETED, CANCELLED, or NO_SHOW. These
are terminal status states for the interview record. Attempting any status
action on a terminal-state interview returns INTERVIEW_IN_TERMINAL_STATE.

### Cancellation finality:

CANCELLED is terminal for the specific interview record. A recruiter who
needs to reschedule a cancelled interview must create a new interview record
for the same application. This design:

```text
1. Preserves a complete audit history of all scheduled interviews.
2. Avoids re-entry ambiguity in the state machine.
3. Keeps the lifecycle simple and unidirectional.
```

### Rescheduling:

Rescheduling is implemented by creating a new interview record (POST /interviews)
for the same application with a new scheduled_at. No reactivation action exists
on cancelled or completed interview records.

### Feedback rules:

```text
Feedback may be recorded when status = SCHEDULED or status = COMPLETED.
Feedback may not be recorded when status = CANCELLED or status = NO_SHOW.
  (Rationale: a cancelled or no-show interview has no substantive feedback to record.)
Recording feedback does not change interview status.
Feedback overwrites any prior feedback text.
The feedback action always returns the updated interview record on success.
```

### Multiple interviews per application:

Multiple interview records for the same application are allowed. No uniqueness
constraint limits the count of interview records per application. An application
at INTERVIEW status may have concurrent SCHEDULED interviews of different types
(e.g., a PHONE_SCREEN and a PANEL in the same pipeline). The service does not
enforce a limit on active interviews per application.

### Application status check for interview creation:

An interview may be created for an application in any non-terminal status:

```text
Permitted:   APPLIED, SCREENING, INTERVIEW, EVALUATION, OFFER
Blocked:     REJECTED, WITHDRAWN, HIRED → return APPLICATION_IN_TERMINAL_STATE
```

The application state machine (M17) governs application status transitions.
Interview creation does not change application status.

## Decision 6 — Offer Schema Scope

The canonical `recruiting.offers` table definition from GD-PRE-PHASE3-002
Decision 6 is authoritative. M18B must implement this exact column set.

### Required Columns

```sql
id             UUID         PRIMARY KEY
tenant_id      UUID         NOT NULL
application_id UUID         NOT NULL
status         VARCHAR(50)  NOT NULL DEFAULT 'DRAFT'
offer_date     TIMESTAMPTZ  NULL
accepted_at    TIMESTAMPTZ  NULL
declined_at    TIMESTAMPTZ  NULL
notes          TEXT         NULL
created_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW()
updated_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW()
deleted_at     TIMESTAMPTZ  NULL
```

### Required Partial Unique Index (hand-edited migration)

```sql
CREATE UNIQUE INDEX idx_offers_application_active
    ON recruiting.offers(application_id)
    WHERE deleted_at IS NULL
      AND status NOT IN ('DECLINED', 'WITHDRAWN');
```

This index enforces one active offer per application at a time. A new offer
may only be created after the previous offer reaches a terminal state (DECLINED
or WITHDRAWN). The index is partial so that re-offer is permitted after a
terminal offer state without conflict.

### Required Indexes and FK Constraint

```sql
CREATE INDEX idx_offers_tenant
    ON recruiting.offers(tenant_id);

CREATE INDEX idx_offers_application
    ON recruiting.offers(tenant_id, application_id);

ALTER TABLE recruiting.offers
    ADD CONSTRAINT fk_offer_application
    FOREIGN KEY (application_id) REFERENCES recruiting.applications(id);
```

### Prisma Model

The Prisma model uses `@@schema("recruiting")`. One-directional relation to
Application. Do not add a back-relation `offers Offer[]` on the Application
model unless OfferService needs to traverse it directly.

## Decision 7 — Offer Lifecycle Rules

### Status values (GD-PRE-PHASE3-002 D8 — authoritative):

```text
DRAFT             — offer is being prepared; initial state; mutable via PUT
PENDING_APPROVAL  — submitted for HR Director / SA review; immutable via PUT
APPROVED          — approved; ready to send; immutable via PUT
SENT              — offer sent to candidate; awaiting response
ACCEPTED          — candidate accepted; terminal state
DECLINED          — candidate declined; terminal state
WITHDRAWN         — agency withdrew the offer; terminal state
```

### State transitions:

```text
DRAFT            → PENDING_APPROVAL  (via POST /offers/:id/submit)
PENDING_APPROVAL → APPROVED          (via POST /offers/:id/approve)
APPROVED         → SENT              (via POST /offers/:id/send)
SENT             → ACCEPTED          (via POST /offers/:id/record-response with response=ACCEPTED)
SENT             → DECLINED          (via POST /offers/:id/record-response with response=DECLINED)

Any non-terminal state → WITHDRAWN   (via POST /offers/:id/withdraw)
  DRAFT → WITHDRAWN
  PENDING_APPROVAL → WITHDRAWN
  APPROVED → WITHDRAWN
  SENT → WITHDRAWN
```

No transitions are permitted from ACCEPTED, DECLINED, or WITHDRAWN. Attempting
any action on a terminal-state offer returns OFFER_IN_TERMINAL_STATE.

### accepted_at and declined_at:

These columns record the timestamp when the corresponding terminal state is
reached:

```text
accepted_at: set by OfferService when status transitions to ACCEPTED
declined_at: set by OfferService when status transitions to DECLINED
```

No column tracks when an offer was withdrawn. The `updated_at` timestamp
reflects when status last changed. This is consistent with the canonical
schema from GD-PRE-PHASE3-002 D6, which does not include a `withdrawn_at`
column.

### PUT scope:

PUT /offers/:id is permitted only when the offer is in DRAFT status. Attempting
PUT on a non-DRAFT offer returns OFFER_NOT_IN_DRAFT. Fields editable via PUT:

```text
offerDate  — optional
notes      — optional; max 5000 characters
```

Status, applicationId, tenantId, createdAt, submittedAt are not editable via PUT.

## Decision 8 — Application-Interview Relationship Rules

### Interview creation requires a valid application:

```text
The submitted applicationId must resolve to an application that:
  - exists (not soft-deleted; deleted_at IS NULL)
  - belongs to the same tenant as the authenticated user (SEC-003)
  - is in a non-terminal status (APPLIED, SCREENING, INTERVIEW, EVALUATION, OFFER)

A soft-deleted or cross-tenant application returns INTERVIEW_NOT_FOUND or
APPLICATION_NOT_FOUND. Service-layer outcomes:
  - application absent or deleted_at IS NOT NULL → APPLICATION_NOT_FOUND
  - application.tenantId ≠ actor.tenantId → APPLICATION_NOT_FOUND (not 403)
  - application.status IN ('REJECTED', 'WITHDRAWN', 'HIRED') →
    APPLICATION_IN_TERMINAL_STATE
```

### Interview does not change application status:

Creating, completing, cancelling, or recording feedback on an interview does
not change the application status. Application status transitions remain
governed exclusively by the application state machine (M17).

### Interview list filtering:

When listing interviews, the following filters are supported:

```text
applicationId  — UUID; filter to one application's interviews
status         — string; filter by interview status
interviewType  — string; filter by interview type
page / pageSize — pagination
```

All interview reads must include WHERE tenant_id = :actorTenantId so that
no cross-tenant interviews are visible.

## Decision 9 — Application-Offer Interaction Rules

### Offer creation requires application.status = OFFER:

An offer may only be created for an application that is currently at OFFER
status in the application state machine. The service must verify:

```text
application.status = 'OFFER'
```

If application.status is any other value, including APPLIED, SCREENING,
INTERVIEW, or EVALUATION, the offer creation is rejected with
APPLICATION_NOT_AT_OFFER_STATUS.

Rationale: an offer is a downstream step in the hiring workflow. It is
semantically incoherent to create a formal offer for an application that
has not been advanced to the offer consideration stage.

### Offer creation does not advance application status:

Creating an offer record does not change application.status. The application
is already at OFFER status as a precondition for offer creation. No status
update to the application is triggered by POST /api/v1/offers.

### Offer acceptance does not advance application status to HIRED:

When an offer is accepted (status → ACCEPTED), the application remains at
OFFER status. The transition from OFFER to HIRED on the application, along
with employee record creation and vacancy FILLED automation (VAC-401 Path B),
is M19 scope and is triggered by POST /applications/:id/hire (not yet
implemented).

This means that after offer acceptance, the system state is:

```text
application.status = 'OFFER'          (unchanged)
offer.status = 'ACCEPTED'             (terminal)
offer.accepted_at = <timestamp>
```

No employee record is created. No vacancy status change occurs.

### Offer decline does not change application status:

When an offer is declined (status → DECLINED), the application remains at
OFFER status. The recruiter may then:

```text
a. Create a new offer (if application remains eligible — see Decision 10)
b. Reject the application via POST /applications/:id/reject
c. Withdraw the application via POST /applications/:id/withdraw
```

No application status change is triggered by offer decline.

## Decision 10 — Re-offer After Terminal Offer State

### Rule:

A new offer may be created for the same application after the prior offer
reaches a terminal state of DECLINED or WITHDRAWN, provided the application
remains eligible:

```text
Eligibility conditions for re-offer:
  - application.status = 'OFFER' (per Decision 9)
  - no active (non-terminal, non-deleted) offer exists for the application
```

Active is defined as:
```text
deleted_at IS NULL AND status NOT IN ('DECLINED', 'WITHDRAWN')
```

This is enforced at both the service layer (pre-check) and the database layer
(partial unique index idx_offers_application_active). If a prior offer exists
in a non-terminal state, the pre-check returns ACTIVE_OFFER_EXISTS.

Rationale: a candidate may decline an initial offer and the agency may
choose to present a revised offer rather than terminating the application.
This is a valid government HR workflow pattern.

## Decision 11 — Hiring Manager Access Disposition

Hiring Manager access to all M18 interview and offer endpoints is denied
(403 Forbidden) for the same reason documented in GD-M17-1 Decision 14.

GD-PRE-PHASE3-003 D3 designates Hiring Manager access as "S" (scoped read)
for interviews tied to vacancies in their department, and scoped read-only
for offers. Inspection confirms that the scoping mechanism remains ungoverned:

```text
UserRole model: fields are userId and roleId only. No departmentId.
User model: no departmentId field.
Vacancy model: no direct departmentId field.
Department reachable from Vacancy only via: vacancy → application → vacancy
  → position → department (multi-hop).
No governed manager-to-department or manager-to-vacancy assignment data
model exists.
```

Implementing unscoped (unrestricted) HM read access would violate
GD-PRE-PHASE3-003 D3 intent. Implementing an ad-hoc scoping mechanism
without governance would introduce an ungoverned pattern.

**M18 must deny all Hiring Manager interview and offer access
(403 on all interview and offer endpoints for the HM role).**

A future governance decision must define:

```text
1. The data model for manager-to-department or manager-to-vacancy assignment
   (e.g., an explicit user_department_assignments table, a departmentId field
   on UserRole, or manager-vacancy assignment at the vacancy level).
2. The exact query filter for HM scoped list and detail access across
   candidates, applications, interviews, and offers.
3. Whether HM feedback authority (POST /interviews/:id/feedback) is granted
   based on interviewer_user_id matching or based on department assignment.
4. The RBAC amendment required to GD-PRE-PHASE3-003 once scoping is defined.
```

Until that decision is issued and implemented, Hiring Manager access to all
interview and offer endpoints remains forbidden. This deferral is recorded in
governance_history.md.

## Decision 14 — Error Codes

M18 defines the following error code vocabulary for interview and offer endpoints.

```text
INTERVIEW_NOT_FOUND              — interview absent, soft-deleted, or cross-tenant

                                   NO_SHOW interview

FORBIDDEN                        — authenticated but insufficient role (403)
INTERNAL_ERROR                   — unexpected server error (500)
```

## Decision 16 — Tenant Isolation (SEC-003)

The following rules apply to all M18 interview and offer endpoints and service
methods:

```text
1. tenant_id on recruiting.interviews and recruiting.offers is derived from
   the authenticated user's JWT token only. It is never accepted from the
   request body, URL parameters, or query parameters.

2. All interview and offer service read operations include:
     WHERE tenant_id = :actorTenantId AND deleted_at IS NULL

3. Cross-tenant interview or offer records return INTERVIEW_NOT_FOUND or
   OFFER_NOT_FOUND (404), not 403. This prevents tenant enumeration.

4. Cross-tenant application references during interview or offer create
   return APPLICATION_NOT_FOUND (404), not 403.

5. tenantId must not appear in any HTTP response body.

6. deletedAt must not appear in any HTTP response body.

7. Soft-deleted interviews and offers (deleted_at IS NOT NULL) are invisible
   to all reads and return INTERVIEW_NOT_FOUND or OFFER_NOT_FOUND.

8. No error message must allow a caller to distinguish "record doesn't exist"
   from "record exists in another tenant."
```
